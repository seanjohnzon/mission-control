# D1.14 — SECRETS_POLICY
**Autonomous AI Organization — Credential Classification, Storage, Activation, and Governance**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document governs how secrets are managed across the organization. A secret is any piece of access material that, if exposed, would allow unauthorized actors to act as the organization: API keys, bot tokens, platform credentials, service account passwords, signing keys, and webhook secrets.

Secrets are not configuration. They are not convenience. They are the keys to the organization's external presence, financial exposure, and communication channels. Their governance requires the same discipline as financial controls — because in this organization, they effectively are financial and operational controls.

**This document is a hard gate.** Nothing requiring a credential may be activated until that credential's governance is defined here. This includes:
- The Telegram bot token (primary coordination channel)
- The Anthropic API key (primary inference)
- Any Discord bot token (optional — not activated by default)
- Any xAI / Grok key (inactive until D1.10 gate is satisfied)
- Any platform, commerce, or service credential

The fact that a key has been obtained does not mean it is ready to use. A secret is not active until it is registered, scoped, stored in an approved location, and the capability it unlocks has been approved per D1.12.

**This document governs:**
- What constitutes a secret and how secrets are classified
- Which environments each secret class is permitted in
- Who has authority over secret creation, scoping, rotation, and revocation
- How secrets are registered (metadata only — values never appear in documents)
- Where secrets may be stored and where they are explicitly forbidden
- How agents may and may not interact with secrets at runtime
- What counts as compromise and what the response is
- What is logged and what is never logged
- Which secrets are activated at each phase

**What this document does not cover:**
- Setup steps for specific secret stores — D2.x runbooks
- Environment boundaries (the host environments these secrets operate in) — D1.13-ENV_CONTROL.md
- Capability acquisition (the capabilities these secrets unlock) — D1.12-CAPABILITY_ACQUISITION_POLICY.md
- Communication channel configuration — D1.11-COMMUNICATION_MODES.md, D1.9-BROKER_POLICY.md

## 3. Secrets Policy Principles

1. **Least privilege, always.** Every secret is scoped to the minimum access needed for the specific capability it unlocks. Broader scope than necessary is not a convenience — it is an unacceptable risk surface. The default scope for any new credential is the narrowest possible.

2. **One authoritative registration record per secret.** Every secret in active use has exactly one registration entry in the secret registry (§7). Secrets that exist but are not registered are non-compliant. Secrets that have multiple registration entries are ambiguous and must be resolved immediately.

3. **Secret values never enter Mission Control git.** The secret registry tracks metadata — ID, type, scope, owner, expiry, status. The actual credential value is never written to a D-series document, a task record, a registry entry, a proof artifact, or any file in the Mission Control workspace. A value committed to git, even briefly, is treated as compromised. (Source: D1.13 §8.3)

4. **Process memory is temporary; files are governed.** Loading a secret into process memory for a single operation is acceptable and expected. Writing a secret to any file — even a temp file — requires that file to be in an approved storage location. A secret value written to a log, a debug output, or a cache file is an exposure incident.

5. **Production secrets do not enter sandbox or experimental environments.** This is not a preference — it is a hard rule. If a capability cannot be tested without production credentials, either the test is incorrectly scoped or a test credential must be provisioned. Reusing a production API key in an experiment to "save time" is a policy violation. (Source: D1.13 §4, §5)

6. **Human approves access scope.** The human decides which agent, on which machine, in which environment, may use a given secret. Agents propose. Humans approve. No agent may self-authorize broader credential access, even if it believes the task requires it. (Source: D1.1 §3 principle 15, D1.4 §3)

7. **Expired or suspected-leaked secrets are treated as compromised.** "It probably wasn't actually leaked" is not a valid operating assumption. If a secret was exposed — even briefly, even in a context that seems internal-only — it is rotated before it is used again. (Source: §11)

8. **Self-managed execution does not imply self-managed credentials.** The swarm manages its own task execution within approved boundaries. Credential scope, rotation, and activation are not execution responsibilities — they are governance responsibilities. They remain with the human regardless of how autonomous the swarm becomes. (Source: D1.1 §3 principle 15)

9. **A secret is not active until its governance is complete.** Obtaining an API key is not the same as activating it. Activation requires: registration, scope approval, storage in an approved location, and approval of the capability it unlocks (D1.12 §9.1). The key sitting in a notes file is not an active capability — it is an unregistered risk.

10. **Credential governance is a prerequisite for reusable workflows.** An approved workflow that depends on a credential cannot be executed reliably if that credential's governance is incomplete. Workflows do not carry credential values — they reference registered credential IDs. The credential management layer resolves the ID to the value at runtime, within scope. (Source: D1.13 §12)

## 4. Secret Classes

### 4.1 API Keys

**What it is:** A key issued by an external provider that authenticates the organization's requests. Revokable by the provider. Usually scoped to an account or project.

**Examples:** Anthropic API key, xAI key, OpenClaw provider key, any model API key, Perplexity API key (pending capability approval)

**Default risk level:** Risk Class 3 (High). Usage-based billing means exposure creates both data and financial risk.

**Environment scope:** Production only, unless a separate test key is provisioned by the provider with explicit test-tier billing caps.

**Perplexity credential class:** When Perplexity (D5.7 CAP-006) is approved for production use, its API key is governed as a Risk Class 3 API key under this section. The key must not be obtained or stored until D1.12 capability adoption is approved and a SEC-### entry is registered. The key is not a prerequisite for the evaluation experiment (TASK-013) — experiments may use the human's own authenticated session rather than a registered org-level key. A registered org-level key is required before Perplexity is used in production agent tasks (TASK-014).

### 4.2 Bot Tokens

**What it is:** A long-lived token that authenticates a bot user on a messaging or coordination platform. Allows the bot to send and receive messages, join channels, and take actions as the bot identity.

**Examples:** Telegram bot token (`HTTP API token` from @BotFather), Discord bot token

**Default risk level:** Risk Class 3 (High). A leaked bot token allows an attacker to impersonate the organization's agent communication channel.

**Environment scope:** Production only. Separate development bots (different bot accounts, different tokens) are used in dev/sandbox if bot testing is needed.

### 4.3 Platform Authentication Tokens

**What it is:** OAuth tokens, session tokens, or API credentials issued by platforms where the organization has an account. Often scoped to specific actions (read, write, publish, transact).

**Examples:** Commerce platform API credentials, social publishing tokens, listing platform auth

**Default risk level:** Risk Class 3–4 depending on whether financial transactions are possible. Any token with financial write access is automatically Risk Class 4.

**Environment scope:** Production only. Test/sandbox accounts at the platform level should be provisioned for non-production use.

### 4.4 Service Account Credentials

**What it is:** Username/password or key-based credentials for a service account at a third-party SaaS or infrastructure provider. Used by agents to authenticate to external services.

**Examples:** Cloud storage account, data provider login, analytics API account

**Default risk level:** Risk Class 2–3 depending on data exposure scope.

**Environment scope:** Scoped per environment. Production service account must not be used in sandbox.

### 4.5 Webhook Secrets / Shared Secrets

**What it is:** A shared secret used to validate that inbound webhook payloads originated from the expected source. Also includes HMAC signing keys for inter-service authenticity.

**Examples:** Stripe webhook signing secret, GitHub webhook secret, inter-service HMAC key

**Default risk level:** Risk Class 2. Exposure allows an attacker to forge inbound events.

**Environment scope:** Per endpoint. A webhook secret is tied to a specific registered endpoint — it does not cross environments.

### 4.6 Local Signing or Encryption Material

**What it is:** Asymmetric key pairs or symmetric encryption keys used for signing, verification, or data protection within the organization's own infrastructure.

**Examples:** SSH private keys for inter-machine access, GPG keys for signing commits or files

**Default risk level:** Risk Class 3. Private keys that control machine access or data encryption are high-consequence if exposed.

**Environment scope:** Machine-specific. An SSH key authorized for Mac Mini access is not reused for Desktop access.

### 4.7 Machine-Local Access Credentials

**What it is:** Credentials used to access a machine directly: local admin passwords, sudo credentials, screen sharing passwords, VPN credentials if applicable.

**Examples:** Mac Mini admin password, Screen Sharing password, local root credential on Linux machines

**Default risk level:** Risk Class 3–4 depending on machine role. Mac Mini (production gateway) is Risk Class 4. Dev machines are Risk Class 3.

**Environment scope:** Machine-specific. Not shared across machines.

## 5. Environment and Secret Boundaries

Every secret is tied to an environment. A secret authorized for one environment is not used in any other. This is enforced through registration scoping (§7) and is not a matter of agent judgment at runtime.

| Environment | Secret Policy |
|---|---|
| Production | Real credentials from real providers. Scoped to minimum necessary access. Human-approved. Registered and active. |
| Sandbox | Separate credentials wherever possible: test API keys with billing caps, separate bot accounts, sandbox-tier platform accounts. If no test credential exists, the capability stays inactive in sandbox. |
| Dev | Test-only credentials. No real billing, no real platform access. Local or mock auth where possible. |
| Experimental | No real credentials. If an experiment requires a real credential to function, the experiment is out of scope for the experimental environment. It must be moved to sandbox with a properly scoped credential. |

**Hard rule: production credentials are never used in experimental or dev environments.** This applies even if the experiment "only reads" from the API, even if it is "just for a quick test," and even if the production credential is the only one available. If sandbox and dev credentials do not yet exist for a given capability, that capability is not tested in experimental — it is tested in sandbox after the appropriate test credential is provisioned.

If an environment lacks properly scoped credentials for a required capability, the capability stays inactive in that environment. Agents do not improvise access by borrowing from a different environment's credential. (Source: D1.13 §4, §5)

## 6. Secret Ownership and Authority

| Action | Authority |
|---|---|
| Obtain a new credential from a provider | Human only |
| Register a credential in the secret registry | Human (with bridge tracking metadata) |
| Approve the scope of a credential (which agent, which machine, which env) | Human only |
| Approve activation of a credential | Human only, concurrent with capability approval (D1.12 §9.1) |
| Rotate a credential on schedule | Human (bridge surfaces reminder, human executes) |
| Revoke a credential immediately | Human (for voluntary revocation); provider (for forced revocation) |
| Widen the scope of a credential | Human only — treated as a new activation decision |
| Detect whether a credential is present, active, valid, or expired | Bridge and any agent within their task scope |
| Halt a workflow when a credential is missing or invalid | Any agent — required behavior, not optional |
| Report a credential anomaly | Any agent — required |
| Fetch a credential value from the approved store at runtime | The authorized agent, on the authorized machine, within the authorized environment only |
| Forward a credential value to another agent | Explicitly forbidden |
| Copy a credential to a new location | Explicitly forbidden without human approval |
| Use a credential outside its declared scope | Explicitly forbidden |

**The authority boundary is absolute.** An agent cannot self-authorize broader credential access because a task is blocked, because time pressure exists, or because it believes the risk is acceptable. Blocked workflows wait. Agents escalate. (Source: D1.8 §8 escalation trigger 9 — invalid credentials)

## 7. Secret Registration Policy

### 7.1 The Secret Registry

The secret registry is the authoritative record of all credentials the organization holds and uses. It lives in Mission Control at a designated path outside the `docs/` directory to reduce accidental exposure risk (path defined at Phase 1 setup — see §16 Priority 1).

**The registry tracks metadata only. Secret values are never written here.** An entry in the registry answers every governance question about a secret without ever containing the secret itself.

### 7.2 Required Metadata Per Registration Entry

Every registered secret must have the following fields populated before it may be activated:

| Field | Description |
|---|---|
| `secret_id` | SEC-### sequential |
| `name` | Short human-readable name (e.g. "Telegram bot token — bridge") |
| `class` | From §4 (api-key / bot-token / platform-auth / service-account / webhook-secret / signing-key / machine-local) |
| `provider` | Who issued it (e.g. Telegram, Anthropic, Stripe) |
| `authorized_environment` | production / sandbox / dev / experimental |
| `authorized_machine` | Specific machine(s) that may load this secret |
| `authorized_agent` | Specific agent(s) that may use this secret at runtime |
| `authorized_capability` | The CAP-### from D5.7 that this secret unlocks (or `machine-access` for machine-local) |
| `approval_date` | YYYY-MM-DD when human approved this registration and scope |
| `storage_location` | Where the value lives (approved store type — not the path — e.g. "Mac Mini OS keychain" or "protected env file") |
| `activation_status` | `registered` / `active` / `inactive` / `revoked` / `compromised` |
| `rotation_due` | YYYY-MM-DD — next scheduled rotation date |
| `last_rotated` | YYYY-MM-DD |
| `expiry_date` | YYYY-MM-DD if the provider imposes an expiry; null if not |
| `compromise_status` | `clean` / `suspected` / `confirmed` |
| `notes` | Any restrictions, known issues, or usage conditions |

### 7.3 Registration Is Required Before Storage

A secret must be registered (metadata committed to Mission Control) before its value is placed in any approved storage location. Registration is the approval record. Storage is the operational act that follows approval. Not the other way around.

### 7.4 Unregistered Secrets Are Non-Compliant

Any secret in active use that does not have a registry entry is non-compliant. If an agent detects a credential being used that has no registry entry, it flags the condition and surfaces it in the next daily digest. The capability that uses the unregistered secret is considered non-compliant until the secret is registered and approved.

## 8. Secret Storage Policy

### 8.1 Approved Storage Locations

Secret values may only be stored in one of the following approved forms, per machine:

| Machine | Approved Storage Forms |
|---|---|
| Mac Mini | macOS Keychain (preferred), or a protected environment file (not tracked by git, not in Mission Control workspace, permissions 0600) |
| Desktop | OS keychain if available; protected environment file with 0600 permissions — exact form confirmed at Phase 2 integration based on OS |
| GPU3060 | OS keychain if available; protected environment file with 0600 permissions — exact form confirmed at Phase 2 integration based on OS |
| MacBook Air | macOS Keychain for any machine-local credentials. Org service credentials do not reside here — this machine is not a worker node. |
| Pixel / Seeker | TBD at Phase 5. Device-native secure storage only. |

### 8.2 Where Secrets Are Explicitly Forbidden

Secret values may never be stored in:
- Any file in the Mission Control workspace (`~/Cursor Projects/mission-control/`)
- Any file tracked by git, regardless of location
- Any task record, proof artifact, or operational log
- Any Telegram message, Discord post, or other channel communication
- Any model prompt passed to a local or cloud LLM
- Any research packet or experiment record
- Any capability register entry (D5.7) or workflow library entry (D5.6)
- Any agent working memory that persists to disk beyond the task session (transient in-process use is acceptable)
- Environment variable files that are readable by other users or processes

### 8.3 One Storage Instance Per Secret

Each secret has exactly one authoritative storage location. If a secret is needed on multiple machines, it is registered with multiple machine authorizations and each machine stores its own copy in its own approved store — each copy is tracked in the registry as a separate instance of the same secret. Copies are not made informally.

### 8.4 Machine-Local Storage Must Match Environment Role

Secrets stored on a machine are consistent with that machine's declared environment role (D1.13 §5). Production credentials live only on machines whose primary environment is Production (Mac Mini, Desktop). Test credentials live only on machines whose primary or secondary environment is Sandbox or Dev (GPU3060, MacBook Air). A production credential has no business being in the GPU3060's storage.

## 9. Secret Exposure and Handling Rules

### 9.1 Explicit Prohibitions

The following are forbidden under all circumstances:

- Placing a secret value in a git-tracked file, even briefly, even in a branch, even "just to test"
- Including a secret value in a completion notice, alert, escalation, or digest message
- Including a secret value in a task record or proof artifact
- Including a secret value in a model prompt sent to any local or cloud model
- Logging a secret value in any system log, application log, or debug output
- Transmitting a secret value over any channel (Telegram, Discord, LAN, or any other)
- Printing or displaying a secret value in any context visible to system logs

A violation of any of these prohibitions triggers an immediate compromise response (§11.3), regardless of whether external exposure is believed to have occurred. Internal exposure is still exposure.

### 9.2 Safe Reference Conventions

Agents refer to secrets by their registry ID (`SEC-###`) or their registered name, not by value or partial value. When communicating about credential state, the acceptable language is:

- "SEC-001 (Telegram bot token) is active and within rotation window."
- "SEC-002 (Anthropic API key) is not loaded on this machine — the authorized machine is Mac Mini."
- "The credential required for CAP-003 is not registered. Halting task."

Agents do not write: "The key starts with `sk-ant-...`" or "The token I found is..." or any partial or hint-level disclosure. A partial secret is still a secret.

### 9.3 Error Reporting Without Leaking

When a credential-related error occurs, agents report:
- The registry ID of the affected secret
- The type of failure (not found / invalid / expired / access denied)
- The capability or service that was halted
- The escalation trigger fired (D1.8 §8 trigger 9)

Agents do not include provider error messages verbatim if those messages might echo credential hints, provider-side identifiers, or account information beyond what is necessary.

## 10. Secret Activation and Runtime Use

### 10.1 What "Active" Means

A secret is active only when all of the following are true:
1. It is registered in the secret registry with an entry in Mission Control
2. The human has approved its scope (authorized agent, machine, environment, capability)
3. The value is stored in an approved location on the authorized machine
4. The capability it unlocks is itself approved and active per D1.12 §9.1
5. The `activation_status` field in the registry reads `active`

A secret that meets conditions 1–4 but has not been explicitly activated (condition 5) is in `registered` status and may not be loaded by agents. The human activates it by setting the status field.

### 10.2 Runtime Load Rules

At runtime, when an agent needs to use a credential:
- The credential is loaded from the approved store only at the moment it is needed
- It is used for the specific operation and then cleared from working scope
- It is not stored in a variable that persists across task sessions
- It is not passed to another agent, sub-process, or tool beyond what the approved capability requires
- If loading fails (credential not found, access denied, store unavailable), the operation halts immediately and bridge escalates (D1.8 §8 trigger 9)

### 10.3 Scope Is Enforced at Load Time

When an agent loads a credential, it validates against the registry:
- Is this agent authorized to use this secret? (authorized_agent field)
- Is this machine authorized? (authorized_machine field)
- Is this environment the authorized environment? (authorized_environment field)
- Is the intended use within the authorized capability? (authorized_capability field)

A mismatch on any dimension means the load is rejected and the operation halts. The agent does not attempt to "work around" the scope restriction by loading from a different location or requesting the value from another agent.

### 10.4 Prerequisites for Activation Must All Be Met

If any activation prerequisite (§10.1) is unmet, the related service or capability stays inactive. There is no partial activation. A Telegram bot that cannot load its registered token does not fall back to another credential — it stays offline and bridge escalates. (Source: D1.13 §4.1)

## 11. Rotation, Revocation, and Compromise Policy

### 11.1 Scheduled Rotation

Secrets are rotated on a schedule based on their risk class. Bridge surfaces rotation reminders when the `rotation_due` date appears in the weekly digest (D1.11 §8.5). The human executes the rotation. Bridge updates the registry metadata after confirmation.

| Risk Class | Rotation Cadence |
|---|---|
| Risk Class 4 (Critical) | Every 90 days |
| Risk Class 3 (High) | Every 180 days |
| Risk Class 2 (Medium) | Every 365 days |
| Risk Class 1 (Low) | Annual or at capability retirement |

Rotation means: a new credential is issued by the provider, the old one is revoked, the new one is stored in the approved location, and the registry entry is updated with the new `last_rotated` and `rotation_due` dates.

### 11.2 Revocation Triggers

A credential is revoked immediately when:
- It is confirmed or suspected compromised (see §11.3)
- The capability it unlocks is retired or deactivated (D1.12 §13)
- The authorized agent, machine, or environment it was scoped to is decommissioned
- The human explicitly revokes it
- The provider reports the credential has been invalidated

After revocation: the `activation_status` is set to `revoked`, the value is removed from approved storage on all machines, and the capability it unlocked is halted until a replacement is registered and activated.

### 11.3 Compromise Response

**What counts as compromise:**
- A secret value appears in any git commit (even in a private repo, even briefly)
- A secret value is observed in any log, message, or output
- A secret value is transmitted over any channel
- Unusual activity is detected on an account protected by the secret
- A machine storing the secret is accessed by an unauthorized party
- The credential is used from an unexpected IP, time, or pattern

**Immediate response to compromise:**
1. The credential is treated as compromised the moment the incident is identified — not after investigation
2. Bridge escalates immediately with `compromise_status: suspected` (D1.8 §8 trigger 9)
3. The human revokes the credential at the provider side before any other action
4. The related capability halts and stays halted
5. The registry entry is updated to `activation_status: revoked`, `compromise_status: confirmed`
6. A replacement credential is provisioned, registered, scoped, stored, and approved before the capability reactivates
7. "Keep using it for now while we investigate" is not an acceptable operating procedure

**Secrets committed to git:** The git history is purged using appropriate tooling before the branch is merged into any tracked branch. The credential is rotated regardless of whether the repo is private. The assumption is that the value is compromised from the moment it enters git.

## 12. Agent Behavior Around Secrets

### 12.1 What Agents May Do

- Validate that a registered secret is present in the approved store (present/absent check — not reading the value)
- Confirm whether a secret is `active`, `inactive`, `revoked`, or `compromised` by querying the registry
- Load a secret into process memory to execute an approved operation within authorized scope
- Report a missing, invalid, or expired credential to bridge and halt the related workflow
- Propose that a new secret is needed for a capability candidate (identifying the need — not provisioning the credential)
- Surface rotation-due reminders from registry metadata in the weekly digest

### 12.2 What Agents May Not Do

- Read or display a secret value in any output, log, or message
- Forward a secret value to another agent, even as part of a task handoff
- Duplicate a secret to a different machine or storage location without human approval
- Self-authorize use of a credential outside its declared scope, even temporarily
- Use a production credential in a sandbox or experimental context
- Proceed with a workflow when a required credential is missing or invalid — the workflow halts, it does not improvise
- Attempt to refresh, rotate, or re-issue a credential from the provider
- "Borrow" a credential from another agent's workspace to unblock a task

### 12.3 Required Agent Behavior When Credentials Are Missing

When an agent encounters a missing, invalid, expired, or out-of-scope credential during task execution:

1. The task is halted immediately at that step
2. The task record is updated: `blocked_reason: credential-unavailable`, referencing the `secret_id`
3. Bridge receives an escalation notice (D1.9 §10.3 format, trigger type: credential)
4. The escalation states: what capability was attempted, which secret was required, what the credential state was, and what action is needed from the human
5. The agent does not retry, does not attempt an alternative credential, and does not downgrade to an unauthenticated path

This is not optional behavior. An agent that continues executing a task despite a credential failure has violated this policy. (Source: D1.8 §8 escalation trigger 9)

## 13. Logging, Audit, and Compliance Rules

### 13.1 What Is Logged

The following metadata is logged and must be answerable from logs at any time:
- When a secret was loaded (timestamp, secret_id, authorized_agent, authorized_machine)
- When a credential validation check was run and what the result was (present/absent/invalid/expired)
- When a credential-related escalation fired
- When a rotation was completed (new `last_rotated` date in registry)
- When a credential was revoked (timestamp, reason, revoked_by)
- When a compromise status was updated

### 13.2 What Is Never Logged

- Secret values, at any length or truncation
- Partial values or credential hints (first/last N characters)
- Provider-side account identifiers that could narrow a brute-force search
- Any field that would narrow the search space for a compromised credential

### 13.3 Audit Questions That Must Be Answerable

At any point, the human must be able to answer the following questions from Mission Control records without looking at the running machines:
- What secrets does the organization currently hold?
- Which machine and agent is each secret authorized for?
- When was each secret last rotated?
- Which secrets are due for rotation within the next 30 days?
- Which secrets are currently active vs. inactive vs. revoked?
- Is any secret currently flagged as suspected or confirmed compromised?
- Which capability does each secret unlock?

### 13.4 Compliance Check

Bridge performs a credential compliance check during its daily hygiene cycle:
- Any secret with `rotation_due` within 30 days → surfaced in daily digest as a notice
- Any secret with `rotation_due` in the past → escalated immediately
- Any secret with `compromise_status: suspected` → escalated immediately if not already escalated
- Any active capability in D5.7 whose required secret is not `activation_status: active` → flagged as misconfigured

The compliance check does not read secret values. It reads registry metadata only.

## 14. Phase-Based Secret Activation Policy

Secrets are activated in sequence with phases. Activating a credential before its phase is not permitted without explicit human approval and a documented justification. Earlier-phase restrictions hold regardless of organizational readiness unless specifically overridden.

### Phase 0 — Currently Active

Only machine-level access credentials needed for manual human administration:
- Mac Mini: SSH authorized key (human's public key), Screen Sharing credential
- No API keys, no bot tokens, no external service credentials

### Phase 1 — To Be Activated at Phase 1 Setup

**Required before Phase 1 is operational:**
- Anthropic API key (SEC-002 candidate): registers and activates when model provider is configured per D1.5. Risk Class 3.
- Telegram bot token (SEC-001 candidate): registers and activates when bot is configured per D1.11 §16 Priority 1. Risk Class 3.

**Optional at Phase 1 (explicit decision required):**
- Discord bot token: activates only if the human explicitly decides to configure Discord at Phase 1. Not activated by default. A Discord token is not registered until the Discord decision in D1.11 §16 Priority 3 is made. (Source: D1.11 §7.1)

**Explicitly inactive at Phase 1:**
- xAI / Grok key: remains inactive. D1.10's 5-condition activation gate is not satisfied (only condition 1 — D1.10 written — is currently met). No registration until the gate is satisfied. (Source: D1.10 §6)
- Any commerce or financial platform credential: no financial system credentials are registered or activated before their governing policies (Track 3 and Track 4 in D1.1 §9.5) are fully written and approved

### Phase 2 — Activated at Cluster Integration

- brain service credentials: registered when Desktop is integrated. The specific credentials depend on which external services brain requires — TBD pending capability acquisition decisions (D1.12).
- forge service credentials: same pattern as brain.
- Local LLM model access if any cloud model routing is involved (D1.5).

### Phase 3 and Beyond — Activated Per Capability Approval

Every additional secret beyond Phase 2 is activated only through the D1.12 capability acquisition process. No secret is registered or activated for a capability that has not been through D1.12 §6–§9. The phase of the capability determines the earliest phase the secret can be activated.

### Scope Creep Is Not a Phase

A capability approved at Phase 2 does not automatically get its credential scope widened at Phase 3. Each scope change (adding a machine, adding an agent, adding an environment) is a new approval decision. Phases describe availability windows, not automatic expansions.

## 16. Immediate Secret Priorities

### Before Phase 1 Runbooks Begin — Activation Pack Preparation

**0. Prepare the Activation Pack before executing any runbook.** Before running RB-01 through RB-05, the human should have gathered and confirmed: the Anthropic API key (will become SEC-002), the Telegram bot token (will become SEC-001), the Mac Mini hardware facts (for TASK-002), and any environment or tool-lane decisions made in advance. This is not a document committed to Mission Control — it is a human-held checklist used to execute the runbooks without mid-session interruption. Credential values flow directly from the human's secure storage into the Mac Mini keychain as each runbook step executes; they are never written to any intermediate file or message. See D2.0 Activation Pack Preparation section. (Source: D1.1 Principle 17, D2.0 §AP-00)

### Before Any Credential Is Registered

1. **Define the secret registry file path and commit the empty registry to Mission Control** — The registry must exist as a committed, tracked file (metadata only) before any secret is registered. The file lives in Mission Control at a path outside `docs/` to reduce accidental context inclusion risk. Suggested: `registry/secrets/SECRET-REGISTRY.md`. Create the file with the required field schema (§7.2) and no entries. This is the prerequisite for everything else. (Source: §7.1)

2. **Ensure `.gitignore` covers all approved secret storage paths** — Before anything is stored anywhere near the Mission Control workspace, confirm that the `.gitignore` in the Mission Control git repo explicitly excludes any path that could hold secret values. The registry metadata file is tracked. The values storage location is explicitly excluded. Review `.gitignore` before the first `git add`. (Source: D1.13 §8.3, §8.1)

### Phase 1 Setup (In Order)

3. **Register Anthropic API key metadata before loading the key** — Create the registry entry for SEC-002 (or whatever ID is assigned) with all required metadata fields. Set `activation_status: registered`. Human approves scope. Store the value in the Mac Mini's approved store. Then set `activation_status: active`. Run `openclaw configure` only after the registry entry exists. (Source: §10.1, D1.5)

4. **Create Telegram bot via @BotFather and immediately register the token metadata** — Do not store the token anywhere before the registry entry exists. The sequence is: create bot → receive token → create registry entry → store token in approved location → set active. The token is not held in notes or a messages app between bot creation and registry entry creation. (Source: §10.1, D1.11 §16 Priority 1)

5. **Make the Discord decision before any Discord token is obtained** — The question is: will Discord be configured at Phase 1? Per D1.11 §16 Priority 3, this is an explicit decision. If the answer is yes, a separate Discord bot is created with its own token, registered, and scoped. If the answer is no, no Discord token is obtained or registered at Phase 1. Do not obtain a token as a "just in case" — an unregistered token is an unmanaged risk. (Source: D1.11 §7.1, §7.3)

6. **Confirm D1.11, D1.9, and this document are consistent for Telegram primary channel** — The Telegram bot token activates the organization's primary coordination channel. D1.11 defines Telegram as primary. D1.9 defines the message routing. This document governs the credential. Before the bot goes live, verify that the token's registered scope (authorized_agent: bridge, authorized_machine: Mac Mini, authorized_environment: production) aligns with D1.9's message routing rules and D1.11's channel hierarchy. (Source: D1.11 §6, D1.9 §8)

7. **Confirm xAI / Grok key remains inactive and unregistered** — No registration entry is created for an xAI key until D1.10's 5-condition activation gate is satisfied. Document this decision in the registry as a declined/pending entry if desired, but no value is obtained or stored. (Source: D1.10 §6)
