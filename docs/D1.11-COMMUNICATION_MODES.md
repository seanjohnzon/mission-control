# D1.11 — COMMUNICATION_MODES
**Autonomous AI Organization — Channel Hierarchy, Message Types, Operating Rhythms, and Noise Control**
_Last updated: 2026-03-11 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document defines how communication works across the organization. It governs which channels exist, what each is for, what message types the human receives, how the swarm communicates internally, and how operating rhythms produce useful signal without creating noise.

The goal is a communication layer that feels like a well-run company: agents self-manage approved work and surface what matters, the human sees meaningful updates without having to dig, and the system stays quiet when there is nothing worth saying.

**This document governs:**
- Channel hierarchy and the role of each channel
- How Mission Control, Telegram, and Discord relate to each other
- What message types the human receives and their format constraints
- How agent-to-agent communication works internally
- Operating rhythms for daily, weekly, and sprint cadences
- How self-managed work is communicated (not just directed work)
- Escalation and alert communication standards
- Digest and summary design
- Rules for preventing clutter and noise

**What this document does not cover:**
- Channel setup steps — D2.1-BRIDGE-RUNBOOK.md
- Secrets and tokens for Telegram or Discord bots — D1.14-SECRETS_POLICY.md
- Broker routing mechanics — D1.9-BROKER_POLICY.md
- Research-specific digest format — D1.10-X_RESEARCH_POLICY.md §11
- Memory recording of events — D1.7-MEMORY_POLICY.md

This policy covers all channels through which the organization communicates:

## 3. Communication Principles

1. **Telegram is the operating room.** Day-to-day coordination, directives, completions, alerts, and digests happen in Telegram. This is where the human manages the swarm and where the swarm surfaces what matters.

2. **Mission Control is the record.** Task state, policies, decisions, workflows, memory, and org structure live in Mission Control. Mission Control is read and written by agents and the human — it is not a chat. Nothing important lives only in a message; it is committed to Mission Control.

3. **Discord is technical and optional.** Discord may carry technical output, build logs, system alerts for developer-level visibility, or integration-specific notifications. It is not where the human manages the swarm day-to-day. At Phase 1, Discord may not be used at all.

4. **The swarm communicates completion, not activity.** Agents do not report that they are working. They report that they are done. In-progress messages exist only for declared long-running tasks at defined intervals. Silence means the swarm is working as expected.

5. **Human-facing messages are calm and company-like.** No jargon, no stack traces, no model names, no internal technical state. Messages read like briefings from a capable team: clear, relevant, action-oriented.

6. **Self-managed work surfaces the same as directed work.** When the swarm picks up and completes a backlog task without a human directive, the human receives the same completion notice format. The distinction — agent-initiated vs human-directed — is visible but not intrusive.

7. **Technical detail is always available, never pushed.** Every completion notice and alert references the Mission Control record or proof artifact. The human reads the summary in Telegram and can drill into the detail if they want it.

8. **One channel per message class.** Each message type has one default destination. Messages do not simultaneously go to Telegram and Discord unless there is an explicit reason. Duplication across channels creates noise.

9. **No channel is the source of truth.** Telegram messages, Discord posts, and email notifications are all transient. The source of truth for every decision, task, and event is Mission Control. Channels deliver information; they do not store it.

10. **Human-facing output is tool-agnostic.** Whether a task result was produced by an agent running in OpenClaw, generated via Cursor, or retrieved through Perplexity (when activated), the human-facing message format is identical: clean, company-like, no tool names, no internal details. The tool that produced the output is not surfaced in messages unless the human explicitly queries for it. Tool lanes are an execution concern, not a communication concern.

## 4. Channel Hierarchy

The channel hierarchy determines which channel is used by default for each communication purpose.

```
Tier 1 — Mission Control (authoritative record)
    ↓ agents write; humans read; no chat

Tier 2 — Telegram (primary coordination)
    ↓ directives, completions, alerts, escalations, digests
    ↓ human ↔ bridge, daily operating rhythm

Tier 3 — Discord (secondary / technical)
    ↓ technical output, optional developer-facing notifications
    ↓ not the operating room; not required at Phase 1

Tier 4 — LAN (internal, Phase 2+)
    ↓ agent-to-agent task packets, not human-facing

Tier 5 — Edge receipts (Phase 5)
    ↓ Pixel/Seeker → bridge → operational memory
```

When there is doubt about which channel a message belongs in, the answer is Telegram for human-facing messages and Mission Control for record-keeping. Discord is used when there is a specific reason, not by default.

## 5. Mission Control Communication Role

Mission Control is not a communication channel. It is the authoritative organizational record. Understanding the distinction matters:

**What Mission Control holds:**
- Task state (backlog, in-progress, blocked, completed)
- Policies (D1.x series)
- Workflows (D5.6)
- Memory entries (via Archivist, as defined in D1.7)
- Org structure, decision rights, autonomy register (D1.4)
- Experiment registry, legacy asset register, skill registry (D5.x)

**How Mission Control is accessed:**
- Agents write task state changes and proof artifact references to it
- Agents read policies, workflows, and canonical memory from it
- The human reads it directly via Mission Control interface (browsing, search)
- The human does not receive Mission Control content pushed to them as messages — they pull it when they need it

**What Mission Control is not:**
- Not a chat interface
- Not the place where the human gets briefed
- Not a log of agent conversations
- Not a notification system

Every Telegram message that summarizes something should reference the Mission Control record or proof artifact where the full detail lives.

## 6. Telegram Communication Role

Telegram is the primary coordination layer. It serves the same function that Slack or a company messaging tool serves: day-to-day operating visibility, fast directives, and timely results.

### 6.1 What Telegram Carries

| Message Type | Direction | Default |
|---|---|---|
| Human directives | Human → Bridge | Primary channel for instructions |
| Directive acknowledgements | Bridge → Human | Immediate, in Telegram |
| Task completion notices | Bridge → Human | After every completed task |
| Self-managed completion notices | Bridge → Human | Same format as directed; labelled as agent-initiated |
| Alerts (non-critical) | Bridge → Human | In Telegram; does not require immediate response |
| Escalations | Bridge → Human | In Telegram; requires response before automation continues |
| Daily digest | Bridge → Human | Scheduled; one message per day |
| Weekly summary | Bridge → Human | Scheduled; one message per week |
| Research signal notices (High Signal) | Bridge → Human | Per D1.10 §11.2 |
| Workflow candidate notices | Bridge → Human | Batched into weekly digest unless high-priority |
| Backlog hygiene flags | Bridge → Human | Batched into daily digest |

### 6.2 What Telegram Does Not Carry

- Internal agent-to-agent task packets
- Raw log files or stack traces
- Heartbeat confirmations
- Intermediate model reasoning
- Every task state change (only completions and escalations generate messages)
- Duplicate notices for the same event

### 6.3 Telegram Channel Structure

At Phase 1, a single Telegram chat between the human and the bridge bot is sufficient. One channel, all operating communication.

As volume grows, a second channel for escalations-only may be added. This is a Phase 2 decision, made when Phase 1 volume data justifies it. Not before.

Channel names, bot token, and chat IDs are registered in D1.14-SECRETS_POLICY.md.

## 7. Discord Communication Role

Discord is secondary and optional at Phase 1. It is not the primary operating room.

### 7.1 When Discord Is Used

Discord may be used for:
- Technical output that a developer-audience wants (build logs, system diagnostics, detailed error reports)
- Integration-specific notifications where Telegram is not the natural target
- Community-facing content if the organization ever develops a public presence

At Phase 1, the default is to not configure Discord at all unless there is a specific justified need. Setting up Discord for its own sake creates maintenance overhead without benefit.

### 7.2 If Discord Is Configured

If Discord is configured, the same message class rules from D1.9-BROKER_POLICY.md apply. Discord is not a second operating room — it carries a subset of message types defined for that channel. The human does not manage the swarm from Discord.

### 7.3 Discord Is Not a Fallback

If Telegram is unavailable, Discord is not automatically promoted to primary and is not a fallback. The degraded mode policy in D1.9 §15 governs what happens when the primary channel is unreachable.

Discord may serve as an explicitly approved emergency secondary path only if all of the following are true:
- It is configured for that purpose
- That configuration is documented in D1.9 and D1.14
- The human has explicitly authorized its use in that degraded role

Absent explicit authorization and documentation in both D1.9 and D1.14, Discord is not used when Telegram is unavailable. "Discord is configured" does not imply it is authorized as an emergency path.

## 8. Human-Facing Message Types

These are the message types the human receives through Telegram. Each has a defined format and defined constraints.

### 8.1 Completion Notice
Sent when a task closes — whether human-directed or agent-initiated.

```
✓ [Task name]
Result: [1–2 sentences — what was produced]
Proof: [MC path or artifact reference]
Initiated by: [human / agent]
```

No elaboration. No technical detail. If the human wants the detail, they follow the proof reference.

### 8.2 Alert (Non-Critical)
Sent when a system condition warrants human awareness but does not halt automation.

```
⚠ [Condition name]
What: [1 sentence description]
Impact: [none / minor / moderate — specify]
Action needed: no / [specify if yes]
```

Alerts that do not require human action are batched into the next daily digest if possible. Only alerts that warrant immediate awareness are sent in real time.

### 8.3 Escalation (Requires Human Response)
Sent when automation is halted and waiting for a human decision. Never batched. Always sent immediately.

```
🔴 Decision needed: [Trigger type]
What stopped: [Task name and ID]
Why: [1 sentence — the blocking condition]
Options: [A] [action] / [B] [action] / [C] close task
Respond to continue.
```

The escalation remains visible and is re-sent at defined intervals (D1.9 §10.3) until acknowledged.

### 8.4 Daily Digest
One message per day summarizing the operating period. Sent on a declared schedule. Suppressed if there is nothing meaningful to report.

```
Daily summary — [date]
Completed: [N tasks] — [names or "see MC for details"]
In progress: [N tasks]
Blocked: [N tasks] — [brief reason each]
Alerts: [N] — [1-line each if any]
Backlog hygiene: [N flags] — [1-line summary or "none"]
Workflow candidates: [N] — [1-line each or "none"]
```

### 8.5 Weekly Summary
One message per week with broader operating context. Sent at end of week.

```
Weekly summary — week of [date]
Completed: [N tasks] total
Sprint progress: [N] of [N] planned tasks completed
Self-managed pickups started: [N]
Self-managed completions: [N]
Escalations: [N resolved] / [N outstanding]
Experiments proposed: [N]
Memory archived: [N entries]
Next week's backlog: [N tasks eligible]
```

### 8.6 Research Signal Notice (High Signal)
Per D1.10 §11.2. Sent per-finding for High Signal items only. Others batched into the daily or weekly digest.

### 8.7 Workflow Candidate Notice
Sent when the swarm identifies a recurring pattern and proposes formalizing it into D5.6.

```
Workflow candidate: [pattern name]
Pattern: [N] similar executions in [N] days
Consistency: [high / moderate / low]
Proposed workflow ID: WF-[draft ID in D5.6]
Action: Approve / Modify / Decline
```

## 9. Agent-to-Agent Communication Modes

Agent-to-agent communication is internal. It does not appear in Telegram or Discord. The human sees only the outcome (a completion notice), not the internal exchange.

### 9.1 Task Packets (Bridge ↔ Agent)

The primary internal communication mode. Bridge (Conductor) issues task packets to brain or forge. Agents return result packets to bridge. The packet format is defined in D1.8 §2 and D1.9 §8.

Task packets travel through bridge — brain and forge do not issue packets to each other directly.

### 9.2 Memory Promotion Packets (Agent → Archivist)

When bridge or forge has a completed task record to commit to canonical memory, it submits a memory promotion packet to brain (Archivist). The format is defined in D1.7 §9. This is the only cross-agent communication that does not require a human directive — it is part of the self-managed operating loop.

### 9.3 LAN Routing (Phase 2+)

When Desktop and GPU3060 are online, task packets travel over LAN. The message format and authentication are defined in D1.9 §13 and D1.14. Until Phase 2, there is no LAN inter-agent communication.

### 9.4 No Side-Channel Communication

Agents do not communicate outside of defined task packets and memory promotion packets. No shared files used as message queues, no direct sockets between brain and forge, no agent reads another agent's state without a declared access path. (Source: D1.9 §8.3, D1.4 guardrail 6)

## 10. Operating Rhythms

Operating rhythms define when things happen without the human having to ask. These are the swarm's heartbeat at the human-visible layer. Internal execution cadence is a bridge scheduling concern; this section defines the communication cadences.

### 10.1 Daily Rhythm

| Time | Event | Channel |
|---|---|---|
| Declared digest time (TBD) | Daily digest assembled and posted | Telegram |
| Throughout day | Completion notices as tasks close | Telegram |
| Immediately when triggered | Escalations | Telegram |
| Immediately when triggered | Critical alerts | Telegram |

The digest time is set by the human during Phase 1 setup. Default suggestion: end of operating day. Exact time is registered as a cron definition in Mission Control's automation register (D1.8 §9.1).

### 10.2 Weekly Rhythm

| Day | Event | Channel |
|---|---|---|
| Declared weekly review day (TBD) | Weekly summary posted | Telegram |
| Same day | Workflow candidates surfaced (if any) | Telegram |
| Same day | Research digest posted (if X research is active) | Telegram |

### 10.3 Sprint / Planning Rhythm

Sprint cadence is TBD (see §16). When defined, it adds:
- Sprint start: bridge posts next sprint's planned tasks to Telegram
- Sprint close: weekly summary serves as sprint retrospective summary
- Mid-sprint backlog review: if bridge identifies capacity, it self-selects next eligible backlog items without waiting for a human sprint planning session

The human sets sprint goals and priorities. Bridge manages execution within them.

### 10.4 Event-Driven (No Cadence)

| Event | Channel | Timing |
|---|---|---|
| Task completes (any) | Telegram | Immediately after close |
| Escalation fires | Telegram | Immediately |
| System enters DEGRADED state | Telegram | Immediately |
| Backlog exceeds high-water mark | Telegram digest | Next digest cycle |
| New workflow candidate (high-priority) | Telegram | Next digest cycle (immediate only if human-requested urgency) |

## 11. Sprint / Backlog Communication Rules

The swarm is self-managed within approved boundaries. This creates specific communication needs that do not exist in purely human-directed systems.

### 11.1 Self-Managed Pickup Visibility

When bridge picks up a backlog task without a human directive, the human learns about it through:
- The task's completion notice (labelled `Initiated by: agent`)
- The daily digest (count of self-managed completions)
- Mission Control (task record shows `initiated_by: conductor`, not a human)

The human does not receive a notification at the moment of pickup — only at completion. This prevents a flood of "I just started working on X" messages.

### 11.2 Backlog Hygiene Communication

Backlog hygiene issues (stale tasks, missing fields, blocked without a recorded reason) are surfaced in the daily digest, not as real-time alerts. The exception is a task that has been blocked for more than a declared threshold (TBD) — that escalates.

Format in the daily digest:
```
Backlog hygiene: [N] flags
- [Task name]: stale / missing field / blocked [N] days — [brief]
```

### 11.3 Blocked Task Communication

When a task becomes blocked:
1. Bridge marks it `blocked` in Mission Control with a reason
2. If the block requires human input: escalation is sent immediately to Telegram
3. If the block is self-resolvable (waiting on another task): bridge holds it and the next digest includes the blocked count

Blocked tasks are not nagged — the human sees them in the digest unless they require a decision, in which case they escalate.

### 11.4 Estimation and Sequencing

When bridge sequences backlog tasks (deciding what to pick up in what order), it does not request human approval per sequence decision. It follows the priority order in Mission Control (set by the human) and the automation rules in D1.8. If there is ambiguity about priority, bridge picks the highest-priority unambiguous task and surfaces the ambiguous ones in the next digest.

## 12. Escalation and Alert Communication

### 12.1 Alert Levels

| Level | Name | Description | Channel | Timing |
|---|---|---|---|---|
| 0 | Info | Routine completion or status. No action needed. | Telegram (digest) | Batched |
| 1 | Notice | Condition worth knowing; no immediate action required | Telegram (digest or real-time if time-sensitive) | Digest or real-time |
| 2 | Warning | System condition that may require attention soon | Telegram (real-time) | Immediate |
| 3 | Escalation | Automation halted; human decision required | Telegram (real-time) | Immediate |
| 4 | Critical | System integrity or irreversibility risk | Telegram (real-time) | Immediate |

### 12.2 Alert Batching Policy

Level 0 and 1 alerts are batched into the daily digest unless:
- The human has declared specific conditions that should always be real-time
- The alert is time-sensitive (e.g. an opportunity that expires)

Level 2 and above are always real-time. They are never held for a digest.

### 12.3 Escalation Acknowledgement

Escalations require explicit human acknowledgement in Telegram before halted automation may resume. The acknowledgement must address the decision requested — not just confirm receipt. Bridge re-sends at defined intervals (D1.9 §10.3) until acknowledged.

### 12.4 Cascading Escalations

If multiple escalations fire within a short window (e.g. machine goes DEGRADED and several tasks halt simultaneously), bridge sends one combined escalation notice listing all affected tasks, not one notice per task. The human acknowledges once; bridge resumes all affected automations.

## 13. Digest and Summary Communication

### 13.1 Design Principles for Digests

- A digest that has nothing to report is suppressed. An empty digest is noise.
- A digest covers the period since the last digest of the same type. It does not overlap with or duplicate prior digests.
- Digests do not contain raw data — they contain summaries. Detail is available in Mission Control.
- The human can act on the digest or ignore it. A digest never requires a response unless it contains an escalation item that was missed.

### 13.2 Daily Digest Structure

See §8.4 for the format. The daily digest is the primary vehicle for batched info: completions that are routine, non-critical alerts, backlog hygiene flags, and minor workflow signals. Its purpose is to give the human a fast read on what happened without requiring them to pull up Mission Control.

### 13.3 Weekly Summary Structure

See §8.5 for the format. The weekly summary is the sprint-level view: how much got done, how much was self-managed vs directed, what's coming, and whether anything needs strategic attention.

### 13.4 Research Digest

Per D1.10 §11.3. Batches Moderate Signal research findings from the X research lane. Sent weekly alongside the weekly summary, or on a separate declared cadence if research volume justifies it.

### 13.5 Digest Timing

All digest types have a declared send time registered in Mission Control's cron register. The human sets these times during Phase 1 setup. Default cadence:

| Digest | Default Cadence | Default Time |
|---|---|---|
| Daily digest | Daily | End of operating day (TBD) |
| Weekly summary | Weekly | End of week (TBD) |
| Research digest | Weekly | Same time as weekly summary |

Timing is adjusted by updating the cron registration — not by modifying this document.

## 14. Noise / Clutter Control

Clutter is the primary enemy of a useful communication layer. The rules below are enforced at the Broker level (D1.9) and define what is never sent regardless of what agents produce.

### 14.1 Messages That Are Never Sent

The following are never sent to any human-facing channel under any circumstances:

- Heartbeat confirmations ("Mac Mini is online," "gateway is running")
- In-progress status messages for tasks that have not declared a progress interval
- Raw model outputs or reasoning traces
- Internal task packet contents
- Stack traces or raw error objects
- Every task state transition (only `completed` and `escalation` generate messages)
- Duplicate notices for the same event within a single digest period
- Empty digests (suppressed)

### 14.2 Volume Cap

Telegram receives a maximum of N individual real-time messages per hour during normal operation (N is calibrated during Phase 1 and recorded in Mission Control's automation register). If more would be generated, non-critical messages are held for the next digest. Escalations and critical alerts always break the cap.

### 14.3 Drill-Down, Don't Push Down

Every Telegram message should be actionable or informational at a glance. Technical detail, full task output, and supporting evidence live in Mission Control or the proof artifact. The message contains a reference to the detail — not the detail itself. The human drills in when they want to, not because the message forced them to read everything to understand it.

### 14.4 Calm Language Standard

Messages from the swarm use plain, direct language:
- No urgency theater ("CRITICAL ALERT: something you should know...")
- No excessive emoji or visual noise
- No corporate filler ("I hope this finds you well...")
- No hedging on what happened ("it seems like the task may have completed...")
- Statements, not speculation

If something went wrong, say what went wrong. If something was completed, say what was completed. If a decision is needed, say what the decision is.

## 16. Immediate Communication Priorities

### Phase 1 — before agents are fully operational

1. **Write D1.14-SECRETS_POLICY.md and register Telegram bot token** — No coordination channel works without credentials. D1.14 must be written before any bot is configured. (Source: §6.3)

2. **Configure Telegram bot and test directive → acknowledgement → completion cycle** — The primary coordination channel must be verified working before Phase 1 is considered operational. Test: human sends directive via Telegram; bridge acknowledges within 30 seconds; task completes; completion notice appears in Telegram with correct format. (Source: D1.9 §10.1, §8.1)

3. **Decide whether to configure Discord at Phase 1** — Make an explicit decision: use Discord for a specific justified purpose, or defer it entirely. Defaulting to "we'll set it up eventually" creates deferred overhead. (Source: §7)

4. **Set daily digest time and weekly summary day** — Register both in Mission Control's cron definition file before the first digest is expected. The human should not have to think about when briefings arrive — they should just appear. (Source: §13.5, D1.8 §9.1)

5. **Set the per-hour message volume cap after the first week** — Do not set it before operating. Set it based on what the first week of Phase 1 communication actually produces. Record it in the Mission Control automation register. (Source: §14.2)

### Phase 2 — after cluster is online and first sprint begins

6. **Define sprint cadence** — Set the sprint length, planning rhythm, and how sprint goals are communicated in Mission Control and surfaced in the weekly summary. This does not need to happen before Phase 1. (Source: §10.3)

7. **Test weekly summary format with real data** — The first weekly summary is generated from Phase 1 operation. Review it with the human. Adjust the format if it is too dense, too sparse, or missing something useful. (Source: §8.5)

8. **Define Telegram structure for Phase 2 volume** — After Phase 1, assess whether the single Telegram channel is sufficient or whether a second escalations-only channel is warranted. Make this decision with Phase 1 data, not speculation. (Source: §6.3)
