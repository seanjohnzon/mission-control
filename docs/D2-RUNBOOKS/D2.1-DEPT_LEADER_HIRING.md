# HIRING PROCESS — New Agent Onboarding

## DRAFT — Captain Review Before Applying

No agent joins the crew without a clear role, scope, and proper documentation. This prevents rogue agents that "go crazy and delete files."

---

## When To Hire
- Department leader identifies they need help (workload analysis)
- OR Nami/Jinbe identify a bottleneck (e.g. Chopper QA is the slowest link)
- OR new capability needed that no current agent covers

## Hiring Checklist — MUST complete ALL before agent goes live

### 1. Role Definition
- [ ] **Name** — One Piece character that fits the department
- [ ] **Department** — which leader owns this agent
- [ ] **Scope** — exactly what this agent does (and does NOT do)
- [ ] **Model** — what model runs this agent (budget-appropriate)
- [ ] **Justification** — why do we need this hire? What bottleneck does it solve?

### 2. Document Setup (BEFORE first session)
- [ ] **SOUL.md** — personality only, ~30 lines, matches the character
- [ ] **AGENTS.md** — session protocol, scope, rules, department structure
  - Must include: session start protocol (read task first)
  - Must include: session end protocol (timestamped comment)
  - Must include: one task = one session rule
  - Must include: what they do NOT do (scope boundaries)
- [ ] **TOOLS.md** — MC API endpoint, relevant Telegram topics, environment data
- [ ] **HEARTBEAT.md** — what to check on heartbeat (fallback behavior)
- [ ] **active-goals.md** — starts empty

### 3. Config Setup
- [ ] Agent added to openclaw.json (model, workspace, agentDir)
- [ ] Agent added to parent department's subagents.allowAgents list
- [ ] Agent added to agentToAgent allow list
- [ ] Workspace directory created with proper structure
- [ ] Skills installed (self-improving-agent + department-specific skills)

### 4. Scope Verification
- [ ] Department leader confirms the agent's scope doesn't overlap with existing agents
- [ ] Usopp reviews: does the scope align with story requirements?
- [ ] Nami approves: budget impact acceptable?

### 5. Test Run
- [ ] Agent responds to a simple task (prove it works)
- [ ] Agent correctly reads MC board for assigned tasks
- [ ] Agent posts session-end comment properly
- [ ] Agent stays within its scope (doesn't touch files outside its workspace)

### 6. Register
- [ ] Added to Teams Dashboard (crew-teams-dashboard.html)
- [ ] Added to TOOLS.md crew roster (if dept leader level)
- [ ] MC task created documenting the hire: role, justification, who approved

## Model Budget Guide
| Tier | Model | Cost | Use For |
|------|-------|------|---------|
| Free | Opus/Sonnet | $0 (subscription) | Anthropic agents — Nami, Jinbe, Zoro, Vivi, Zeus, Carina |
| Free | GPT-5.4 | $0 (subscription) | OpenAI agents — Franky, Sanji, Chopper, Usopp + their subs |
| Cheap | Haiku 3.5 | $0 (subscription) | Simple patrol/scan agents — Fisher-Tiger, Tashigi, etc |
| Variable | Grok 3 Fast | ~$0.01/task | Robin's sub-agents |
| Variable | Gemini Flash | ~$0/task (usually free) | Brook's sub-agents |
| Expensive | Grok 4 | ~$0.10/task | Robin only |
| Expensive | Gemini 2.5 Pro | ~$0.05/task | Brook only |

**Rule:** New hires should use subscription models (Opus/Sonnet/GPT-5.4/Haiku) whenever possible. Variable-cost models only for specialized roles that need specific capabilities (real-time search, image gen, etc).

---

## Firing / Decommissioning
If an agent is:
- Consistently failing tasks
- Out of scope with no clear role
- Redundant with another agent
- Burning budget without output

Then:
1. Disable in config (don't delete — we might re-hire)
2. Archive workspace
3. Remove from Teams Dashboard
4. Log the decommission on MC board
