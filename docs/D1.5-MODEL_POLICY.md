# D1.5 — MODEL_POLICY
**Autonomous AI Organization — Model Selection, Routing, and Cost Policy**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md | Org: D1.4-ORG_MAP.md_

---

## 1. Purpose

This document defines the policy for model selection and inference routing across the organization. It answers which model runs where, when cloud inference is permitted, how routing decisions are made, and what governs cost and usage.

**What this document does:**
- Defines model classes by capability tier
- States the routing policy (local-first, cloud-fallback)
- Maps task types to appropriate model classes
- Governs cost, usage, and cron/heartbeat behavior
- Sets per-agent model policy for bridge, brain, and forge
- Defines the conditions under which cloud inference is allowed
- Marks decisions that are still open

**What this document is not:**
- Not a setup guide or install script (see D2.x Runbooks)
- Not a local inference plan — hardware constraints and benchmarks are in D1.6-LOCAL_INFERENCE_PLAN.md
- Not a secrets or credentials policy (see D1.14-SECRETS_POLICY.md)
- Not a broker routing spec — inter-agent message routing is in D1.9-BROKER_POLICY.md

When a model is added, changed, or retired for any agent, this document is updated first. Runbooks that reference specific model configurations must be flagged for review when this file changes.

## 3. Policy Principles

These principles govern every model decision in the organization. They are not preferences — they are rules. Deviations require explicit human approval and must be documented.

1. **Local inference is the default.** Every worker machine (Mac Mini, Desktop, GPU3060) must run at least one local model. Local inference is attempted first for every task. Cloud inference is not the silent default.

2. **Cloud is fallback, not supplement.** Cloud inference is used when local inference cannot satisfy a task — due to capability limits, availability, or context size. It is not used simply because it is convenient or faster.

3. **No model-hopping mid-task.** A task must be assigned a model class before it begins. The model does not change during execution unless the task explicitly fails and falls back. Switching models mid-task without a failure trigger is not permitted.

4. **Recurring low-value work runs local.** Cron jobs, heartbeats, status checks, and recurring summaries MUST use local inference. These tasks MUST NOT burn cloud API tokens on a schedule.

5. **High-value irreversible tasks may use cloud.** Tasks that produce outputs a human will review and act on — a plan, a commerce recommendation, a financial flag — may use a higher-tier model including cloud, if the local model cannot meet the quality bar.

6. **Cost visibility is required.** Cloud token usage must be logged and reviewable. Invisible cost accumulation is not acceptable. Agents do not call cloud APIs in loops or without a defined output target.

7. **Model policy is written before model config.** No agent is configured with a model API key until this policy is approved. The policy comes first; the key is entered after.

8. **Experimental routing is opt-in.** Proxy routing, multi-model ensemble approaches, and similar techniques are permitted only in a designated experimental context (see §13). They are not used in production task execution without explicit policy addition here.

## 4. Model Classes

Model classes are capability tiers, not specific models. Specific model selection within a class is decided per agent in §9–11 and in D1.6. Classes allow the policy to remain stable when specific model versions change.

| Class | Tier | Capability Profile | Typical Use |
|---|---|---|---|
| **Local-Nano** | Low | Small models (1–4B parameters). Fast. Low RAM. Limited reasoning depth. | Routing decisions, keyword extraction, simple classification, status checks |
| **Local-Mid** | Medium | Mid-range models (7–14B parameters). Balanced speed and capability. | Task summaries, research extraction, code scaffolding, structured output generation |
| **Local-High** | High | Large local models (30B+ parameters). Slower. Higher RAM requirement. GPU required. | Complex planning, code review, multi-step reasoning, high-quality content generation |
| **Cloud-Standard** | Cloud | Provider API, standard tier (e.g. Anthropic claude-haiku-4-5). Moderate cost. | Overflow from local, tasks requiring recency or web-augmented retrieval |
| **Cloud-Premium** | Cloud | Provider API, premium tier (e.g. Anthropic claude-sonnet-4-6, claude-opus-4-6). Higher cost. | High-stakes planning, financial analysis, escalation-triggered tasks, human-review outputs |
| **Research-External** | External | xAI/Grok or equivalent research lane. Not for production routing. | Research intake only (see §14). Not activated at Phase 1. |

**Rule:** Local-Nano and Local-Mid classes must be available on Mac Mini before Phase 1 exits. Local-High class must be available on GPU3060 before any forge inference task is assigned.

## 5. Routing Policy

The routing policy defines the decision tree for model selection at task execution time. This tree is applied by bridge (Conductor) when dispatching a task, or by the executing agent when starting work.

```
Task received
      │
      ▼
Is a model class assigned in the task packet?
      │
      ├── YES → Use the assigned class. Apply agent-specific model within that class (§9–11).
      │
      └── NO → Assign class based on task type map (§6).
                     │
                     ▼
              Is local inference available for the assigned class on this agent's machine?
                     │
                     ├── YES → Execute locally. Log model used and token estimate.
                     │
                     └── NO → Is cloud fallback permitted for this task type? (§12)
                                    │
                                    ├── YES → Use cloud fallback. Log provider, model, and token count.
                                    │          Post escalation note to Telegram if cost threshold exceeded.
                                    │
                                    └── NO → Mark task BLOCKED. Escalate to human.
                                             Do not execute.
```

### Routing Constraints

- A task marked as requiring local-only MUST NOT fall back to cloud even if local is slow or unavailable.
- A task that fails local inference due to context size overflow MAY fall back to cloud if the task type permits it.
- Bridge (Conductor) must log every routing decision: which class was selected, which model was used, and whether it was local or cloud.

## 7. Cost / Usage Policy

### Approved Cloud Providers

| Provider | Status | Use |
|---|---|---|
| Anthropic (Claude API) | Approved — Phase 1 | Primary cloud fallback lane |
| xAI / Grok | Not activated | Research lane only — see §14 |
| OpenAI | Not approved | Not part of current policy. Requires explicit human approval before use. |
| Other providers | Not approved | Same. Requires human approval and policy update before any use. |

### Cost Controls

1. **No API key is entered on any agent until this policy is approved.** (Source: D1.4 §15 item 1)
2. **Cloud usage is logged per task.** Every cloud API call must emit a log entry with: task ID, model used, estimated token count, and reason for cloud routing.
3. **Recurring tasks MUST NOT use cloud.** Cron jobs, scheduled digests, and heartbeats are local-only. A cron task that silently burns cloud tokens is a policy violation.
4. **A cost threshold alert is sent to Telegram (ops-channel) when accumulated cloud spend in a session exceeds a defined limit.** Threshold TBD — set before any cloud key is activated. Logged in D1.14-SECRETS_POLICY.md alongside key management.
5. **No cloud API call is made in a loop without a defined completion condition.** Unbounded cloud calls are a policy violation regardless of context.
6. **Usage is reviewed at each phase boundary** by the human operator. If cost is disproportionate to output, routing decisions are revised.

### Token Budget Guidance (Non-Binding, Direction Only)

These are not hard limits — they are guidance for task design:

| Task Type | Suggested Budget |
|---|---|
| Routing / classification | < 500 tokens total |
| Research brief | < 4,000 tokens total |
| Code scaffold | < 8,000 tokens total |
| Planning document | < 12,000 tokens total |
| Financial digest | < 6,000 tokens total |

## 8. Heartbeat / Cron Policy

Recurring tasks and health checks are the highest-risk category for invisible cost accumulation. These rules apply to all scheduled or triggered recurring model calls.

| Rule | Detail |
|---|---|
| Heartbeats are signal-only | Status pings, health checks, and presence signals MUST NOT invoke a language model. They are system-level calls, not model calls. |
| Recurring summaries use local | Any scheduled summary, digest, or report generation uses Local-Nano or Local-Mid only. Cloud is not permitted on a schedule. |
| Cron tasks must have a defined output | No cron task invokes a model without a defined output target (a file, a Telegram message, a Mission Control entry). |
| Cron frequency is approved by human | New recurring tasks that invoke a model must be approved by the human before the cron is registered. Frequency must be stated in the approval. |
| Cron tasks log every execution | Every cron-triggered model call logs: timestamp, model, task ID, output destination. No silent executions. |
| Escalation on consecutive failures | If a cron task fails 3 consecutive times, bridge (Conductor) marks it BLOCKED and posts an escalation to Telegram. It does not retry indefinitely. |

## 9. Bridge Model Policy

Bridge handles routing, scheduling, comms brokering, and low-complexity task orchestration. It is always-on and must operate even when cloud is unavailable.

| Field | Policy |
|---|---|
| Primary class | Local-Nano for routing and dispatch; Local-Mid for task orchestration and comms generation |
| Cloud permitted | Only for escalation response drafts (Cloud-Premium) or when explicitly task-assigned |
| Cloud forbidden | All cron tasks, all routing decisions, all heartbeats, all status checks |
| Model must be available at boot | Yes — once local inference is installed. **Bootstrap exception (CHG-002):** During Phase 1 bootstrap, before the local model is selected and installed, bridge may operate with Anthropic cloud inference (Cloud-Premium) as its primary inference lane. This state must be declared in the task register (TASK-005, TASK-008) and closes when the local model passes the D1.6 stability check. The bootstrap exception does not permit cloud use for cron, routing, or heartbeat tasks — those remain blocked until local is available. |
| Acceptable failure mode | If local model is unavailable after the bootstrap period closes, bridge marks itself DEGRADED in Mission Control and posts to Telegram. It does not silently fall back to cloud for routing tasks. During bootstrap, operating on cloud is declared state, not a failure mode. |
| Local model requirement | Minimum: at least one suitable local bridge model before Phase 1 exits (Source: BRG-INF-01). Target: one Local-Nano and one Local-Mid, so routing and task orchestration each run the appropriate tier. Hardware survey (TASK-002) must complete before model selection. |
| Specific model | TBD — see D1.6-LOCAL_INFERENCE_PLAN.md for Mac Mini hardware constraints and benchmark results |
| Routing policy approval | The routing decision tree in §5 is the approved bridge routing policy. This satisfies BRG-INF-02. |

## 12. Cloud Fallback Policy

Cloud inference is not a default — it is a defined exception. This section governs when cloud is permitted, how it is triggered, and what must happen when it is used.

### Conditions for Cloud Use

Cloud inference is permitted only when ALL of the following are true:
1. The task type is listed as cloud-permitted in §6
2. Local inference was attempted and failed, OR the task is explicitly designated as Cloud-Standard or Cloud-Premium class in its task packet
3. The task is not a cron task, heartbeat, or recurring background process
4. The agent's cloud API key is configured and the provider is approved (see §7)

Cloud inference is NOT permitted when:
- The task type is listed as cloud-forbidden in §6
- The task is recurring or cron-triggered
- No API key is configured for the agent
- The human has not yet approved this policy (D1.5 must be current before any cloud key is entered)

### Fallback Trigger Protocol

When an agent triggers a cloud fallback, it MUST:
1. Log the fallback: task ID, reason for fallback, local failure mode (if applicable), cloud provider, model selected
2. Post a Telegram notification if the cloud call is expected to cost above a defined threshold (threshold TBD in D1.14)
3. Complete the task and file the result normally — cloud fallback does not change task output requirements

### Provider Selection

Primary cloud provider is Anthropic. Within Anthropic:
- Use Claude-Haiku (Cloud-Standard) for overflow tasks that do not require deep reasoning
- Use Claude-Sonnet (Cloud-Premium) for planning, financial, and human-reviewed outputs
- Use Claude-Opus (Cloud-Premium) for escalation-triggered tasks where quality is critical

Do not mix providers within a single task execution.

## 13. Experimental Routing Policy

Experimental routing includes: multi-model ensemble approaches, proxy routing through a self-hosted OpenAI-compatible endpoint, model cascades, and any routing logic not described in §5.

### Current Status

Experimental routing is NOT active in Phase 1. It is a declared future option, not a current dependency.

### Conditions for Activation

Experimental routing MAY be activated in Phase 2 or later if:
1. The human approves the specific approach
2. A written experiment spec exists in D5.4-EXPERIMENT-REGISTRY.md
3. The experiment runs in a designated test context, not on the production task queue
4. Results are benchmarked against the standard routing policy before any production adoption

### What Is Not Experimental

The following are NOT classified as experimental and do not require this process:
- Adding a new local model to an approved model class on an existing machine
- Changing the specific model within an approved class (e.g. upgrading the Local-Mid model on Mac Mini)
- Adding Anthropic as a fallback if it is not yet configured but is already approved

## 14. Tool Lane Boundary — Perplexity Is Not Model Routing



Perplexity is a candidate external callable capability (D5.7 CAP-006) for live web research, current-information retrieval, and browser/computer-use tasks. It is **not** within the scope of this model routing policy.

- This document governs model selection and inference routing for bridge, brain, and forge using local and cloud LLM inference
- Perplexity is a CAP-SVC external service, not a language model in the inference stack; it does not receive routing decisions from the D1.5 §5 decision tree
- When Perplexity is activated, tasks that route to it do so because the task class maps to the Perplexity lane (D1.1 Principle 16, D1.2 §12), not because the D1.5 inference routing policy sent them there
- Cost and usage tracking for Perplexity, if activated, follows D1.12 capability governance and D1.8 automation policy — not the token budget guidance in §7 of this document

No change to this document is required when Perplexity is activated. Perplexity activation is governed by D1.12 (capability acquisition) and D1.15 (change control).

## 15. Tool Lane Preference and Routing Guidance

This section defines guidance for selecting which tool or lane handles a given class of work. It supplements the tool lane architecture in D1.1 Principle 16 and D1.2 §12 with preference rules and access path hierarchy.

### 15.1 OpenClaw as the Primary Autonomous Runtime

OpenClaw is the primary autonomous runtime and the default lane for approved backlog work, Mission Control buildout, and self-managed swarm execution. For tasks involving diagnosis, debugging, fix planning, code review, PR-style fixes, or complex issue resolution — especially issues involving OpenClaw configuration, bridge behavior, or agent context — Claude (via OpenClaw) is the preferred lane. Routing a debugging or fix task to Cursor first is appropriate for implementation work, but initial reasoning, root cause analysis, and fix design should be handled in the OpenClaw/Claude context where the agent's full task state is available.

**Preference order for reasoning, debug, and backlog tasks:**
1. OpenClaw (Claude) — primary autonomous runtime; has access to task context, mission control state, and operational history; default for approved backlog work
2. Cursor — supervised/manual lane for implementation of an approved fix once the diagnosis is clear, or for explicit engineering assignments
3. Perplexity (when activated) — for lookups requiring live external information, and as preferred UI-pilot for app/device/chatbox interaction

### 15.2 Cursor as the Supervised/Manual Engineering Lane

Cursor is the designated supervised/manual lane for engineering, code generation, QA, document authoring, and configuration tasks. **Cursor is not the swarm's autonomous runtime** — it is a supervised execution surface that requires human-in-the-loop or explicit task assignment. Agents do not improvise ad-hoc engineering execution in the OpenClaw context when Cursor is available and the task is within the engineering lane. This separation keeps the OpenClaw context clean and engineering outputs reviewable.

**Cursor is used when:**
- A human is intentionally doing assisted/manual engineering work
- OpenClaw is blocked and needs a manual engineering fallback
- A specific engineering task is explicitly assigned to the Cursor lane
- OpenClaw delegates a UI-mediated engineering task to the Perplexity/device lane, which then operates Cursor Claude chat through the interface (indirect usage — see §15.5)

**Cursor is NOT used as:**
- The default lane for approved backlog work (that is OpenClaw)
- The primary builder of Mission Control (OpenClaw leads buildout)
- An autonomous execution runtime that picks up and completes work without human involvement

### 15.3 Access Path Hierarchy

When selecting how to access a resource, capability, or external system, agents follow this preference order:

1. **Stable built-in access** — OpenClaw native capabilities, local file access, existing approved CAP-SVC calls. Zero additional maintenance surface.
2. **Approved API-based access** — Registered capabilities in D5.7 with active approval. Introduces an external dependency but is governed.
3. **Device-mediated access** — Computer-use, device-piloting, GUI automation. Used when paths 1 and 2 cannot fulfill the task, or when device-mediated access is actually the simpler and more reliable path (e.g., operating a chat UI instead of integrating a complex API). Requires explicit task packet approval (D1.3 BRG-TRL-04, CAP-007 candidate).

**API is not always the preferred path.** If an existing built-in or approved capability can fulfill the need without a new external dependency, that path is used. When direct API access adds unnecessary credential churn, expiry burden, or setup complexity, the system may prefer device-mediated access through Perplexity/UI operation. New API dependencies are introduced only when justified by demonstrated need.

### 15.4 Perplexity as Preferred UI-Pilot Lane (When Activated)

When Perplexity (CAP-006) is activated, it is preferred for:
- Tasks requiring live web retrieval or current information not in model weights
- Tasks requiring browser-style access to real-time data
- **UI-pilot tasks** where the best access path is operating an app, browser, device, or chatbox through the interface — Perplexity is the preferred operator lane for these interactions
- Device-mediated access tasks where Perplexity's computer-use capability is the designated lane
- Indirect Cursor usage — operating Cursor Claude chat through the UI (see §15.5)

Perplexity does not replace OpenClaw for orchestration. It is the preferred operator lane for UI-first interactions. OpenClaw remains the orchestrator and may delegate UI-mediated execution to Perplexity.

### 15.5 Indirect Cursor Usage via Device-Mediated Control

Cursor Claude chat may be used indirectly through device-mediated control — without requiring direct API integration. This means:
- Perplexity and/or OpenClaw may navigate the Cursor IDE UI, paste prompts, read responses, and advance supervised work through the interface
- This is a legitimate, documented tool lane interaction — not a workaround or undocumented hack
- Direct API integration with Cursor is not required for the swarm to leverage Cursor's engineering capabilities
- Cursor itself remains the supervised execution surface being operated — it is not the autonomous runtime

**Responsibility split for indirect Cursor usage:**
- Perplexity is the preferred UI-pilot/operator when the task involves navigating the Cursor chat interface, copy-pasting into chat tools, or operating through human-style interaction
- OpenClaw remains the orchestrator and may delegate UI-mediated execution to Perplexity/device-mediated access when that is the more reliable or lower-friction path
- Cursor provides supervised/manual execution capability through Claude chat, but is not itself the autonomous runtime

**Activation status:** Indirect Cursor usage depends on CAP-006 (Perplexity) and/or CAP-007 (device-mediated access) being activated. Neither is active at Phase 1. This section documents the intended access path for when those capabilities are adopted.

### 15.6 Visual/UI Acceptance Boundary

No dashboard, Mission Control screen, visual surface, or user-facing interface is accepted solely because an agent marks it complete. Agent completion status for visual outputs uses the following progression:
1. **Drafted** — initial version generated
2. **Built** — implementation complete, not yet reviewed
3. **Ready for visual review** — agent believes output meets requirements, awaiting human eyes
4. **Approved by human** — human has visually reviewed and accepted

Only step 4 constitutes acceptance. Agents must not treat steps 1–3 as final. This applies especially to Mission Control surfaces.

## 16. X / Grok Research Policy Hook


xAI/Grok is a future research intake lane. It is not activated for Phase 1. This section reserves the policy slot and states the conditions under which it may be activated.

### Current Status

Not activated. No Grok API key is configured on any agent. No agent routes tasks to Grok.

### Intended Role

When activated, Grok's role is research intake only:
- Sweeping X/Grok content for signal relevant to the organization's research lanes
- Providing real-time or near-real-time context for Researcher tasks on forge
- Augmenting research briefs with current content not available in static model weights

Grok is NOT used for:
- Task orchestration
- Planning or commerce tasks
- Memory writing
- Replacing the primary cloud fallback (Anthropic remains primary)

### Activation Gate

Before Grok is activated, the following must exist:
1. A written X/Grok intake policy in D1.10-X_RESEARCH_POLICY.md
2. A Grok API key managed under D1.14-SECRETS_POLICY.md
3. An explicit task type added to the §6 mapping table for research intake via Grok
4. Human approval of the activation
