# D1.19 — Task Sizing Policy

Every task MUST complete in ONE session. If it can't, it's too big — break it further.

Usopp sizes STORIES and sets guardrails. Dept leaders size TASKS under the story.

## Anti-patterns
- ✕ "Audit AND fix this file" → split into 2 tasks
- ✕ "Read 5 files and report" → one file per task
- ✕ "Build entire feature" → scaffold → build → verify
- ✕ Complex task for Haiku → will fail or hallucinate

## Good patterns
- → "Audit AGENTS.md, log findings in task notes" (read + document)
- → "Fix issues 1-3 from audit task" (focused execution)
- → "Build nav component to local" (one build action)
- → "Screenshot /dashboard, post PASS/FAIL" (one QA check)

## File Analysis Anti-pattern (learned 2026-03-24)
- ✕ "Read this 40KB file and analyze it" → TOO BIG for one session
- → Create a summary file (<5KB) first, THEN have the agent analyze the summary
- → Or split: Task 1 = read + summarize key points, Task 2 = analyze the summary
- This applies to dashboard HTML, large AGENTS.md files, codebase reviews, etc.
