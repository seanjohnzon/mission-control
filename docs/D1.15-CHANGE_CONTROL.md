# D1.15 — CHANGE_CONTROL
**Autonomous AI Organization — How Changes Are Proposed, Classified, Approved, Executed, and Recorded**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document defines how the organization manages change. Every meaningful modification to the system — to a policy, to an agent's configuration, to a workflow, to a capability, to a machine's baseline state, to a communication channel, to an automation registration, to a model configuration — is a change. Changes that happen without a record are drift. Changes that happen without approval are scope violations. Changes that happen without a rollback plan are bets against reliability.

This policy is the enforcement layer that makes the rest of the D1.x stack operational rather than aspirational. D1.8 defines what may be automated. D1.12 defines how capabilities are acquired. D1.13 defines what environments look like. D1.14 defines how credentials are governed. This document defines what process a change must pass through before any of those policies are modified or implemented.

The swarm is self-managed operationally. Agents are expected to pull approved change work, plan it, estimate it, execute it, verify it, and close it without human re-prompting at every step. That is what self-managed means. What it does not mean is that agents may decide what counts as a change, approve their own scope expansion, or alter governance boundaries without authorization. The change control process governs the boundary between execution and authorization. Execution is swarm-managed. Authorization is human-controlled.

**This document governs:**
- What constitutes a controlled change and how changes are categorized
- How changes are risk-classified
- What a change record must contain before execution begins
- Who approves which change categories
- How approved changes enter the backlog, are planned and sequenced, and are executed
- What counts as verified completion
- What happens when changes fail or must be rolled back
- How emergency changes are handled with minimum process compromise
- How change history is maintained and auditable
- How the swarm identifies repeatable patterns in change work and surfaces them as workflow candidates

**What this document does not cover:**
- The specific rules for what capabilities are allowed — D1.12
- Environment and baseline state definitions — D1.13
- Credential governance — D1.14
- Workflow definitions and format — D5.6
- Capability registration format — D5.7
- Automation execution policy — D1.8
- Communication channel policy — D1.11

## 2. Scope

This policy applies to all changes across the organization's systems and governing documents. A change is in scope if it meaningfully alters any of the following:

| Domain | Examples |
|---|---|
| Policies and governance documents | Modifying a D1.x–D5.x document |
| Agent configuration | OpenClaw workspace settings, tool access list, model routing |
| Workflows and reusable methods | Adding, updating, or retiring entries in D5.6 or D5.7 (CAP-WFM) |
| Capabilities | Additions, scope changes, retirements in D5.7 |
| Environment and machine baseline | Altering a declared machine baseline state (D1.13 §10) |
| Secrets and credential scope | Adding, widening, or revoking a secret's scope (D1.14) |
| Communication channel structure | Adding or removing a Telegram/Discord channel or changing a bot's scope |
| Automation registrations | Adding, modifying, or removing a cron job or trigger definition |
| Model and inference configuration | Changing model routing policy, switching a model on a machine, adding a model class |
| Infrastructure and services | Changes to running services, startup definitions, network config |
| Memory behavior | Changes to retention windows, promotion rules, indexing policy |
| Tool lane routing policy | Adding, changing, or retiring a tool lane assignment (e.g. activating Perplexity as the preferred UI-pilot and live research lane, changing which task classes route to the supervised Cursor engineering lane, enabling indirect Cursor usage via device-mediated control) — these are controlled changes because they alter what services agents may invoke and for what purpose. Note: Cursor is the supervised/manual engineering lane, not the autonomous runtime (see D1.5 §15.2). |

**What is not a controlled change:**
- Reading policies or registries
- Updating operational status fields in the task register (in-progress, completed, blocked) — these are part of normal execution, not governance changes
- Adding new task entries to the backlog — task creation is normal operations
- Writing a working memory note or experiment draft that has not yet been submitted for adoption
- A bridge capability register update within the explicitly bounded write scope (D1.12 §8.1)

## 3. Change Control Principles

1. **Document before change.** If a change affects any D-series document, the document is updated first and committed before the change is applied to the running environment. Implementation does not precede its governing record. (Source: D1.13 §8.4, D1.1 §3 principle 1)

2. **Every change has an owner, a scope, a target, an expected effect, a proof requirement, and a rollback path.** A change without all six declared is not ready to execute. Missing fields are not assumed — they are filled in or the change is blocked until they are.

3. **No silent changes.** A change that happens without a change record is drift. Agents that make configuration adjustments, install services, or alter any governed surface without a corresponding change record have violated this policy, regardless of how minor the change seemed.

4. **Production changes are boring and deliberate.** Production changes happen slowly, with proof, and with pre-declared rollback plans. Urgency is not a reason to skip the process. The only narrow exception is a confirmed emergency (§12) — and even then, the record is created, the scope is bounded, and the audit trail is preserved.

5. **Reusable methods are promoted, not rediscovered.** A change approach that has been executed successfully multiple times is an organizational asset. After the second or third execution of the same change pattern, it becomes a workflow candidate (D5.6) or a CAP-WFM candidate (D5.7). The swarm surfaces this; the human approves it. Repeating the same change ad hoc is waste.

6. **A successful repeated change pattern becomes a workflow candidate.** When a change type has been executed 3 or more times using a consistent approach, Conductor surfaces it as a workflow candidate per D5.6 §5.3. This is not optional — pattern detection is part of Conductor's operational responsibility.

7. **Self-managed task execution does not imply self-approved change.** The swarm pulls approved change work and executes it. It does not convert a proposed change into an active production change by executing it before the required approval. The approval gate is not optional because the swarm is well-intentioned.

8. **Changes are classified before they are executed.** The risk level and category are declared in the change record before execution begins. If an agent discovers mid-execution that the change is more risky than classified, it halts, updates the record, and escalates.

9. **One change record per meaningful change.** A single change record covers one coherent, bounded modification. Bundling unrelated changes into one record to reduce approval friction is a violation of this principle. Each change is evaluated on its own risk level.

10. **Verification is required before closure.** A change is not closed because the commands ran without error. It is closed when the expected effect has been confirmed, the affected registries and documents are updated, and the proof artifact exists.

## 4. Change Categories

Every change is assigned to one category at the time of proposal. The category determines the default risk class and who may propose and execute it.

### 4.1 Policy and Governance Changes

**What it is:** Modifications to any D1.x–D5.x document — adding, removing, or altering policy language, principles, tables, thresholds, or sections.

**Examples:** Updating a retention window in D1.7, adding a new escalation trigger to D1.8, patching a contradiction in D1.11.

**Default risk class:** Risk Class 3 (High). Policy changes affect all downstream behavior.

**Agent may propose:** Yes — agents may draft policy change proposals.

**Agent may execute:** No — policy documents are human-approved and human-committed (or committed by the human's direction after approval). Agents do not modify D-series documents on their own initiative outside of explicitly approved write surfaces.

### 4.2 Documentation Structure Changes

**What it is:** Adding, removing, or reorganizing files in the Mission Control document tree without changing policy substance.

**Examples:** Adding a new section skeleton, creating a new register file, restructuring a registry file's layout.

**Default risk class:** Risk Class 1 (Low) for structural-only. Risk Class 3 if the structure change alters how agents read and interpret policy.

**Agent may propose:** Yes.

**Agent may execute:** Low-risk structural additions only, within the agent change authority defined in D1.13 §11.1; bounded operational register updates per D1.12 §8.1 where applicable.

### 4.3 Workflow and Reusable Method Changes

**What it is:** Adding, updating, or retiring entries in D5.6-WORKFLOW_LIBRARY.md or a CAP-WFM entry in D5.7-CAPABILITY-REGISTER.md.

**Examples:** Drafting a new workflow definition, versioning an existing workflow, retiring a method that is no longer used.

**Default risk class:** Risk Class 2 (Medium). Workflow changes affect consistency of execution across many tasks.

**Agent may propose:** Yes — Conductor surfaces workflow candidates; Forge drafts methods.

**Agent may execute:** Agents may create draft entries (status: `draft`). Promotion to `active` requires human approval (D5.6 §5.4).

### 4.4 Capability Additions, Updates, and Retirements

**What it is:** Changes to the capability register (D5.7) — new capability adoption, scope changes, version updates, deprecation, retirement.

**Default risk class:** Per D1.12 §12. Defaults range from Risk Class 1 (CAP-WFM) to Risk Class 4 (financial or credential-scope capabilities).

**Agent may propose:** Yes — agents surface candidates and draft proposals per D1.12 §6.

**Agent may execute:** Operational field updates within bounded scope (D1.12 §8.1). Approval-gated fields require human decision.

### 4.5 Agent Configuration Changes

**What it is:** Changes to an agent's OpenClaw workspace configuration — tool access list, model routing preference, task class defaults, agent scope definition.

**Examples:** Adding a new tool to forge's access list, changing bridge's default model routing rule.

**Default risk class:** Risk Class 2 (Medium). Agent configuration changes affect all tasks that agent runs.

**Agent may propose:** Yes.

**Agent may execute:** Within Layer 3 config bounds (D1.13 §7), with change logged. Changes that expand scope or affect production behavior require human approval.

### 4.6 Machine-Local Configuration Changes

**What it is:** Changes to a machine's OS-level settings, startup services, network configuration, sleep/power policy, or Layer 2 config values (D1.13 §7).

**Examples:** Changing Mac Mini's sleep setting, adding a new launchd service, changing a DHCP reservation.

**Default risk class:** Risk Class 2 (Medium) for non-production machines. Risk Class 3 (High) for Mac Mini (production gateway).

**Agent may propose:** Yes.

**Agent may execute:** Non-critical, logged changes within pre-approved scope. Mac Mini changes require human approval before execution.

### 4.7 Environment and Baseline Changes

**What it is:** Changes to the declared machine baseline state in D1.13 §10, or changes to environment type assignments for machines.

**Default risk class:** Risk Class 3 (High) for baseline changes. Risk Class 4 for environment type reassignment (e.g., moving GPU3060 from Experimental to Production scope).

**Agent may propose:** Yes.

**Agent may execute:** No. Environment and baseline changes require human approval and a D1.13 document update before execution.

### 4.8 Secrets and Credential Scope Changes

**What it is:** Any change to a secret's authorization scope — adding an authorized agent, adding an authorized machine, widening the authorized environment, changing the authorized capability.

**Default risk class:** Risk Class 4 (Critical) for any scope widening. Risk Class 2 for narrowing (revoking authorization).

**Agent may propose:** Yes — agents identify credential gaps and surface them.

**Agent may execute:** No. All scope changes require human approval per D1.14 §6.

### 4.9 Communication Channel Changes

**What it is:** Adding or removing a Telegram or Discord channel, changing a bot's permissions or scope, changing the channel hierarchy in D1.11, enabling or disabling an emergency path.

**Default risk class:** Risk Class 3 (High). Communication channel changes affect the organization's primary coordination layer.

**Agent may propose:** Yes.

**Agent may execute:** No. Channel changes require human approval and D1.11 update before execution.

### 4.10 Automation Registration Changes

**What it is:** Adding, modifying, or removing a cron job definition, automation trigger, or scheduled task in the Mission Control automation register (D1.8 §9.1).

**Examples:** Adding a new daily digest cron, changing a sweep frequency, disabling a scheduled task.

**Default risk class:** Risk Class 2 (Medium).

**Agent may propose:** Yes — Conductor may propose schedule changes.

**Agent may execute:** Adding new cron definitions requires human approval of the definition before activation. Modifying existing definitions also requires approval. Bridge may execute the schedule (run the cron), not define it.

### 4.11 Model and Inference Configuration Changes

**What it is:** Changes to model routing policy, model selection per machine, model class additions (D1.5), or inference resource allocation.

**Default risk class:** Risk Class 2–3. Model changes affect all agent outputs. Changes that enable cloud inference or activate a new API class are Risk Class 3.

**Agent may propose:** Yes — Forge may propose model changes based on benchmark results.

**Agent may execute:** Local model file installation on GPU3060 is within forge's scope. Routing policy changes require human approval and D1.5 update first.

### 4.12 Infrastructure and Service Changes

**What it is:** Changes to running services beyond what is declared in machine baselines, changes to OS-level infrastructure, changes to networking rules or firewall config.

**Default risk class:** Risk Class 3 (High). Infrastructure changes can affect multiple agents, machines, and capabilities simultaneously.

**Agent may propose:** Yes.

**Agent may execute:** No. Infrastructure changes require human approval, D1.2 and D1.13 updates as needed, and an explicit rollback plan.

## 5. Change Risk Levels

Risk is assessed at the time of proposal. It reflects blast radius (what breaks if this goes wrong), reversibility (can the change be undone cleanly), external effect (does this touch external systems, credentials, or finances), and dependency impact (how many workflows or capabilities depend on what is being changed).

| Risk Level | Name | Gate | Characteristics |
|---|---|---|---|
| 1 | Low | Human acknowledgement | Scoped to one internal surface, no external effect, easily reversible, no credential or financial dimension, no production impact |
| 2 | Medium | Human explicit approval | Affects one agent or machine, reversible with effort, no credential or financial scope, limited production impact |
| 3 | High | Explicit approval + execution window | Affects multiple agents/machines or a production surface, not easily reversible, or touches external services without financial write access |
| 4 | Critical | Explicit approval + written MC record + pre-execution review | Touches credentials, financial systems, production gateway, external write-access surfaces, or policy boundaries |

**Automatic Risk Class 4 designations (no exceptions):**
- Any change to secret scope (D1.14)
- Any change involving financial accounts, payment credentials, or transaction capabilities
- Any change to Mac Mini's production gateway services or startup configuration
- Any change to a D1.x policy document that affects an autonomy level, escalation threshold, or human-authority boundary
- Any change that activates a capability in an external environment with write access (platforms, commerce channels)

**Risk class may be elevated during review.** If a reviewer identifies factors not captured in the initial classification, the risk class is updated before the change proceeds. Risk class may not be lowered during execution without a new review.

## 6. Change Request and Proposal Requirements

Every change begins with a change record. The record must exist before execution begins. The following fields are required for all changes; additional fields may be added by category.

### 6.1 Required Change Record Fields

```
---
change_id: CHG-### sequential
date_proposed: YYYY-MM-DD
proposed_by: human | bridge | brain | forge
category: <from §4>
risk_level: 1 | 2 | 3 | 4
target: <machine / agent / document / registry / service being changed>
environment: production | sandbox | dev | experimental
why_needed: <1–3 sentences — the demonstrated need, not speculation>
current_state: <what exists now — specific and accurate>
proposed_state: <what will exist after — specific and accurate>
affected_workflows: <D5.6 workflow IDs affected, or "none">
affected_capabilities: <D5.7 CAP-IDs affected, or "none">
affected_secrets: <D1.14 SEC-IDs affected, or "none">
expected_effect: <what the change is expected to produce — testable>
verification_method: <how the expected effect will be confirmed>
rollback_plan: <what will be done if the change must be reversed — specific>
approval_status: proposed | approved | in-progress | completed | rolled-back | rejected
approved_by: human | — (if not yet approved)
approval_date: YYYY-MM-DD | —
implementation_window: <date/time range if relevant, or "on approval">
notes: <any constraints, dependencies, or conditions>
---
```

### 6.2 Completeness Requirement

A change record with unfilled required fields is not ready for approval. Agents do not submit change proposals with placeholder values. If information is genuinely unknown, the change is blocked until the information is obtained — the unknown is not approximated.

### 6.3 Change Register

All change records are filed in Mission Control at `registry/changes/CHANGE-REGISTER.md` (exact path confirmed at Phase 1 setup). Each change has a record in the register. The register is the audit trail. It is version-controlled. It is not deleted — completed and rolled-back changes are retained.

## 7. Approval Model and Authority

### 7.1 Approval Requirements by Risk Level

| Risk Level | Approval Required | Record Location |
|---|---|---|
| 1 (Low) | Human acknowledgement (Telegram sufficient) | MC change record updated with `approved_by` and `approval_date` |
| 2 (Medium) | Human explicit approval | Telegram approval notification; MC change record is the authoritative approval record |
| 3 (High) | Human explicit approval + execution window confirmation | MC change record required before execution begins |
| 4 (Critical) | Human explicit approval + written MC record + pre-execution review | MC record must be committed and confirmed before any execution step |

An approval communicated in Telegram is valid — but the record must be reflected in Mission Control before the change is executed. A Telegram message alone, without a corresponding MC change record update, is not an approval. (Source: D1.11 §3 principle 9)

### 7.2 What Human Approval Covers

Human approval authorizes the specific change as described in the change record at the time of approval. It does not authorize:
- Scope expansion discovered mid-execution
- Related changes not included in the original record
- The same change in a different environment or on a different machine

If mid-execution scope expansion is discovered, the agent halts, updates the change record, and escalates for re-approval before continuing. (Source: D1.8 §8 escalation trigger 1)

### 7.3 Agent Execution Authority (Self-Managed Scope)

Agents may execute changes without per-step human confirmation when:
- The change has been approved (all approval gate requirements satisfied)
- The change is within the agent's declared execution authority for its category
- The change follows an approved workflow from D5.6 (when one exists)
- No scope expansion has been encountered

Agents may not:
- Approve their own proposed changes
- Widen a change's scope mid-execution without re-approval
- Convert a `proposed` change record to `in-progress` without the required approval being recorded
- Execute a Risk Class 3 or 4 change outside its declared implementation window

### 7.4 Bridge Operational Registry Updates

Bridge may update operational fields in Mission Control registries within its bounded write scope (D1.12 §8.1, D1.13 §11.1) as part of normal execution, without a change record. These updates are part of execution, not governance changes. The boundary is: bridge updates fields it is authorized to update during normal operations. Bridge does not reclassify, reapprove, or redefine any registered item.

## 8. Planning, Scheduling, and Backlog Rules

### 8.1 Approved Changes Enter the Backlog

When the human approves a change, it enters the Mission Control task backlog as an approved change task. It does not execute immediately unless the implementation window is "on approval." Approved changes sit in the backlog until Conductor sequences them.

### 8.2 Conductor Manages Sequencing

Conductor (bridge) sequences approved changes within the swarm's operational backlog. The sequencing follows:
- Human-declared priorities (explicitly prioritized items first)
- Risk level (higher-risk changes when the operator is available for monitoring)
- Dependencies (changes that other changes depend on run first)
- Implementation windows (time-constrained changes respect their declared window)

The human does not need to manually push each approved change task. Conductor identifies what is eligible, in what order, and executes within approved scope. This is the operational self-management model in practice.

### 8.3 Sprint-Like Execution Cycles

Agents run sprint-like cycles within approved change backlog: pull eligible tasks, plan them, estimate them, execute them, verify them, and close them. The human sets priorities and approvals. The swarm manages the motion. Estimation and sequencing are the swarm's operational responsibility, not the human's.

If there is ambiguity about sequence or priority, Conductor picks the highest-priority unambiguous item and surfaces the ambiguous ones in the next digest.

### 8.4 Mac Mini Production Path Change Scheduling

Changes to Mac Mini's production path (gateway services, startup config, machine-local config) have stricter scheduling protection:
- They are scheduled during a declared low-traffic window (to be defined based on Phase 1 operating data)
- The human is notified before execution begins, not only at completion
- Bridge monitors the gateway's availability during and after the change
- If the gateway becomes unreachable after the change, rollback is initiated immediately without waiting for human confirmation (the rollback plan declared in the change record governs)

### 8.5 Repeated Change Patterns Are Workflowized

When Conductor observes the same change type executed successfully 3 or more times with a consistent approach:
1. It surfaces a workflow candidate using the D5.6 §5.3 format in the next weekly digest
2. It drafts a workflow entry for human review
3. Future executions of that change type use the approved workflow — not ad-hoc reconstruction of the approach

The goal is that repeatable build and change tasks become consistent and automated over time. Agents do not re-improvise approaches they have already proven out. (Source: D1.13 §12)

## 9. Execution Rules

### 9.1 Execution Prerequisites

Before beginning execution of an approved change:
- The change record is `approved` (not `proposed`)
- All required fields are populated
- The rollback plan is declared
- Any affected workflows or capabilities have been reviewed for dependency impact
- If a D5.6 workflow exists for this change category — it is used (D5.6 §4.1)

### 9.2 Use Approved Workflows When They Exist

If an approved workflow in D5.6 covers the change type, that workflow is used. A change agent does not improvise a different approach for a known, workflowed change class. Deviations from an approved workflow must be flagged in the change record and the next digest. (Source: D1.13 §12.3)

If no workflow exists, the change record notes `approach: ad-hoc` and Conductor tracks the pattern for workflow candidacy (§8.5).

### 9.3 Environment Promotion Path Is Respected

Changes that affect configuration or capabilities follow the environment promotion path (D1.13 §9): Experimental → Sandbox → Production. The change record must declare which environment the change is being applied to. Changes do not jump from Experimental to Production without the required intermediate validation, regardless of urgency.

### 9.4 No Mid-Change Scope Expansion

If execution reveals that the change requires more than was declared in the change record:
1. Execution halts at the boundary of the declared scope
2. The change record is updated with the newly discovered scope
3. Bridge escalates: the expanded change requires re-approval before continuing
4. The partial state is documented — the system does not proceed on the assumption that the expansion is acceptable

Partial completion is recorded honestly. The change stays `in-progress` until it is either completed as re-approved or rolled back to a clean prior state.

### 9.5 Policies and Registries Are Updated Concurrently

Changes that affect a D-series document, a capability record, or a secret registry entry update those records as part of the change execution — not as a follow-up task. "We'll update the docs later" is not acceptable. The record and the environment change together.

## 10. Verification and Acceptance

### 10.1 What "Complete" Means

A change is complete only when all of the following are true:
1. The expected effect declared in the change record has been confirmed
2. The confirmation method declared in the change record was used (not a different, more convenient method)
3. All affected policies, registries, and documents are updated and committed
4. A proof artifact exists (a log, a test result, a screenshot, a confirmation message — something concrete that confirms the expected effect)
5. The change record is updated to `completed` with the proof artifact referenced

"The command ran without error" satisfies none of these criteria on its own.

### 10.2 Human Review Requirement by Risk Level

| Risk Level | Human Review at Completion |
|---|---|
| 1 | Not required — proof artifact in change record is sufficient |
| 2 | Bridge notifies human via Telegram completion notice; human may review |
| 3 | Human reviews the proof artifact before change record is closed |
| 4 | Human explicitly confirms completion in Mission Control or Telegram before the change record is closed |

### 10.3 Reusable Pattern Detection at Completion

When a change is verified complete, Conductor checks: has this change type been executed successfully before? If yes, how many times? If 3 or more, surface a workflow candidate per D5.6 §5.3. Completion is the trigger for this check — not a separate scheduled review.

## 11. Rollback and Recovery

### 11.1 Rollback Plan Is Mandatory Before Execution

Every change at Risk Level 2 and above requires a declared rollback plan in the change record before execution begins. The rollback plan must be specific: what commands or actions restore the prior state, how long restoration is expected to take, and whether any data loss is involved in rollback. "We'll figure it out if it breaks" is not a rollback plan.

Risk Level 1 changes are encouraged but not required to declare a rollback plan.

### 11.2 Rollback Restores Declared Baseline, Not Improvised State

When a rollback is executed, the goal is to restore the declared baseline (D1.13 §10) or the prior registered state (for capability or config changes) — not to reconstruct what the system was doing at the moment of failure. If the declared baseline is outdated, the baseline is updated through the change process first. (Source: D1.13 §14.4)

### 11.3 Partial-Change Failure

If a change partially applies and then fails:
1. Do not continue applying the remaining steps
2. Assess whether the partial state is stable or unstable
3. If unstable: execute rollback immediately; do not wait for human confirmation (the pre-declared rollback governs)
4. If stable: halt, document the partial state honestly in the change record, escalate to human before deciding whether to continue or roll back

The system does not maintain a broken partial state because completing the change "would fix it." An incomplete change that has already caused failures is rolled back unless the human explicitly approves continuing under the partial-state conditions.

### 11.4 Mac Mini Production Path Rollback

Changes to Mac Mini production services have an additional rule: if the OpenClaw gateway becomes unreachable after a change, rollback is initiated immediately — without waiting for human direction. The rollback plan in the change record is executed autonomously. Bridge then escalates with the rollback result. Speed of recovery takes precedence over waiting for permission when the production path is down.

### 11.5 Rollback Is Not Failure

A completed rollback is not a failure record. It is evidence that the change control process worked: the change was bounded, the rollback was pre-declared, the system restored to a known state, and the condition is auditable. The change record is marked `rolled-back` with the cause documented. A future attempt at the same change starts from this record.

## 12. Emergency Changes

### 12.1 What Qualifies as an Emergency

A change qualifies for the emergency path when:
- A production system is actively degraded or unreachable and the change is required to restore it
- A confirmed security incident (credential compromise, active unauthorized access) requires immediate action
- A change cannot wait for normal approval cadence without causing irreversible harm

Urgency alone does not qualify. A change that would be convenient to do quickly but can wait for normal process is not an emergency.

### 12.2 Emergency Change Authorization

An emergency change may be authorized verbally or via Telegram by the human. The authorization must include:
- Explicit statement that this is an emergency authorization
- What change is being authorized
- The time window in which the authorization is valid

Bridge (Conductor) confirms receipt before proceeding. If the human is unreachable, bridge may apply a self-healing action only within the pre-declared rollback scope of an existing active change, or may restart a declared required service that has stopped. Bridge may not apply novel production changes when the human is unreachable.

### 12.3 What Cannot Be Skipped Even in an Emergency

The following must happen even in an emergency:
- Scope remains bounded to what is required to resolve the immediate issue — no opportunistic additional changes
- Every action taken is logged in real-time or as immediately after as possible
- A change record is created — it may be created after execution if during-execution documentation is impractical, but it must exist within 24 hours
- Secrets and credential governance (D1.14) applies — no credential shortcut is taken even in emergency
- Rollback capability must still be preserved — an emergency change that makes rollback impossible requires explicit human authorization of that irreversibility

### 12.4 After-Action Record

Within 24 hours of an emergency change, the change record must be complete: all fields populated, proof artifact referenced, cause documented, and whether the emergency classification was justified noted. Emergency does not erase audit. It only defers documentation briefly.

## 13. Change Logging, Audit, and Traceability

### 13.1 The Change Register Is the Record

Mission Control's change register (`registry/changes/CHANGE-REGISTER.md`) is the authoritative record of all changes. Telegram messages, verbal discussions, and Discord posts are communication — the change register is the record. Every change that was proposed, approved, executed, and verified must be traceable from the register.

### 13.2 What Must Be Traceable

For any change in the register, the following must be answerable without accessing the running machines:
- Who proposed the change and when
- What the change was (current state, proposed state, target)
- Who approved it and when
- What the declared rollback was
- When execution began and ended
- What proof of completion exists
- Whether the change was completed, rolled back, or rejected
- Which capabilities, workflows, or secrets were affected

### 13.3 Mission Control Is the Record; Telegram Is the Coordination Surface

Approvals communicated in Telegram must be reflected in Mission Control before the change is executed. Telegram is where the human says "approved." Mission Control is where the approval is recorded. These are two steps, not one. The change record is not complete until both have happened. (Source: D1.11 §3 principle 9, D1.13 §8.4)

### 13.4 No Change Exists Only in Chat

A change that was discussed in Telegram, executed, and never entered into the change register is invisible to audit, invisible to dependency tracking, and invisible to future agents attempting to understand the system's current state. It is also a policy violation. The swarm builds a traceable record of what changed and why — not just what is currently running.

## 14. Self-Managed Swarm Role in Change Control

The swarm is expected to be the primary executor of approved change work. This section defines explicitly what that means and where it stops.

### 14.1 What the Swarm Is Expected to Do

Within the bounds of approved change work, agents are expected to operate like a well-run internal team managing their own board:

- **Pull approved change tasks** from the backlog without waiting for the human to push each one. If a change has been approved and is eligible to execute, Conductor picks it up.
- **Plan and estimate** changes within the sprint cycle. Conductor sequences work, considers dependencies, and respects implementation windows — without human scheduling each item.
- **Execute using approved workflows** when D5.6 workflows exist for the change type. Consistency of execution is the goal. When no workflow exists, the ad-hoc approach is documented and tracked as a workflow candidate.
- **Hand off** between agents cleanly. A change that involves bridge configuring a service and forge validating the result follows a declared handoff — not an informal message.
- **Verify and close** change records when the expected effect is confirmed. Agents do not leave change records in `in-progress` when they are done.
- **Surface repeatable patterns.** When the same change type succeeds repeatedly, Conductor drafts a workflow candidate and surfaces it in the weekly digest. This is an operational responsibility.
- **Keep the board current.** The change register reflects actual state. Stale, incomplete, or inaccurate change records are a hygiene failure. Backlog hygiene for the change register is part of the daily hygiene cycle.

### 14.2 What the Swarm May Not Do

The self-management model ends at the boundary of authorization:

- **No agent may approve its own proposed changes.** A change record stays `proposed` until the human approves it. An agent writing `approved_by: bridge` in its own change record is a violation.
- **No agent may widen a change's scope without re-approval.** Mid-execution scope expansion requires halt and escalation, not improvised execution.
- **No agent may change its own policy constraints.** Escalation thresholds, autonomy levels, credential scope, and D1.x policy content are human-controlled. An agent that edits these to make its own task easier has violated the most fundamental boundary in this system.
- **No agent may classify an uncategorized change as low-risk to avoid the approval gate.** If the risk level is unclear, it defaults to Risk Class 3 until the human classifies it.
- **No agent may skip the change record because the change seems minor.** "Minor" is a judgment call that belongs to the classification process, not to the agent executing the change.

### 14.3 The Model in Practice

The swarm should require the human to make decisions — approvals, priority setting, risk classification — but should not require the human to manage the operational motion of executing approved work. When the human approves a batch of changes, the swarm should be able to plan, sequence, execute, verify, and close all of them in a sprint cycle without check-ins at every step. That is what self-managed means here. The human sets the boundaries; the swarm executes within them.
