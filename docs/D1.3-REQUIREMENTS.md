# D1.3 — REQUIREMENTS
**Autonomous AI Organization — Verifiable Conditions by Layer and Phase**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md | Architecture: D1.2-ARCHITECTURE.md_

---

## 1. Purpose

This document defines the verifiable conditions that must be true before the autonomous AI organization can be considered operational at each phase. It is a requirements register — not a runbook, not a policy document, not a design spec.

**What this document does:**
- States what must be true (not how to make it true)
- Assigns requirements to phases and machine layers
- Distinguishes MUST (required) from SHOULD (desired but not blocking)
- Marks items as TBD or BLOCKED when implementation facts are not yet known
- Provides the acceptance criteria referenced in D1.1 Phase exit conditions

**What this document is not:**
- Not a runbook (see D2.x — Runbooks)
- Not an architecture description (see D1.2)
- Not a policy document (see D1.5–D1.15)
- Not an org role definition (see D1.4)

When a requirement changes — scope shifts, a constraint is resolved, a blocker clears — this document is updated first. Phase exit gates in D1.1 must be reviewed when this file changes.

## 2. Scope

Requirements in this document cover:

| Scope Layer | What It Covers |
|---|---|
| System-level | Cross-cutting properties that apply regardless of which machine or agent is involved |
| Bridge (Mac Mini) | Requirements specific to the always-on gateway node |
| Brain (Desktop) | Requirements specific to the memory and planning node |
| Forge (GPU3060 Laptop) | Requirements specific to the build, QA, and inference node |
| Edge (Pixel / Seeker) | Requirements specific to edge execution surfaces |
| Mission Control | Documentation and state management requirements |
| Communications | Telegram (primary) and inter-agent messaging requirements |
| Memory | State persistence and retrieval requirements |
| Local Inference | On-device model serving requirements |
| Legacy Assets | Carry-forward and migration requirements |

Requirements are tagged by phase (P0–P4) to indicate when they become binding. A requirement tagged P0 blocks Phase 0 exit. A requirement tagged P2 is not binding until Phase 2 begins.

## 3. Requirement Language

| Keyword | Meaning |
|---|---|
| **MUST** | Non-negotiable. The system fails this phase if this condition is not true. |
| **MUST NOT** | Absolute prohibition. Violation constitutes failure. |
| **SHOULD** | Strongly desired. Deviation requires documented justification. |
| **SHOULD NOT** | Strongly discouraged. Acceptable only with documented reason. |
| **MAY** | Optional. No justification required for omission. |
| **TBD** | Requirement is directionally understood but exact condition cannot be written until more facts are known. |
| **BLOCKED** | Requirement cannot be verified until a specific blocker is resolved. Blocker is named inline. |

All MUST requirements for a phase must be verifiable — meaning a yes/no test exists for each one. Requirements that cannot be tested are either rewritten or marked TBD until they can be.

## 4. System-Level Requirements

These apply to the organization as a whole, not to any single machine or agent.

### 4.1 Durability

| ID | Phase | Level | Requirement |
|---|---|---|---|
| SYS-DUR-01 | P0 | MUST | The organization MUST continue operating autonomously when the MacBook Air (Control Hub) is offline, asleep, or unreachable. |
| SYS-DUR-02 | P0 | MUST | Any node that fails unexpectedly MUST restart its core services automatically without human intervention. |
| SYS-DUR-03 | P1 | MUST | State required for ongoing tasks MUST NOT be stored exclusively in volatile memory. It must survive a process crash. |
| SYS-DUR-04 | P1 | SHOULD | The system SHOULD continue degraded operation when any single non-bridge node is offline. |
| SYS-DUR-05 | P2 | MUST | Recovery from a full Bridge reboot MUST complete within 5 minutes of machine power-on with no human action. |

### 4.2 Legibility

| ID | Phase | Level | Requirement |
|---|---|---|---|
| SYS-LEG-01 | P0 | MUST | Every agent, role, and active task MUST have a canonical record in Mission Control at all times. |
| SYS-LEG-02 | P1 | MUST | Every directive issued by a human MUST be traceable to an outcome — completed, rejected, delegated, or stalled — with a reason. |
| SYS-LEG-03 | P1 | SHOULD | Any agent action that modifies state SHOULD produce a timestamped log entry readable by the human operator. |
| SYS-LEG-04 | P2 | MUST | The system MUST support audit of any decision made in the past 30 days without requiring agent cooperation. |

### 4.3 Security Boundary

| ID | Phase | Level | Requirement |
|---|---|---|---|
| SYS-SEC-01 | P0 | MUST | No agent MUST have write access to Mission Control unless that access is explicitly granted by the human operator. |
| SYS-SEC-02 | P0 | MUST | No agent MUST be able to initiate financial transactions without an explicit, per-transaction human approval step until Autonomy Level 4 is granted for that task type. |
| SYS-SEC-03 | P1 | MUST | Agent credentials (API keys, tokens) MUST NOT be stored in plain text in any document committed to Mission Control. |
| SYS-SEC-04 | P1 | SHOULD | External-facing services SHOULD require authentication. Open unauthenticated endpoints are acceptable only for local-loopback services explicitly documented as internal. |

### 4.4 Autonomy Progression

| ID | Phase | Level | Requirement |
|---|---|---|---|
| SYS-AUT-01 | P0 | MUST | All agents MUST start at Autonomy Level 0 (executes only when human explicitly calls it). |
| SYS-AUT-02 | P1 | MUST | Autonomy level MUST NOT be increased for any task type until the agent has demonstrated correctness on that task at the current level for at least 3 observed cycles. |
| SYS-AUT-03 | P2 | SHOULD | The system SHOULD track autonomy level per task type, not per agent. A single agent may hold different autonomy levels for different tasks. |

### 5.6 Self-Managed Execution

The swarm is self-managed within approved boundaries. Bridge (Conductor) is responsible for identifying eligible work, pulling tasks, and maintaining queue health without human prompting per task. Agents do not self-expand authority.

| ID | Phase | Level | Requirement |
|---|---|---|---|
| BRG-SWM-01 | P1 | MUST | Bridge (Conductor) MUST be capable of identifying and pulling eligible tasks from the Mission Control backlog without a human prompt per task pickup. |
| BRG-SWM-02 | P1 | MUST | Bridge MUST perform backlog hygiene: tasks that are stale, missing required fields, or blocked without a recorded reason MUST be flagged — not silently left. |
| BRG-SWM-03 | P2 | MUST | When a task class has an approved workflow in the Workflow Library (D5.6), bridge MUST execute using the approved workflow rather than improvising a new approach for a known pattern. |
| BRG-SWM-04 | P2 | SHOULD | Bridge SHOULD identify recurring ad-hoc task patterns and surface them as workflow candidates for human approval and promotion to D5.6. |
| SWM-AUTH-01 | P1 | MUST | Self-managed execution MUST NOT expand agent authority beyond the approved automation class, autonomy level, or escalation policy defined in D1.8. Agents are self-managed, but not self-authorizing beyond policy. |

### 5.7 Tool Lane Self-Management

Agents select the correct tool lane for each task class automatically within approved policy. Lane assignments are defined in D1.1 Principle 16 and the D1.2 §12 tool lane architecture.

| ID | Phase | Level | Requirement |
|---|---|---|---|
| BRG-TRL-01 | P1 | MUST | Engineering tasks (code generation, QA, build, document authoring) MUST route to Cursor as the designated supervised/manual engineering lane. Cursor requires human-in-the-loop or explicit task assignment — it is not the autonomous runtime. Bridge does not improvise ad-hoc engineering execution outside this lane. Cursor may also be operated indirectly through device-mediated control (Perplexity or OpenClaw navigating the Cursor UI) without requiring direct API integration. |
| BRG-TRL-02 | P1 | MUST | Orchestration, scheduling, inter-agent dispatch, gateway management, and approved backlog execution MUST execute within OpenClaw as the primary autonomous runtime. OpenClaw is the default lane for self-managed swarm work and Mission Control buildout. These functions are not delegated to Cursor or any external tool lane. |
| BRG-TRL-03 | P1 | SHOULD | When Perplexity (CAP-006) is activated, tasks requiring live web retrieval, current information, or browser/computer-use SHOULD route to Perplexity rather than falling back to local inference for real-time information. Perplexity is also the preferred UI-pilot lane for app, device, and chatbox interaction — including indirect operation of Cursor Claude chat through the UI. Perplexity is not activated at Phase 1 — this requirement becomes binding upon CAP-006 adoption. (Source: D1.3 INF-CON-01: local inference must not be used for tasks requiring real-time information without retrieval augmentation) |
| BRG-TRL-04 | P2 | SHOULD | Device-mediated access (computer-use, device-piloting, GUI automation as a fallback) SHOULD only be used when no stable API-based or built-in-capability path exists for the required task. Device-mediated fallback is a governed candidate capability class (CAP-007 candidate), not an ad-hoc improvisation. Before any device-mediated action is executed, it MUST have an approved task packet declaring the target surface, scope, expected output, and why stable paths are unavailable. Indirect Cursor usage through device-mediated control is a specific, documented use case of this lane. (Source: D1.1 Principle 17) |
| BRG-TRL-05 | P1 | MUST | Visual surfaces (dashboards, Mission Control screens, user-facing interfaces) MUST NOT be accepted solely because an agent marks them complete. Agent completion status for visual outputs may be "drafted," "built," or "ready for visual review." Final acceptance requires human visual approval. (Source: D1.1 §4 Tool Lanes) |

### 5.8 Access Minimization

The organization minimizes the number of external access surfaces agents must maintain. Stable, built-in access paths are preferred over device-mediated fallbacks, which are preferred over new external API dependencies.

| ID | Phase | Level | Requirement |
|---|---|---|---|
| BRG-AMP-01 | P1 | SHOULD | Before establishing a new external API dependency or device-mediated access path for a recurring task, bridge (Conductor) SHOULD verify that an existing approved capability (OpenClaw built-in, active CAP-SVC, or Cursor lane) cannot fulfill the same need. New access surfaces are a last resort, not a first option. (Source: D1.1 Principle 17) |

| COM-DSC-02 | P1 | MUST | If a Discord bot token is obtained, it MUST NOT be stored in plain text in Mission Control. It must be stored in an environment variable or secrets file excluded from version control. |
| COM-DSC-03 | P1 | MAY | If Discord is activated, Bridge MAY receive directives from Discord as a secondary channel. Telegram acknowledgement and routing remain primary. |
| COM-DSC-04 | P2 | SHOULD | If Discord is activated, agent status updates SHOULD be surfaced to a designated Discord channel without requiring the human to poll. |
| COM-DSC-05 | P2 | SHOULD | If Discord is activated, distinct agent roles SHOULD post to distinct Discord channels or threads so human can track activity by role without filtering. |

### 10.3 Inter-Agent Communication

| COM-IAC-02 | P2 | MUST | Inter-agent messages MUST be logged with enough detail to reconstruct the exchange without agent cooperation. |

## 11. Memory Requirements

Memory is the persistence layer for all organizational state beyond the current conversation.

### 11.1 Operational Memory (Mac Mini)

| ID | Phase | Level | Requirement |
|---|---|---|---|
| MEM-OPS-01 | P1 | MUST | OpenClaw state (`~/.openclaw/`) on Mac Mini MUST persist across reboots. It MUST NOT be on a tmpfs or volatile path. |
| MEM-OPS-02 | P1 | MUST | Active task state MUST be recoverable after an unplanned Mac Mini reboot without loss of task identity or prior steps. |
| MEM-OPS-03 | P2 | SHOULD | Operational memory SHOULD be structured so Bridge can hand off context to Brain without reformatting or manual copy. |

### 11.2 Canonical Memory (Desktop)

| ID | Phase | Level | Requirement |
|---|---|---|---|
| MEM-CAN-01 | P2 | MUST | Long-term organizational memory MUST live on Desktop and MUST NOT be duplicated as the authoritative copy on any other machine. |
| MEM-CAN-02 | P2 | MUST | Canonical memory MUST be accessible to Bridge without requiring the human to manually transfer data. |
| MEM-CAN-03 | P3 | MUST | When a task completes, a summary of inputs, decisions, and outputs MUST be committed to canonical memory. |
| MEM-CAN-04 | P3 | TBD | TBD: Retention policy. How long do task records live before archival or deletion? Decision deferred. |

### 11.3 Human-Readable Memory Surface

| ID | Phase | Level | Requirement |
|---|---|---|---|
| MEM-HUM-01 | P1 | MUST | Human-readable memory browsing by day MUST be a required surface of Mission Control — not only an internal storage concern. The human operator MUST be able to read what was remembered without querying a database or reading raw files. |
| MEM-HUM-02 | P2 | MUST | Long-term memory entries MUST be accessible in a distinct view from daily operational memory so the human can distinguish persistent organizational knowledge from transient task context. |

### 11.4 Working Memory (GPU3060)

| ID | Phase | Level | Requirement |
|---|---|---|---|
| MEM-WRK-01 | P2 | MUST | Working memory on Forge (in-progress build artifacts, QA states, research scratch) MUST be cleaned up after task completion to prevent unbounded growth. |
| MEM-WRK-02 | P3 | SHOULD | Working memory SHOULD be readable by Brain so that research findings on Forge can be archived to canonical memory without going through Bridge. |

## 13. Legacy Asset Requirements

Legacy systems may contain real, recoverable value. These requirements protect against accidental destruction.

### 13.1 Inventory

| ID | Phase | Level | Requirement |
|---|---|---|---|
| LEG-INV-01 | P1 | MUST | A complete inventory of legacy assets on Desktop MUST exist in D5.3-LEGACY-ASSET-REGISTER.md before any existing system is modified, migrated, or decommissioned. |
| LEG-INV-02 | P1 | MUST | The inventory MUST include: system name, current status (running/stopped/unknown), estimated value/risk rating, and migration disposition (keep/migrate/decommission/evaluate). |
| LEG-INV-03 | P1 | MUST NOT | Legacy systems MUST NOT be powered off, wiped, or reconfigured until they appear in the inventory with a disposition decision. |

### 13.2 Migration

| ID | Phase | Level | Requirement |
|---|---|---|---|
| LEG-MIG-01 | P2 | MUST | Each legacy asset marked "migrate" MUST have a migration plan written before migration begins. |
| LEG-MIG-02 | P2 | MUST | Legacy migrations MUST be staged: run old and new in parallel until new is verified, then decommission old. No big-bang replacements. |
| LEG-MIG-03 | P2 | SHOULD | Commerce automation and revenue-generating systems SHOULD be migrated last, after lower-risk systems are proven in the new architecture. |
| LEG-MIG-04 | P1 | MUST | Previously built operational assets that already satisfy a Mission Control screen requirement (tasks, calendar, projects, memory, docs) MUST be evaluated for migration before any reimplementation begins. Building from scratch when a working prior system exists is waste and requires explicit justification. |

## 16. Non-Requirements

The following are explicitly out of scope for this requirements register. They are documented here to prevent scope creep.

| Item | Why Out of Scope |
|---|---|
| How to install OpenClaw | Runbook concern (D2.x), not a requirement |
| Which Discord bot library to use | Implementation detail, not a requirement |
| Agent personality or tone | Policy concern (D1.5+), not a requirement |
| UI for monitoring agents | No UI required — legibility via Mission Control docs and Discord is sufficient |
| Multi-user access to Mission Control | Single operator. Multi-user is not a current requirement. |
| High availability (active-active failover) | Out of phase. Durability (restart on failure) is sufficient for P0–P2. |
| Kubernetes or container orchestration | Not required at this scale. Individual process management is sufficient. |
| External customers or SLAs | The organization serves internal use cases only through P3. |
| Agent self-modification of code | Prohibited until explicit Autonomy Level grant. Not a Phase 0–2 requirement. |
| Automatic scaling | No cloud infra. Fixed hardware. Scaling is not a current concern. |
