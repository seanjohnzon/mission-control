# D2.0 — PHASE 1 ACTIVATION RUNBOOKS
**Autonomous AI Organization — Minimum Runbook Set for Phase 1 Operational Activation**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## Purpose

This document contains the minimum set of runbooks required to bring the organization from current state (policy written, no live operation) to Phase 1 **activated** state (bridge running, Telegram connected, Anthropic inference active, Mac Mini baseline verified, version control initialized).

**Phase 1 activated ≠ Phase 1 complete.** These runbooks achieve activation — bridge is running and can accept work. Phase 1 is not declared **complete** until local inference is operational on Mac Mini (D1.3 BRG-INF-01, TASK-003/TASK-008). During the period between activation and Phase 1 completion, bridge operates on Anthropic cloud inference as a declared bootstrap state (CHG-002, D1.5 §9 bootstrap exception). The local inference gap must be closed before Phase 1 exits and before any delegated unattended operation is approved at Level 2+.

Each runbook is a sequential checklist. Steps are atomic: each step has a verification check before moving to the next. No step is marked done until its verification passes.

**Perplexity is not a Phase 1 activation prerequisite.** Perplexity (D5.7 CAP-006) is a planned controlled capability addition that comes after Phase 1 activation. It is not required for any runbook in this document (RB-01 through RB-05). Perplexity evaluation and scope decision are tracked in TASK-013 and TASK-014. Activating Perplexity requires a separate D1.12 adoption proposal and D1.15 change record — separate from and subsequent to Phase 1 activation.

**Governing policies:** D1.13 (environments), D1.14 (secrets), D1.15 (change control), D1.12 (capabilities)
**Sequence dependency:** Runbooks must be executed in order. AP-00 (Activation Pack) is a human-side preparation step that precedes all runbooks. RB-01 (git init) must complete before any other runbook produces a committed artifact. RB-02 (Telegram) must complete before bridge can communicate. RB-03 (Anthropic) must complete before inference is active. RB-04 (Mac Mini baseline verification) must complete before the production environment is declared operational. RB-05 (bridge startup) is the final activation step.

**Activation Pack:** The full operator-ready Phase 1 Activation Pack — including all prerequisites, required inputs, human one-time provisions, the exact activation sequence with step-by-step detail, stop conditions, and the final go/no-go checklist — is in **D2.1-ACTIVATION-PACK.md**. Complete D2.1 Section N (Final Human Go/No-Go Checklist) before beginning AP-00.

## RB-02 — Telegram Bot Setup and Token Registration

**Purpose:** Create the Telegram bot, register the token metadata in the secret registry, store the value in the Mac Mini keychain, and verify bridge can send a test message.
**Authority:** D1.14 §14 Phase 1 Required; D1.14 §16 Priority 4
**Risk class:** Medium (Risk Class 3 secret)
**Change record:** New CHG record per D1.15 §4.8 (secrets scope change)

### Steps

**Step 1 — Create Telegram bot via BotFather**
On Telegram app (any device):
1. Open `@BotFather`
2. Send `/newbot`
3. Choose a name (e.g. "Mission Control Bridge")
4. Choose a username (e.g. `mc_bridge_bot` — must end in `bot`)
5. BotFather responds with the bot token (format: `1234567890:ABC-DEF...`)

DO NOT paste this value anywhere in Mission Control documents. Keep it only in the conversation with BotFather until you store it in the keychain.
Verification: BotFather confirms bot creation and provides token. ✓ / STOP

**Step 2 — Update SEC-002 metadata in secret registry**
Open `registry/secrets/SECRET-REGISTRY.md` and fill in the blank fields for SEC-002:
```
approval_date:    [today YYYY-MM-DD]
activation_status: registered
rotation_due:     [today + 180 days]
last_rotated:     [today]
```
Do NOT write the token value. Write metadata only.
Verification: SEC-002 entry updated; no token value in the file. ✓ / STOP

**Step 3 — Store token in Mac Mini keychain**
On Mac Mini terminal:
```
security add-generic-password \
  -s "mc-bridge-telegram-bot-token" \
  -a "bridge" \
  -w "<paste-token-here>" \
  -T "" \
  -U
```
Replace `<paste-token-here>` with the actual token. After running, the token is in the keychain and should not be typed again in terminal history or elsewhere.

Clear terminal history if needed:
```
history -c
```
Verification: `security find-generic-password -s "mc-bridge-telegram-bot-token" -a "bridge"` returns entry (without `-w` flag — do not print the value). ✓ / STOP

**Step 4 — Get chat ID for ops-channel**
Send a message from your Telegram account to the bot (start a conversation or add it to a group). Then retrieve the chat ID using the token:
```
curl "https://api.telegram.org/bot<TOKEN>/getUpdates"
```
Note the `chat.id` value from the response. This is the ops-channel ID bridge will send to.
Verification: chat ID retrieved and noted. ✓ / STOP

**Step 5 — Test send from bridge**
Configure bridge with the keychain secret retrieval and ops-channel chat ID. Send a test message:
```
Telegram API test: curl \
  "https://api.telegram.org/bot<TOKEN>/sendMessage" \
  -d "chat_id=<CHAT_ID>&text=MC+Bridge+test+message+OK"
```
The token value should not appear in any persisted script — bridge retrieves it from the keychain at runtime.
Verification: test message appears in Telegram ops-channel. ✓ / STOP

**Step 6 — Activate SEC-002 in registry**
Update SEC-002 `activation_status` from `registered` to `active`.
Commit the metadata update:
```
git add registry/secrets/SECRET-REGISTRY.md
git commit -m "Activate SEC-002: Telegram bot token registered and active"
```
Verification: commit recorded; SECRET-REGISTRY shows `activation_status: active` for SEC-002. ✓ / STOP

**Step 7 — Update CAP-002 in D5.7**
Open `docs/D5.7-CAPABILITY-REGISTER.md` and confirm CAP-002 (Telegram Bot API) is listed with `status: active`. If it still shows the bootstrap placeholder, update `approval_date` to today and status to `active`.
```
git add docs/D5.7-CAPABILITY-REGISTER.md
git commit -m "Activate CAP-002: Telegram Bot API active in Phase 1"
```
Verification: D5.7 updated and committed. ✓ / STOP

**RB-02 complete.** Bridge can now communicate with the human via Telegram.

## RB-05 — Bridge Startup and Validation Cycle

Before starting bridge, confirm the following configuration is set in bridge's OpenClaw workspace config:
- Inference model: `claude-sonnet-4-6` (CAP-001)
- Secret retrieval: keychain, not hardcoded
- Telegram ops-channel ID: set (from RB-02 Step 4)
- Mission Control workspace path: `~/Cursor Projects/mission-control/`
- Task register path: [declare path — e.g. `registry/tasks/TASK-REGISTER.md`]

Verification: config file reviewed; no hardcoded secrets. ✓ / STOP

**Step 2 — Start bridge**
Start the OpenClaw gateway and bridge agent per the installation method. If LaunchAgent is configured, bridge should have started at login — confirm it is running:
```
curl -s http://localhost:18789/health
```
Verification: gateway health check passes. ✓ / STOP

**Step 3 — Bridge self-check**
Bridge runs its startup validation cycle:
1. Confirms gateway is accessible (local)
2. Loads SEC-001 from keychain — confirms loadable (does not print value)
3. Loads SEC-002 from keychain — confirms loadable (does not print value)
4. Pings Anthropic API with a minimal inference call — confirms active
5. Reads current `TASK-REGISTER.md` — confirms accessible
6. Reads `CHANGE-REGISTER.md` — confirms accessible
7. Reads `D5.7-CAPABILITY-REGISTER.md` — confirms accessible
8. Posts startup notice to Telegram ops-channel

Expected startup Telegram message:
```
[STARTUP] Bridge online. Phase 1 activation complete.
  - Gateway: OK
  - Inference (CAP-001 / Anthropic cloud): OK [BOOTSTRAP MODE]
  - Telegram (CAP-002): OK
  - Mission Control: OK
  - Secrets (SEC-001, SEC-002): loadable

Ready for directives. Note: Phase 1 bootstrap mode active. Local inference required before Phase 1 exit.
```

**Bootstrap mode note:** The `[BOOTSTRAP MODE]` flag in the startup message is correct and expected. It is not a warning — it is a declared state. It closes when TASK-008 (local model installed and stable) completes and bridge switches to local-primary routing.
Verification: startup message received in Telegram. ✓ / STOP

**Step 4 — First directive test**
