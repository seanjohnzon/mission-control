## 1. Purpose

This document is the canonical structural map of the system. It describes what machines exist, what runs on them, how they connect, and how the layers of the organization fit together.

**What this document anchors:**
- Which machines hold which runtime agents
- What the network looks like now vs. what it must become
- Where state, memory, and config live by machine
- How Mission Control, Telegram (primary channel), local inference, and tooling relate to each other
- What is decided vs. what is still open

**What this document is not:**
- It is not a requirements spec (see D1.3)
- It is not a policy document (see D1.5–D1.15)
- It is not a runbook (see D2.x)
- It does not define org roles or decision rights (see D1.4)

When any structural decision changes — a machine is added, a network path changes, an agent moves — this document is updated first. Downstream documents that reference architecture must be flagged for review when this file changes.

## 2. System Mission Context

The mission is to build and operate an autonomous organization of AI agents that produces real value 24 hours a day without requiring constant human attention. The human role is architect, director, and judge — not operator.

That mission requires a physical substrate. Agents don't run in the abstract — they run on machines, consume memory, use network paths, and call tools. This document describes that substrate: the actual hardware, the actual network, and the actual software topology the organization runs on.

The architecture is built to be durable before it is clever. Machines must stay reachable. State must not be lost. Local inference must work even when cloud APIs are unavailable. The communication layer must be async-first. Mission Control must remain the source of truth regardless of what else changes.

## 4. Runtime Architecture

### The Three Real Agents

The organization runs exactly three OpenClaw runtime agents:

```
bridge (Nami)    →  Mac Mini (10.0.0.152)   Routing, scheduling, comms brokering, gateway management
brain  (Franky)  →  Desktop  (10.0.0.251)   Engineering, builds, deployments, code execution
forge  (Chopper) →  GPU3060  (10.0.0.16)    QA, verification, security checks, health monitoring
```

Each agent runs on one machine. Each agent has one OpenClaw workspace. Each agent has one model configuration. Each agent has bounded, documented responsibilities.

### Logical Roles Live Inside Agents — They Are Not Separate Agents

Named roles are **logical labels** — they describe what a given agent is doing in a given prompt context. They are not separate OpenClaw agents, not separate processes, and not separate instances.

| Role | Lives In (D1.x) | Crew Name | What it does |
|---|---|---|---|
| Conductor | bridge | Nami | Coordinates task flow across agents |
| Broker | bridge | Nami | Routes messages, manages channel I/O |
| Navigator | bridge | Nami | Strategic planning, sprint management |
| Builder | brain | Franky | Code generation and execution |
| Engineer | brain | Franky | Architecture, deployments, infrastructure |
| Critic | forge | Chopper | QA, testing, evaluation |
| Watchdog | forge | Chopper | Health monitoring, security checks, incident detection |
| Researcher | brain/forge | Franky/Chopper | Reads sources, extracts signal |

### Why Agent Count Is Low at First

Fewer agents means fewer failure modes. Each agent is a runtime process on a real machine with real memory, real costs, and real failure modes. Adding a new agent before the existing agents have a proven track record adds complexity that compounds — coordination overhead, more things to monitor, more things to recover when they fail.

The logical role model gives us the flexibility of many specialized behaviors while keeping the operational footprint small. When a new capability is genuinely needed and cannot be absorbed into an existing agent's role, adding a fourth agent becomes a considered architectural decision — not an impulse.

```
    │   └── DHCP reservations confirmed for all crew machines

    │

                          │
```

```
                          ├── Franky / Desktop:          10.0.0.251
                          ├── Chopper / GPU3060:          10.0.0.16
                          └── [additional switch ports available]
```

**Confirmed design:**
- All crew machines are on the same flat `10.0.0.x` LAN. No NAT between them.
- Each machine has a DHCP reservation at the router — MAC-to-IP binding. IPs do not drift.
- Nami's Ethernet (en0) is primary. Wi-Fi (en1) is backup.
- The switch is a dumb Layer 2 switch. The router handles routing and DHCP.
- The OpenClaw gateway on Nami (`ws://10.0.0.152:18789`) is LAN-accessible. Franky and Chopper gateways are also LAN-accessible at their IPs on port `18789`.
- Nami does not act as a NAT router. All machines are peers on the same LAN.

**Open decisions (unchanged):**
- Whether to expose any crew gateway beyond the home LAN (Tailscale or similar) — Phase B+ decision, recorded in D1.9 BROKER_POLICY.md.
- Whether to add a management VLAN — deferred; flat LAN preferred at this scale.

## 8. Local Inference Topology

Local inference is required on every worker machine. Cloud inference is a fallback, not a primary dependency. This is a founding architectural constraint, not a preference.

### Mac Mini — bridge inference

- Handles bridge-level reasoning: task routing, scheduling decisions, message triage, short planning tasks
- Inference load is expected to be moderate and bursty — not continuous
- Apple Silicon Neural Engine available for lightweight models
- Model selection: must run comfortably on Apple Silicon without degrading gateway responsiveness
- Model TBD — likely a small-to-medium quantized model (e.g. quantized Qwen, Phi, or Llama family)

### Desktop — brain inference

- Handles memory synthesis, planning, commerce reasoning, and knowledge indexing
- Inference load is moderate and sustained — brain is expected to process and index continuously
- Hardware: TBD (CPU-only vs GPU-capable — confirm on integration)
- Model TBD — must support embedding generation and coherent long-context reasoning
- If Desktop has a capable GPU, it takes over some load from GPU3060 for non-build tasks

### GPU3060 Laptop — forge inference

- Primary high-load inference node
- Handles code generation, QA reasoning, research summarization, and any task requiring extended context or fast iteration
- RTX 3060 drives GPU-accelerated inference
- Model TBD — likely a quantized 7B–13B model; possibly larger with aggressive quantization
- forge is the experimental inference surface — new models are tested here before being promoted to other machines

### Cloud inference — fallback and supplement

- Used when local models are insufficient for a specific task (context length, capability gap)
- Routing policy governs when cloud is invoked — see D1.5 MODEL_POLICY.md (to be written)
- Provider: Anthropic API (claude-sonnet-4-6 or equivalent) as primary cloud fallback
- Cloud inference costs money per token — the Cost/Usage lane in D1.1 tracks this
- Cloud inference must never be the silent default. It must be explicit in routing policy.

### Proxy-style routing (experimental — not a dependency)

There is a concept of routing inference requests across machines — e.g., bridge offloading heavy inference to forge via a local API proxy. This is architecturally interesting and worth exploring in the Experiment lane. It is **not** a dependency for Phase 1 or Phase 2. Do not build for it. Do not block on it.

## 9. Memory Topology

Memory in this system is not a single store. It is layered by purpose and machine.

### Canonical Memory — Desktop (brain)

- The authoritative knowledge base of the organization
- Contains: indexed research, project history, decisions, world model, commerce state, prompt packs
- Managed by brain (Archivist role)
- Must be backed up. Data loss here is organizational loss.
- Indexing method: TBD (vector store, graph, hybrid — to be specified in D1.7 MEMORY_POLICY.md)

### Operational Memory — Mac Mini (bridge)

- Task queue state, routing decisions, session logs, scheduling history
- Managed by bridge (Conductor role)
- Path: `~/.openclaw/agents/main/` (currently default agent; will be bridge agent once defined)
- Session store: `~/.openclaw/agents/main/sessions/`
- Shorter retention — operational logs, not permanent knowledge
- Promotes important decisions and outcomes to canonical memory via brain

### Working Memory — GPU3060 (forge)

- Active build context, QA results, research packets in progress, experiment logs
- Managed by forge (Builder, Researcher roles)
- Short-lived by design — outputs are the artifact, not the working state
- Experiment outcomes that are adopted promote into canonical memory via brain
- Experiment outcomes that are retired go into D5.4 EXPERIMENT-REGISTRY.md

### Edge Receipts — Pixel / Seeker

- Minimal local state only — task receipts, notification logs
- Not a memory store
- Nothing important lives here permanently

### Memory Promotion Path

```
forge working memory
    → (on adoption/completion) → brain canonical memory

bridge operational memory
    → (on significant decisions) → brain canonical memory

edge receipts
    → (nothing promoted — ephemeral only)

brain canonical memory
    → (source of truth for org knowledge)
    → (read by all agents as needed)
```

## 10. Mission Control Architecture

### Where It Lives First

Mission Control is hosted on the Mac Mini at:
```
~/Cursor Projects/mission-control/
```

This is the current working location. It may migrate in later phases if a better host is identified (e.g., a shared volume, a dedicated lightweight server, or a git-backed remote). Until then, it lives on the Mac Mini, which is always on and always reachable.

The `docs/` subdirectory holds the canonical D-series document set. Additional subdirectories will be added as Mission Control grows (e.g., `tasks/`, `status/`, `registry/`, `proofs/`).

### MVP Contents

The first version of Mission Control must contain and keep current:

| Element | What it tracks |
|---|---|
| Task queue | Active, queued, and blocked tasks with owner and status |
| Machine status | Which nodes are online and what they are running |
| Document set | Which D-series documents exist, their version, and completeness status |
| Delegation status | Which task types are delegated and at what maturity level (0–5) |
| Proof links | References to output artifacts and logs from completed delegations |
| Improvement intake | Incoming research signals, experiments queued for review |
| Experiment registry | Log of experiments run, outcomes, and adoption decisions |

### Implementation Form

The MVP form is a structured directory with markdown files and optionally a lightweight status dashboard. A full web UI is deferred. The design principle is: **legibility before beauty**. An agent or a human must be able to read Mission Control's current state in under two minutes without a GUI.

Specific implementation choices (static site, Notion, custom app, etc.) are deferred to D1.13 ENV_CONTROL.md and the Phase 1 architecture review.

## 11. Communications Architecture

### Telegram — Primary Communication Channel

- Primary communication channel between agents and human (D1.11 §3, §7.1)
- Bridge posts results, digests, alerts, and status updates via the Telegram bot (SEC-002)
- Human issues directives via Telegram for bridge to receive and route
- Telegram is **not** the source of truth — it is the delivery mechanism for notifications and directives
- Telegram state is ephemeral relative to Mission Control — what matters is filed in Mission Control, not left in chat

### Mission Control — Source of Truth

- All task assignments, delegations, plans, and outcomes are recorded in Mission Control
- Telegram messages that contain decisions or outputs must be reflected in Mission Control
- Agents do not treat Telegram history as authoritative — they treat Mission Control documents as authoritative

### Design Principle: Minimal Channel Footprint

- Operational messages route through one primary channel (ops-channel)
- Alerts are distinct from digests; escalations are visually marked
- Channel structure details are governed by D1.11; this document defines the architecture intent only

### Discord

Discord is optional. It is not the primary channel and is not activated by default. Discord may be configured as an explicitly approved secondary channel only if the human decides to activate it, with the bot token registered per D1.14, and usage rules documented in D1.11 and D1.9. Discord is not a fallback for Telegram outages — see D1.9 §3 Principle 1 and D1.11 §7.3.

## 12. Capability Architecture

Capabilities are acquired in layers, gated in order. No new capability type should be adopted before the previous layer is stable.

```
Layer 1 — Built-in tools        (OpenClaw native: browser, exec, memory, cron, files)
Layer 2 — Skills / plugins      (OpenClaw skill registry, community skills, custom skills)
Layer 3 — Brokered tools        (API integrations, webhooks, third-party services via bridge)
Layer 4 — Automation substrates (n8n workflows, browser automation, custom scripts)
Layer 5 — Custom integrations   (built-in-house for specific org needs)
```

**Rules:**
- A new capability must have a clear owner agent and a documented purpose before it is activated
- Capability acquisition is recorded in D5.2 SKILL-REGISTRY.md
- Capability adoption decisions go through the Skill/Plugin lane in D1.1
- No capability is permanently installed without a deliberate adoption decision
- Capabilities can be retired — retirement is recorded

**Tool Lane Architecture:**

Three designated tool lanes serve specific task classes within this capability model. Agents self-manage lane selection within approved policy:

| Lane | Tool | Category | Task Classes | Current Status |
|---|---|---|---|---|
| Orchestration | OpenClaw | CAP-INF (active) | Primary autonomous runtime. Task dispatch, scheduling, gateway, inter-agent routing. Default lane for approved backlog work and Mission Control buildout. | Active — Phase 1 |
| Engineering (supervised) | Cursor | CAP-SKL (active) | Supervised/manual engineering lane. Code, QA, document authoring, configuration. Requires human-in-the-loop or explicit task assignment. Not the autonomous runtime. May be operated indirectly via device-mediated control. | Active — Phase 1 (supervised) |
| Live Research / UI-Pilot | Perplexity | CAP-SVC (candidate) | Live web retrieval, current information, browser/computer-use. Preferred UI-pilot lane for app/device/chatbox interaction including indirect Cursor usage. | Candidate — D5.7 CAP-006 |

**OpenClaw is the primary autonomous runtime.** Approved backlog work defaults to OpenClaw. Cursor is the supervised/manual engineering surface — not the swarm's autonomous executor. Cursor may be operated indirectly through device-mediated control (Perplexity or OpenClaw navigating the Cursor UI) without requiring direct API integration. This is a documented access path.

**Perplexity is the preferred UI-pilot lane** for tasks requiring browser, app, device, or chatbox interaction. OpenClaw orchestrates; Perplexity operates UI-first tools when that is the best access path.

Perplexity is a callable external service invoked by agents during task execution — not a runtime agent, not a model, and not an orchestration layer. It produces outputs (research packets, web facts) that travel through bridge (Broker) before reaching human-facing channels or canonical memory. Activation requires the D1.12 adoption process and a D1.15 change record.

## 13. Environment Architecture

### Stable vs. Experiment

| Environment | Purpose | Location |
|---|---|---|
| Production | Live org operation — agents running real tasks | Mac Mini (bridge), Desktop (brain), GPU3060 (forge) |
| Sandbox | Experiments, model tests, new tool evaluation | GPU3060 (forge) — isolated from production data |
| Dev | Local testing of new code or agent config before promotion | MacBook Air or dedicated forge path |

### Machine Responsibility Boundaries

- Mac Mini: production gateway and bridge only. Experiments that could affect gateway stability do not run here.
- Desktop: canonical memory is production. Separate experiment path for brain research tasks.
- GPU3060: primary experiment surface. Safe to break here. Outputs are promoted to production only after validation.

### No Secret Leakage

Production secrets (API keys, tokens, credentials) do not enter experiment environments. Experiment environments use test credentials or no credentials. This is enforced by configuration, not trust.

See D1.14 SECRETS_POLICY.md (to be written) for full secrets management design.

## 14.5 Gateway Federation Architecture

_Added 2026-03-16 (CREW-003). Documents the three-gateway topology confirmed during crew activation._

### Overview

The crew operates three **independent OpenClaw gateways**, one per machine, connected over the home LAN. This is a federated model — each gateway manages its own agent, sessions, and auth; inter-crew communication happens via authenticated gateway RPC calls.

### Gateway Endpoints

| Crew | Machine | Gateway URL | Auth |
|---|---|---|---|
| Nami | Mac Mini (10.0.0.152) | `ws://10.0.0.152:18789` | Token-authenticated |
| Franky | Desktop (10.0.0.251) | `ws://10.0.0.251:18789` | Token-authenticated |
| Chopper | GPU3060 (10.0.0.16) | `ws://10.0.0.16:18789` | Token-authenticated |

All three gateways share the same inter-crew token. Nami is the **orchestrator** — it dispatches tasks to Franky and Chopper by calling `chat.send` on their gateways.

### RPC Protocol

Nami dispatches tasks via:
```bash
OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1 openclaw gateway call chat.send \
  --url ws://<crew-ip>:18789 \
  --token <shared-token> \
  --params '{"message":"TASK FROM NAMI — ...","sessionKey":"agent:main:main","idempotencyKey":"CREW-XXX-assign"}' \
  --expect-final --timeout 120000 --json
```

**Idempotency keys** prevent duplicate task delivery. Format: `CREW-XXX-assign` for assignment, `CREW-XXX-verify` for verification requests.

### Device Pairing

Franky and Chopper are paired as **nodes** to Nami's OpenClaw instance (in addition to running their own gateways). This dual-mode setup allows:
- Nami to dispatch tasks via RPC to crew gateways
- Crew machines to appear as paired nodes for camera/screen/notification access

### Current Limitations

- `--expect-final` on `chat.send` returns `started` status but does not await the crew agent's reply. Responses are asynchronous — Nami polls or the crew agent messages back on completion.
- Gateway calls require `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` for LAN ws:// connections (not wss://).

## 16. Guardrails

These are the non-negotiable constraints this architecture enforces. They do not change without a deliberate architectural decision recorded in this document.

1. **Three real agents at launch.** bridge, brain, forge. No fourth agent without explicit decision.
2. **Logical roles are not separate agents.** Conductor, Broker, Archivist, and others live inside existing agents. They are not spawned as new processes.
3. **Local inference runs on every worker machine.** Mac Mini, Desktop, GPU3060 all run local models. Cloud is the fallback, not the default.
4. **Mission Control is the source of truth.** Telegram is the primary delivery channel. n8n is an execution substrate. Neither replaces Mission Control.
5. **Temporary workarounds must not persist.** If a hack is needed (e.g., Internet Sharing, a manual IP assignment), it is documented as temporary and has an explicit exit condition.
6. **Documents precede implementation.** New capabilities, new agents, and new integrations require documentation before build. The spec is not retroactive.
7. **The MacBook Air is not a server.** Org operation does not depend on it being awake.
8. **Pixel and Seeker are edge surfaces, not agents.** They do not host runtime agents in early phases.
