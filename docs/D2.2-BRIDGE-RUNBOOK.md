## 3. Normal Operations

### 3.1 Starting Bridge

Bridge auto-starts via launchd (`KeepAlive`, `RunAtLoad`). Manual start:

```bash
openclaw gateway start
```

Or via launchctl:
```bash
launchctl load ~/Library/LaunchAgents/ai.openclaw.gateway.plist
```

### 3.2 Stopping Bridge

```bash
openclaw gateway stop
```

Or:
```bash
launchctl unload ~/Library/LaunchAgents/ai.openclaw.gateway.plist
```

### 3.3 Restarting Bridge

```bash
openclaw gateway restart
```

### 3.4 Checking Status

```bash
openclaw status
# or
openclaw gateway status
# or health check:
curl -s http://127.0.0.1:18789/health
```

Expected healthy response: `{"ok":true,"status":"live"}`

### 3.5 Viewing Logs

```bash
# Live log follow
openclaw logs --follow

# Gateway log file
tail -f ~/.openclaw/logs/gateway.log

# Error log
tail -f ~/.openclaw/logs/gateway.err.log
```

### 4.3 Telegram Troubleshooting

| Symptom | Check | Fix |
|---|---|---|
| No messages received | `openclaw status` — check Telegram channel status | Restart gateway |
| Messages sent but not delivered | Check bot token validity | Verify SEC-002 in `.env` |
| Group messages dropped | Group policy is `allowlist` — group must be in allowlist | Add group to allowlist in `openclaw.json` or set `groupPolicy: "open"` |
| Bot not responding | Gateway may be down | `openclaw gateway restart` |

| Symptom | Check | Fix |
|---|---|---|
| Ollama not responding | `curl http://127.0.0.1:11434/` | Open Ollama app or `ollama serve` |

## 6. Configuration Management

### 6.1 Config Files

| File | Purpose | Edit Frequency |
|---|---|---|
| `~/.openclaw/openclaw.json` | Gateway config (channels, auth, commands) | Rare — via `openclaw configure` or gateway tool |
| `~/.openclaw/.env` | Runtime secrets (API keys, tokens) | Rare — only on key rotation |
| `~/.openclaw/agents/main/agent/models.json` | Model routing (Ollama endpoints) | Rare — only when models change |

### 6.2 Making Config Changes

Always use the gateway tool for config changes (validates + restarts):
```
gateway config.patch → for partial updates
gateway config.apply → for full config replacement
```

Never edit `openclaw.json` directly without restarting the gateway.

### 6.3 Validating Config

```bash
openclaw config validate
```

## 7. Recovery Procedures

### 7.1 Gateway Won't Start

1. Check logs: `~/.openclaw/logs/gateway.err.log`
2. Validate config: `openclaw config validate`
3. Check Node.js: `~/.local/node22/bin/node --version` (expect v22.22.1)
4. Check port conflict: `lsof -i :18789`
5. If all clear, try manual start: `openclaw gateway start`

### 7.2 Lost Connectivity to Telegram

1. Check internet: `ping -c 3 api.telegram.org`
2. Check bot token: Verify `TELEGRAM_BOT_TOKEN` in `~/.openclaw/.env`
3. Restart gateway: `openclaw gateway restart`
4. If token is invalid: Regenerate via @BotFather, update `.env`, restart

### 7.3 Anthropic API Errors

1. Check API status: `curl -s https://status.anthropic.com/`
2. Check key: Verify `ANTHROPIC_API_KEY` in `~/.openclaw/.env`
3. Check billing: Anthropic console
4. If key is invalid: Regenerate via Anthropic console, update `.env`, restart

### 7.4 Mac Mini Power Loss / Reboot

Bridge should auto-recover:
1. launchd starts gateway on login (`RunAtLoad`, `KeepAlive`)
2. Ollama app starts on login (if in Login Items)
3. Verify: `openclaw status` should show healthy within 60s of login

If not auto-recovering:
1. Check launchd: `launchctl list | grep openclaw`
2. Manual start: `openclaw gateway start`
3. Check Ollama: `ollama list`

### 7.5 Disk Space Low

1. Check: `df -h /`
2. OpenClaw logs: `du -sh ~/.openclaw/logs/`
3. Ollama models: `du -sh ~/.ollama/models/`
4. Clear old logs if needed (keep last 7 days minimum)

```bash
# Check current version

```

**Rule:** Always check CHANGE-REGISTER before and after updates. File a CHG record for non-trivial updates.

### 8.2 Ollama Update

```bash
# Check current version
ollama --version

# Update: re-download from ollama.com or brew upgrade ollama
# Re-verify models after update: ollama list
# Re-run stability test if major version change
```

### 8.3 Node.js Update

Node.js is managed at `~/.local/node22/`. Updates should be tested before production use. File a CHG record.

### 8.4 macOS Update

**Always ask human before applying macOS updates.** macOS updates can:
- Break launchd configs
- Change security permissions
- Require re-approval of terminal/automation access
- Cause unplanned reboots
