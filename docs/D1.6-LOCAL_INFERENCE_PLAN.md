## 1. Purpose

This document defines the local inference plan for each worker machine in the organization. It translates the model classes and routing policy established in D1.5 into a machine-specific plan: what runs where, under what constraints, in what order, and what must be surveyed before decisions can be finalized.

**What this document does:**
- States what is known and what is unknown about each machine's inference capacity
- Defines the target local model classes per machine, consistent with D1.5 §4 and §9–11
- Sets resource budgets and constraints to protect machine stability
- Defines the benchmark plan that will resolve open decisions
- Describes rollout order and failure handling
- Marks all decisions that cannot be made until hardware is surveyed

**What this document is not:**
- Not a setup guide or install runbook (see D2.x)
- Not a model policy — routing rules, cloud fallback conditions, and cost controls live in D1.5
- Not a benchmark report — no benchmark results exist yet; this document plans the benchmarking, not the outcomes
- Not a purchasing guide — hardware acquisition is out of scope

When hardware is surveyed and benchmarks are run, this document is updated with confirmed values and TBD entries are resolved. D1.5 is updated concurrently if specific model selections change the routing behavior.

## 3. Planning Principles

These principles govern all local inference decisions in this document. They are consistent with D1.5 §3 and extend them at the machine level.

1. **Gateway stability is non-negotiable on Mac Mini.** The Mac Mini runs the OpenClaw gateway 24/7. A local inference model that consumes enough RAM or CPU to destabilize the gateway is not acceptable, regardless of capability. Model ambition yields to uptime.

2. **Survey before selecting.** No specific model is chosen for Desktop or GPU3060 until those machines are powered on and their hardware is confirmed. Choosing a model for unknown hardware is waste.

3. **Nano first, mid second.** On Mac Mini, the Local-Nano model is installed and verified stable before any Local-Mid model is added. Adding both at once before stability is confirmed is unnecessary risk.

4. **Forge is the experimental surface.** New model stack options, quantization approaches, and serving configurations are tested on forge (GPU3060) first. Stable, proven options are then considered for promotion to Mac Mini or Desktop.

5. **Models live on their machine.** Local inference models are stored on the machine that runs them. They are not served from a shared network path. Model storage is local disk only.

6. **Resource budgets are set before models are installed.** Maximum RAM and CPU limits for inference are decided per machine before any model is installed. A model that cannot fit within the budget is not installed, regardless of capability.

7. **Serving stack choice follows hardware reality.** The inference stack (Ollama, llama.cpp, or other) is chosen based on what the hardware and OS support reliably, not on what is newest or most popular. Stability wins.

8. **No model runs in production without a passing stability check.** A model is not used for live bridge, brain, or forge tasks until it has passed a defined stability test: runs a defined set of prompts without crashing, within the resource budget, for a defined duration.

## 5. Mac Mini Local Inference Plan

### Context

Mac Mini hosts the bridge agent and the OpenClaw gateway. It is always-on. Local inference here must coexist with the gateway without competing for resources in a way that risks uptime. This is a constrained environment, not a high-throughput inference server.

### Target Model Classes

Per D1.5 §9:
- **Local-Nano** (routing, dispatch, status classification) — required before Phase 1 exits
- **Local-Mid** (task orchestration, comms generation, short-form reasoning) — target before Phase 1 exits; secondary priority after Nano is stable

### Inference Stack Candidates

The following stacks are candidates for Mac Mini. Final selection depends on hardware survey results and stability testing on the actual machine.

| Stack | Notes |
|---|---|
| **Ollama** | Simple setup, good Apple Silicon support via Metal acceleration, automatic model management, HTTP API. Leading candidate for Mac Mini given macOS environment and ease of integration. |
| **llama.cpp** | Lower-level, more control over resource limits. Requires more configuration. Suitable if Ollama proves too RAM-heavy for the gateway coexistence requirement. |
| **LM Studio** | Desktop GUI tool. Not appropriate for a headless server role. Excluded from Mac Mini consideration. |

Stack decision is made after RAM is confirmed. If RAM is tight (≤ 8GB), llama.cpp with strict context limits is preferred. If RAM is adequate (≥ 16GB), Ollama is the default candidate.

### Model Selection Criteria for Mac Mini

The Mac Mini model selection must satisfy all of the following:
1. Runs within the defined RAM budget (see §11) without evicting gateway process memory
2. Responds to a routing-class prompt (short input, structured output) in under 5 seconds on the available hardware
3. Does not peg CPU at sustained 100% during inference — gateway must remain responsive during model calls
4. Supports quantized variants (GGUF or equivalent) to fit within RAM budget

Specific model names are TBD. They are chosen from within the Local-Nano and Local-Mid parameter ranges (1–4B for Nano, 7–14B for Mid) after the RAM survey and stability test.

### Stability Test Definition (Mac Mini)

Before a model is used for any live bridge task, it must pass the following:
- Run 20 consecutive inference calls of routing-class length without crashing or stalling
- Complete each call within the response time target (TBD in §12 Benchmark Plan)
- Leave the OpenClaw gateway process responsive (pingable) throughout the test run
- Not exceed the defined RAM budget ceiling at any point during the test

## 6. Desktop Local Inference Plan

### Context

Desktop hosts the brain agent. Brain tasks include memory synthesis (Archivist), planning (Planner), and commerce logic (Merchant). These tasks are more reasoning-intensive than bridge tasks and benefit from a larger model. Desktop is not always-on — it runs on WOL-on-demand or a scheduled power window.

### Target Model Classes

Per D1.5 §10:
- **Local-Mid** (archiving, commerce monitoring) — required before brain agent is activated
- **Local-High** (planning, synthesis) — target for full brain capability; feasible only if Desktop has sufficient RAM

### Constraints

All of the following are unknown until Desktop is powered on and surveyed:
- RAM capacity
- CPU capability and core count
- Whether a discrete GPU is present (would enable GPU-accelerated Local-High)
- Available disk space for model storage
- OS and package manager

### Planning Assumptions (Provisional)

These assumptions hold until the hardware survey contradicts them:

1. Desktop will have at least enough RAM to run a Local-Mid model. If this assumption is false, the brain agent's scope is limited until hardware is upgraded.
2. Desktop may have a discrete GPU. If it does, Local-High models become viable. If it does not, Local-High class falls to CPU inference and is slow — which shifts high-quality planning tasks toward cloud fallback (Cloud-Premium, per D1.5 §10).
3. Desktop OS is assumed to be macOS or Linux. Windows is not assumed but is not excluded. Inference stack options differ by OS.

### Inference Stack Decision Tree (Desktop)

```
Desktop hardware confirmed
        │
        ▼
Does Desktop have a discrete GPU?
        │
        ├── YES → Is CUDA / Metal available?
        │              ├── YES → GPU-accelerated stack (Ollama with GPU, llama.cpp with GPU)
        │              └── NO  → CPU stack only; Local-High is slow but feasible
        │
        └── NO  → CPU-only stack. Local-Mid is primary. Local-High only if RAM ≥ 32GB.
```

### Completion Gate

Desktop inference plan cannot be finalized until:
- Machine is powered on
- OS is confirmed
- RAM, CPU, and GPU are recorded
- Available disk is confirmed
- A hardware record is added to D1.2-ARCHITECTURE.md

## 7. GPU3060 Local Inference Plan

### Context

The GPU3060 Laptop (forge node) is the primary high-load local inference machine. It hosts the Builder, Critic, and Researcher roles. Inference tasks here are longer, more complex, and more frequent than on Mac Mini. The RTX 3060 GPU makes this the only machine currently confirmed to support GPU-accelerated inference.

### Target Model Classes

Per D1.5 §11:
- **Local-Mid** (CPU-capable fallback for research extraction, code scaffolding) — required before forge is activated
- **Local-High** (GPU-accelerated, for complex code generation, QA, content) — target for full forge capability; depends on VRAM confirmation

### VRAM Constraints

The RTX 3060 comes in two VRAM configurations:
- **6GB VRAM variant** — Can run quantized models up to approximately 7B parameters (Q4 quantization). Local-High class (30B+) is not feasible in GPU memory. Would require CPU offloading, which is slow.
- **12GB VRAM variant** — Can run quantized models up to approximately 13–20B parameters in GPU memory. Local-High class at 30B+ may require CPU offloading but mid-high (13–20B) is feasible fully on GPU.

**VRAM confirmation is required before any model is selected for forge.** The difference between 6GB and 12GB determines whether Local-High class is viable on GPU or must fall back to slower CPU inference.

### Inference Stack Candidates for Forge

| Stack | Notes |
|---|---|
| **Ollama** | GPU support via CUDA on Linux/Windows. Simple API. Good community support for GGUF models. Leading candidate if CUDA is available. |
| **llama.cpp** | Direct CUDA support. More control over layer offloading. Preferred if fine-grained VRAM management is needed on 6GB variant. |
| **text-generation-webui** | Feature-rich but heavier. More appropriate for research/experimental use than production inference. May be used for model evaluation only. |
| **vLLM** | High throughput serving. Requires Linux. Suitable for Phase 3 when forge serves inference to other agents over LAN. Not required for Phase 2. |

### Forge as Experimental Surface

Forge is designated as the experimental inference surface (D1.6 §3 Principle 4). New model stacks, new quantization approaches, and serving configurations are tested here before promotion. This means:
- Forge may run a different inference stack than Mac Mini or Desktop during experimentation
- Experimental stack choices on forge do not require policy updates if they are contained to the experimental task context (see D1.5 §13)
- Results from forge experiments feed into stack decisions for Mac Mini and Desktop via D5.4-EXPERIMENT-REGISTRY.md

### LAN Inference Serving (Phase 3 Target)

Per D1.5 §11, forge SHOULD serve inference requests to bridge and brain over LAN at Phase 3. This enables Mac Mini and Desktop to offload high-complexity tasks to forge's GPU without requiring cloud fallback.

LAN serving requirements:
- Forge must have a stable LAN IP before serving can be configured (requires switch uplink — see D1.2)
- Authentication method for the inference endpoint is TBD (see D1.9-BROKER_POLICY.md and D1.14-SECRETS_POLICY.md)
- The serving stack must support an HTTP API compatible with the agent's tool-calling interface

This is a Phase 3 target. It is not required for Phase 2 forge activation.

## 9. Serving Stack Decision Criteria

The inference stack for each machine is chosen based on the following criteria, in priority order:

| Priority | Criterion | Why It Matters |
|---|---|---|
| 1 | OS compatibility | Stack must run on the machine's actual OS without significant porting work |
| 2 | Hardware acceleration | GPU support (CUDA / Metal) dramatically improves throughput; CPU-only stacks are acceptable for Nano/Mid |
| 3 | RAM efficiency | Stack must not exceed the resource budget defined in §11 for the machine |
| 4 | API compatibility | Stack must expose an HTTP API that bridge, brain, or forge can call without custom integration work |
| 5 | Model format support | Stack must support GGUF or the format of the chosen model without conversion overhead |
| 6 | Stability track record | Stack must be actively maintained and have a known-stable version compatible with the hardware |
| 7 | Operational simplicity | Stack must be manageable without a GUI and must support headless / launchd / systemd management |

Ollama satisfies most criteria for macOS environments and is the default candidate for Mac Mini. GPU3060 stack depends on OS confirmation (CUDA availability on Linux vs Windows differs).

## 10. Model Storage and Paths

### Policy

- Models are stored on the local disk of the machine that runs them
- Models are NOT stored on network mounts, shared drives, or cloud storage
- Model storage paths are recorded in D1.2-ARCHITECTURE.md (paths table) once confirmed
- Models are not symlinked or referenced from paths outside the serving stack's native storage directory

### Candidate Paths (TBD — confirmed after stack selection)

| Machine | Candidate Path | Notes |
|---|---|---|
| Mac Mini | `~/.ollama/models/` (if Ollama) or stack-defined path | Confirmed after stack selection |
| Desktop | TBD | Depends on OS and stack |
| GPU3060 | TBD | Depends on OS and stack |

### Storage Budget

Model files are large. The storage budget per machine must account for:
- At least 2 models installed simultaneously (Nano + Mid, or Mid + High)
- Each model in a quantized format (GGUF Q4 or Q5): approximate sizes
  - 1–4B (Local-Nano): 1–3 GB per model
  - 7–14B (Local-Mid): 4–9 GB per model
  - 30B+ (Local-High): 18–25 GB per model in Q4 quantization
- At least one backup/experimental model slot (for forge only)

Minimum storage reserved for inference on Mac Mini: 15 GB (Nano + Mid + breathing room).
Minimum storage reserved for inference on GPU3060: 40 GB (Mid + High + experimental slot).
Desktop storage budget: TBD pending hardware survey.

Resource budgets are the hard limits for inference on each machine. A model that cannot operate within these limits is not installed in production.

### Mac Mini Resource Budget

| Resource | Budget | Rationale |
|---|---|---|

| RAM ceiling | No more than 60% of total system RAM | Protects gateway uptime under sustained load |

## 13. Failure Modes and Fallbacks

These are the defined failure modes for local inference on each machine and how they are handled. Cloud fallback rules are defined in D1.5 §12 and are referenced here, not restated.

### Mac Mini Failure Modes

| Failure | Detection | Response |
|---|---|---|
| Inference stack crashes | Gateway cannot reach local inference endpoint | Bridge marks itself DEGRADED in Mission Control. Posts to Discord. Routes cloud-permitted tasks to cloud fallback (D1.5 §12). Routing-only tasks are queued. |
| OOM — model evicted from RAM | Inference calls return errors or hang | Stack restart attempted. If repeated, model is flagged for replacement with a smaller variant. Human notified via Discord. |
| Inference too slow — gateway impacted | Gateway latency exceeds 500ms threshold | Inference is paused. Bridge queues tasks. Human notified. Stack or model reviewed. |
| Mac Mini reboots unexpectedly | Gateway restarts via launchd | Inference stack must also auto-restart. If stack does not auto-restart, bridge marks inference as UNAVAILABLE on next task attempt. |

### Desktop Failure Modes (Provisional)

| Failure | Detection | Response |
|---|---|---|
| Desktop offline | Bridge cannot reach brain agent | Tasks requiring brain are queued by bridge (Conductor). Not rerouted to other agents. Human notified if queue exceeds a timeout threshold (TBD in D1.9). |
| Local model unavailable on Desktop | Brain agent cannot complete task requiring inference | Task marked BLOCKED. Bridge escalates to Discord. Cloud fallback applies only if task type permits it (D1.5 §10). |

### GPU3060 Failure Modes (Provisional)

| Failure | Detection | Response |
|---|---|---|
| GPU3060 offline | Bridge cannot reach forge agent | Tasks requiring forge are queued. Research, build, and QA tasks do not fall back to bridge or brain. Human notified. |
| VRAM OOM during inference | Model crashes or returns OOM error | Retry with smaller quantization or fall back to CPU-based Local-Mid. If CPU-based also fails, task marked BLOCKED. |
| LAN serving unavailable (Phase 3) | Bridge cannot reach forge inference endpoint | Bridge falls back to local Mac Mini inference for tasks within Mac Mini's capability. Tasks requiring Local-High are BLOCKED. |

## 14. Rollout Order

Local inference is rolled out in the following sequence. Each step must be completed before the next begins.

### Step 1 — Mac Mini Survey (Current Blocker)
Survey Mac Mini hardware: RAM, CPU model, available disk. Record results in D1.2-ARCHITECTURE.md and §4 of this document.

### Step 2 — Mac Mini Stack Selection
Based on survey results, select inference stack (Ollama vs llama.cpp) and Local-Nano model. Decision recorded here in §5 and §9.

### Step 3 — Mac Mini Local-Nano Installation and Testing
Install Local-Nano model. Run stability test defined in §5. Verify gateway coexistence. Record benchmark results in D5.4.

### Step 4 — Mac Mini Local-Mid Installation (if Nano passes)
Install Local-Mid model. Run stability test. Verify both models can coexist within resource budget. Record results.

### Step 5 — Phase 1 Bridge Inference Active
At least one suitable local bridge model passing its stability test = bridge inference is operational. BRG-INF-01 is satisfied. A second model (Local-Mid) is the target but is not required to exit Phase 1. If Local-Mid is installed, run its stability test before activating it for production tasks; coexistence with Local-Nano must stay within resource budgets.

### Step 6 — Power On Desktop and GPU3060 (requires switch uplink)
Both machines require the basement switch to be uplinked before they are network-reachable. Physical action required first.

### Step 7 — Desktop Survey
Power on Desktop, confirm OS, RAM, CPU, GPU. Record in D1.2 and §4 of this document. Select stack and models for Desktop.

### Step 8 — GPU3060 Survey
Power on GPU3060, confirm OS, VRAM, CUDA availability. Record in D1.2 and §4. Select stack and models for Forge.

### Step 9 — Desktop and Forge Inference Installation
Install Local-Mid on Desktop. Install Local-Mid + Local-High on GPU3060. Run stability tests. Record results.

### Step 10 — Phase 2 Inference Active
Desktop and forge inference passing = brain and forge agents can be activated with full inference capability. FRG-INF-01 satisfied.

### Step 11 — Forge LAN Serving (Phase 3)
After LAN is stable and forge has a fixed IP, configure inference serving over LAN. Benchmark LAN latency. Activate for other agents.
