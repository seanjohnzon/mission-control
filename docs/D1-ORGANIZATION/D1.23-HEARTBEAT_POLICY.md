# UNIVERSAL HEARTBEAT RULES — Applies to ALL Agents

## Purpose
Heartbeats are the FALLBACK when cron chains fail. They also do routine health checks.

## When Heartbeat Fires
1. **Check cron chain health** — did your last cron executor run successfully?
   - If stale (>10 min since last run): heartbeat IS the fallback executor
   - Pick up the highest priority task and execute directly
   - Report: "Heartbeat fallback — cron chain was stale, executing [task]"

2. **Check your assigned tasks** — any queued/in-progress work?
   - If yes and no active cron handling it: execute the task
   - If all tasks done: HEARTBEAT_OK

3. **Quick health checks** (rotate, don't do all every time):
   - Gateway health: curl -s http://127.0.0.1:18789/health
   - Your own session errors: check .learnings/mistakes.md
   - Stale work: any task in-progress >2 hours with no session comment?

## Rules
- Heartbeats are on your REAL model (not qwen2.5:3b) — you CAN execute work.
- Keep heartbeat actions fast — don't try to do a full task if timeout is tight.
- If you find something broken: fix it if quick, otherwise create an MC task.
- Late night (23:00-08:00): only act on urgent blockers. Otherwise HEARTBEAT_OK.
- Don't repeat work already done by cron chains — check timestamps first.

## Priority Rule (Risk D fix — 2026-03-24)
**Active task session ALWAYS takes priority over heartbeat.**
- If you are executing a task when heartbeat fires → reply HEARTBEAT_OK, do NOT interrupt your work
- Heartbeat should only activate work when the agent has NO active task
- Never let a heartbeat pull you off a running task — that causes half-finished work and broken handoffs

## Session Isolation — CRITICAL FIX (2026-03-24)
**Heartbeat MUST run in a separate session from agent:X:main.**

Problem: Heartbeat and sessions_send both use agent:X:main by default. Heartbeat 
"check queue → nothing → HEARTBEAT_OK" pattern accumulates in main session and 
drowns out agent-to-agent messages sent via sessions_send.

Fix: Set `heartbeat.session` to a dedicated key so heartbeat runs in 
`agent:X:heartbeat` instead of `agent:X:main`.

Config example:
```json5
heartbeat: {
  session: "heartbeat",  // ← runs in agent:X:heartbeat, NOT agent:X:main
  every: "30m",
  lightContext: true,     // only loads HEARTBEAT.md, not full workspace
  target: "telegram",
  to: "-1003814818004",
}
```

This keeps agent:X:main clean for:
- sessions_send (agent-to-agent comms)
- Direct messages from Captain/crew
- Task execution

Reference: OpenClaw docs → /gateway/heartbeat → "session" field
