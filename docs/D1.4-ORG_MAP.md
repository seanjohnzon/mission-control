## 1. Purpose

This document defines who owns what in the autonomous AI organization. It maps runtime agents to machines, logical roles to agents, decision rights to people and agents, and responsibility for every declared surface and strategic track.

**What this document answers:**
- Which runtime agent is responsible for each class of work?
- Which logical role inside that agent handles a given task?
- What does the human operator own exclusively?
- Who decides when something escalates?
- Who owns each Mission Control screen?
- Who owns each strategic outcome track?

**What this document is not:**
- Not a policy document — it describes ownership, not the rules agents follow (see D1.5–D1.15)
- Not a runbook — it does not describe how to do work (see D2.x)
- Not a requirements spec — it does not define acceptance criteria (see D1.3)
- Not an architecture description — it does not describe the hardware or network (see D1.2)

When the organizational structure changes — a new agent is added, a role is reassigned, a strategic track changes ownership — this document is updated first. Documents that reference specific ownership must be reviewed when this file changes.

## 4. Logical Role Map

Logical roles are labels that describe what a runtime agent is doing in a given context. They are not separate OpenClaw agents. They are not separate processes. They do not have separate credentials. A single agent prompt context shifts between roles depending on the task it has been assigned.

| Role | Hosted In | Crew Agent | Responsibility | Does Not Do |
|---|---|---|---|---|
| **Conductor** | bridge | Nami | Coordinates task flow. Assigns tasks to Franky or Chopper. Manages cron, sprint, scheduled work. Owns tool lane routing governance. | Does not execute build or QA tasks itself. |
| **Broker** | bridge | Nami | Routes messages. Manages channel I/O (Telegram + Discord). Handles directives. Routes results and digests. | Does not make decisions about task content. Routes only. |
| **Navigator** | bridge | Nami | Sprint management, planning, backlog grooming, roadmap tracking. | Does not execute tasks directly. Plans and tracks only. |
| **Builder** | brain | Franky | Generates code, scaffolds projects, builds deployments. Returns artifacts to Nami or directly to Mission Control. | Does not deploy to production without Chopper QA at current autonomy levels. |
| **Engineer** | brain | Franky | Architecture, system design, infrastructure builds. | Does not override QA verdicts. |
| **Critic** | forge | Chopper | Evaluates build output. Runs tests. Produces pass/fail verdicts. Files QA reports to Discord #qa. | Does not modify code directly. Reviews only. |
| **Watchdog** | forge | Chopper | Monitors gateway health, SSH status, machine health. Flags anomalies. Posts to Discord #ops. | Does not make infrastructure changes. Monitors and flags only. |
| **Researcher** | forge | Chopper | Reads sources, extracts signal, produces research briefs. Also available from Franky for technical research. | Does not make adoption decisions. Surfaces findings only. |

### Role Activation Rule

A runtime agent may operate in multiple logical roles within a single session, but:
- It MUST NOT switch roles without a task boundary or explicit directive
- Each role activation MUST be traceable to a task in Mission Control
- Role scope is bounded by the task packet that triggered it

## 5. Human Operator Role

The human operator is the architect, director, and judge of the organization. The human is not a day-to-day operator.

### What the Human Owns Exclusively

| Domain | Human Action Required |
|---|---|
| Organizational structure | Adding, removing, or reassigning runtime agents or logical roles |
| Autonomy level grants | Increasing autonomy level for any task type at any agent |
| Financial authorization | Any transaction above approved routine commerce parameters |
| Infrastructure changes | New machines, new networks, OS changes, OpenClaw upgrades |
| New capability adoption | Approving new models, plugins, or tools from D5.2 SKILL-REGISTRY.md |
| Strategic track direction | Setting or changing objectives for any of the 5 strategic outcome tracks |
| Policy changes | Authoring or modifying any D1.x policy document |
| Phase advancement | Declaring a phase complete and authorizing the next phase to begin |

- Manually route tasks between agents
- Manually move task cards on the Kanban board (agents do this)
- Trigger research sweeps (forge does this on schedule)
- Write memory entries (brain does this)
- Issue detailed sub-task instructions for known task types (agents handle these within delegation specs)

| File a research brief | — | — | ✓ | ✓ | No |

| Approve new capability | ✓ | — | — | — | Always |

| Assign a logical role to a task | — | ✓ | ✓ | ✓ | No, within agent scope |

### Escalation Triggers

An agent MUST escalate to the human when any of the following occur:
1. A task requires an action outside its currently granted autonomy level
2. A task requires spending money or accessing financial systems beyond approved parameters
3. A task requires modifying infrastructure (machines, network, OS, OpenClaw config)
4. A task produces an ambiguous result that requires a strategic decision to proceed
5. A task has been blocked for more than a defined timeout (TBD in D1.9-BROKER_POLICY.md)
6. Two agents disagree on a decision that affects shared state (Mission Control, memory store)
7. Any unexpected external event occurs that was not anticipated in the task spec

## 9. Delegation and Escalation Model

### Delegation Flow

A task moves through the organization in the following sequence:

```
1. Human issues directive (Telegram or Mission Control)
         ↓
2. bridge (Broker) receives and acknowledges the directive
         ↓
3. bridge (Conductor) creates a task packet in Mission Control:
   - owner assigned
   - scope defined
   - route defined
   - output format defined
   - handoff target defined
         ↓
4. Task dispatched to the owning agent (brain or forge)
         ↓
5. Owning agent executes, produces proof artifact
         ↓
6. Result filed to Mission Control and/or canonical memory
         ↓
7. bridge (Broker) routes result to Telegram (ops-channel) for human review
         ↓
8. Human reviews result, approves, or issues follow-on directive
```

### Task Packet Requirements

A task is not delegated work until it has all of the following:
- **Owner** — which runtime agent is responsible
- **Scope** — what the task does and does not include
- **Route** — where the result goes when complete
- **Output format** — what proof artifact is expected
- **Handoff target** — who receives the result

A directive that skips this structure is chat, not delegation. Bridge (Conductor) is responsible for converting directives into proper task packets.

### Self-Managed Execution Flow

Alongside the human-directive flow above, the swarm operates a self-managed loop for approved task classes. This is not a separate system — it is the same task packet and Mission Control structure, initiated by Conductor rather than by a human directive:

```
1. bridge (Conductor) scans Mission Control backlog for eligible tasks
   (eligible = task class approved, automation level ≥ current maturity,
    no active block, machine within resource budget)
         ↓
2. bridge (Conductor) pulls the task, creates or validates its task packet
         ↓
3. Task dispatched to owning agent (or executed internally by Conductor)
         ↓
4. Owning agent executes, produces proof artifact
         ↓
5. Result filed to Mission Control and/or canonical memory
         ↓
6. bridge (Broker) routes completion notice to Telegram
         ↓
7. Human reviews digest or individual completion notices — no mid-task intervention needed
```

**Backlog hygiene** is Conductor's responsibility. Tasks in backlog that are stale, missing required fields, or blocked without a recorded reason are flagged to the human in the next digest — they are not silently accumulated.

**Workflow reuse.** When a task class has an approved workflow in D5.6-WORKFLOW_LIBRARY.md, Conductor executes using that workflow. When no workflow exists and the pattern repeats, Conductor surfaces it as a workflow candidate for human approval.

**The boundary.** Self-managed execution operates within approved scope, automation class, and autonomy level. Conductor does not pull tasks above its current autonomy level, does not expand task scope mid-execution, and does not initiate new task classes without human approval. Self-managed, but not self-authorizing beyond policy.

### Escalation Flow

When an agent cannot proceed without human input:

```
1. Agent marks task as BLOCKED in Mission Control with reason
2. bridge (Broker) posts escalation to Telegram (ops-channel) with:
   - task ID
   - blocking reason
   - what decision or action is needed
   - options (if applicable)
3. Human reviews and issues a decision via Telegram or Mission Control
4. bridge (Conductor) resumes the task with the human decision recorded
```

## 10. Autonomy Ownership Model

Autonomy is tracked per task type, not per agent. A single agent may hold different autonomy levels for different task types.

### Autonomy Level Reference

| Level | Name | Human Involvement |
|---|---|---|
| 0 | Manual | Human does the work or explicitly calls the agent each time |
| 1 | Assisted | Agent drafts, human executes |
| 2 | Supervised | Agent executes, human reviews every result |
| 3 | Delegated | Agent executes, human spot-checks on schedule |
| 4 | Trusted | Agent executes, human reviews only flagged items |
| 5 | Autonomous | Agent executes and self-corrects, human reviews digests |

### Current Autonomy State

All task types are currently at Level 0. No autonomy level has been granted to any agent for any task type.

### Autonomy Authority Table

| Action | Authority |
|---|---|
| Propose an autonomy level increase | Any agent or human |
| Approve an autonomy level increase | Human only |
| Grant autonomy for a new task type | Human only, after minimum 3 observed cycles at current level |
| Revoke an autonomy level | Human only, immediate, no agent approval required |
| Track autonomy level per task type | bridge (Conductor) maintains the register in Mission Control |

### Autonomy Register Location

The current autonomy level per task type is maintained in Mission Control. The register is updated by bridge (Conductor) when the human approves a change. The register is read-only for brain and forge.

## 11. Capability Ownership

New capabilities — models, plugins, skills, tools, automations — enter the organization through defined lanes. Ownership of each stage is explicit.

| Stage | Owner | Description |
|---|---|---|
| Discovery | forge (Researcher) or human | Identifies a new capability from research, X posts, or experimentation |
| Evaluation | forge (Critic) + human | forge benchmarks or tests the capability; human reviews the evaluation |
| Adoption decision | Human only | Human decides whether to adopt, defer, or reject |
| Registration | bridge (Conductor) | Files the adopted capability in D5.2 SKILL-REGISTRY.md |
| Installation | bridge or human (depending on risk) | Installs and configures the capability on the relevant node |
| Active use | Assigned runtime agent | Uses the capability within task scope |

**Rule:** No capability is used in production before it appears in D5.2 SKILL-REGISTRY.md with an adoption decision recorded.

### Perplexity — Candidate Callable Capability

Perplexity is a candidate external callable capability (D5.7 CAP-006) for live web research, current-information retrieval, and browser/computer-use tasks. It is:

- **A callable service** — agents invoke it for specific task classes; it is not a runtime agent, not a messaging endpoint, and not an orchestration layer
- **Not a subagent** — Perplexity does not hold task state, manage memory, or route messages; it returns outputs that agents process and route normally through bridge (Broker)
- **Not a model routing target** — Perplexity is outside the D1.5 inference routing policy; it is a CAP-SVC capability governed by D1.12 and D1.15
- **Candidate status** — not approved or active at Phase 1; activation requires the full D1.12 adoption process

Ownership of the Perplexity evaluation and adoption decision follows the capability ownership stages above. forge (Researcher / Critic) evaluates; human decides; bridge registers.

## 13. Org Guardrails

These rules protect the organizational structure from drift. They apply permanently unless explicitly revised by the human operator and documented in this file.

1. **Three agents, no more.** The organization runs on bridge, brain, and forge. No additional runtime agent is created without an architectural decision in D1.2 and human approval.

2. **Logical roles are not agents.** Conductor, Broker, Archivist, Planner, Merchant, Builder, Critic, and Researcher are context modes — not separate processes. Creating a new OpenClaw agent to represent a logical role is a violation of this rule.

3. **Human owns the top.** Strategic direction, phase advancement, autonomy grants, financial authorization, and policy changes are never delegated. These are permanently in the human domain.

4. **Autonomy is earned, not assumed.** Every new task type starts at Level 0. Promotion requires observed cycles and explicit human approval. No agent self-promotes.

5. **Mission Control is the record.** Agent-to-agent verbal agreements, Telegram or Discord messages, or in-flight context do not constitute organization state. State is only real when it exists in Mission Control.

6. **Bridge is the gateway.** All inter-agent task routing passes through bridge (Conductor). Agents do not call each other directly without bridge orchestrating the exchange.

7. **Memory is brain's domain.** No agent writes to the canonical memory store except brain (Archivist). Other agents file requests; brain writes.

8. **Proof or it didn't happen.** Every delegated task must produce a proof artifact — a file, log, or structured result — before it is considered complete. A channel reply (Telegram, Discord, or other) is not proof.

9. **No unbounded subagent expansion.** Agents may generate child prompts only within approved task classes, with bounded depth and defined handoff rules. Unbounded prompt chains are not delegation — they are drift.

10. **Pixel and Seeker are surfaces, not agents.** Edge devices receive tasks and return results. They do not make decisions. Until a formal edge agent architecture is defined, they operate as execution relays only.
