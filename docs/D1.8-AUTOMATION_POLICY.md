# D1.8 — AUTOMATION_POLICY
**Autonomous AI Organization — What May Be Automated, Under What Gates, and When Agents May Continue Without the Human**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document defines the automation policy for the organization. It governs what the system is permitted to run automatically, under what approval model, at what risk level, and when agents may continue working without the human present.

The goal of automation is to remove human execution bottlenecks. Policies here must be protective without reintroducing the bottleneck — if a task is approved up front and stays within its declared scope, it must be allowed to run to completion without requiring mid-task check-ins.

**This document governs:**
- What counts as automation and what counts as a one-time manual action
- Risk classification of automation types
- Approval gates for each risk level
- When unattended execution is permitted and when it is not
- Escalation triggers that halt automation and require human input
- Rules for scheduled jobs, cross-agent workflows, child prompt generation, and external actions
- Audit and proof requirements for all automated work

**What this document does not cover:**
- Specific runbook steps for implementing an automation — D2.x
- Delegation specs for individual task types — D3.x
- Memory policy for recording automation outputs — D1.7-MEMORY_POLICY.md
- Secrets and credentials used by automations — D1.14-SECRETS_POLICY.md
- Message routing and channel priorities — D1.9-BROKER_POLICY.md

## 2. Scope

An action is governed by this policy if it is:
- **Recurring** — runs on a schedule or trigger without the human initiating each run
- **Unattended** — runs to completion without requiring human presence or mid-run input
- **Sequenced** — involves multiple steps where the output of one step feeds the next automatically
- **Cross-agent** — involves one agent handing work to another without human mediation

An action is NOT governed by this policy if it is:
- A single human-initiated command with immediate human review of the result
- A one-time setup action performed by the human directly
- A model inference call made inside an already-approved task — those are covered by D1.5-MODEL_POLICY.md

If there is ambiguity about whether an action is automated, treat it as automated and apply this policy.

## 3. Policy Principles

1. **Approval gates must block the first run, not every run.** A task type that has been approved and tested may run repeatedly without re-approval as long as it stays within its declared scope, inputs, and output targets. Re-approval is triggered by scope changes, not cadence.

2. **Every automation has an owner, scope, route, output, and proof.** These five fields are required for every automation, regardless of risk level. An automation without any one of these fields is not a defined automation — it is an uncontrolled loop.

3. **No silent perpetual automation.** Every scheduled or triggered automation has a defined completion state, a defined check-in interval, or a defined stop condition. Automations that run indefinitely with no reporting or stopping condition are not permitted.

4. **Escalation gates are hardcoded.** Certain conditions always halt automation and require human input, regardless of the automation's approval level. These are listed in §8 and cannot be overridden by task packet scope.

5. **External actions require stronger gates.** Any action that writes to a system outside the organization's controlled environment (external APIs, external services, publishing platforms, financial systems) carries a higher risk level and requires a higher approval gate than equivalent internal actions.

6. **Irreversibility is the key risk multiplier.** Two actions that appear similar in scope may differ dramatically in risk if one can be undone and one cannot. Irreversible actions always require at least one risk level higher than their reversible equivalent.

7. **Machine capacity is a hard constraint.** Automation that would push a host machine beyond its defined resource budget is not started. It waits in queue or is rejected. This applies to all machines. (Source: D1.6 §11)

8. **Child prompt generation is bounded.** Agents may generate sub-tasks and child prompts only within the scope of an approved parent task, to a defined depth limit, with task IDs for each child. Unbounded chain generation is prohibited. (Source: D1.1 §3 principle 13)

9. **Mission Control is the source of truth; Telegram is the primary notification and escalation channel.** Automation state, completion status, and approval records live in Mission Control. Telegram is where agents post notifications, alerts, and results — it is not where approval decisions are recorded. (Source: D1.11 §3, §7.1)

10. **Tool lane selection is self-managed within approved policy.** Once tool lane assignments are approved (D1.1 Principle 16, D1.2 §12), agents select the correct lane — OpenClaw for orchestration, Cursor for engineering, Perplexity for live research when activated — without per-task human instruction. Tool lane changes require D1.15 change control; lane selection within approved policy is an execution decision. Screen-driving Cursor (visual GUI automation of the Cursor IDE) is not a primary production automation pattern and requires explicit task packet approval before use.

11. **Agents earn the right to run unattended.** All task types start at Level 0 by default. Promotion to any higher maturity level requires explicit human approval and the appropriate delegation spec or task class approval. Unattended execution is not permitted below Level 2. No task type reaches Level 2 without being evaluated at Levels 0 and 1 first. Advancement to Level 3+ requires a track record of successful supervised runs. See §6.

12. **Access minimization governs external action design.** When designing automated tasks that require access to external resources or services, agents prefer stable built-in paths over new API dependencies, and API-based access over device-mediated fallback. A new external access surface is introduced only when existing approved capabilities cannot fulfill the task. Device-mediated automation (computer-use, GUI piloting) is governed as a candidate capability class — it is never an ad-hoc improvisation. (Source: D1.1 Principle 17, D1.3 BRG-AMP-01)

## 4. Automation Classes

Automations are classified by their structural type. Risk level is assigned per instance based on what the automation does, not merely its class.

### Class 1 — Scheduled / Cron
Runs on a defined time schedule. Requires no external trigger. Examples: daily digest generation, weekly memory archival sweep, periodic health check.

### Class 2 — Triggered
Fires when a defined condition is met. The trigger may be a Telegram message, a file being deposited in a watched directory, a machine coming online, or a threshold being crossed. Examples: bridge receiving a directive via Telegram, forge activating when a build job arrives.

### Class 3 — Delegated Single-Step
Human assigns a task packet. Agent executes one bounded action and returns a result. No intermediate decisions by the agent. Examples: "summarize this article," "generate a code scaffold for this spec," "file this finding in memory."

### Class 4 — Delegated Multi-Step
Human assigns a complex task packet. Agent decomposes it into sub-steps, executes them sequentially, and returns a final result. Each sub-step must stay within the scope declared in the packet. Examples: research intake (fetch → extract → summarize → file), build + QA (scaffold → test → fix → report).

### Class 5 — Continuous Pipeline
Multi-agent workflow where the output of one agent automatically becomes the input for the next without human mediation between steps. Examples: Researcher on forge extracts findings → hands off to Archivist on brain for canonical filing → Conductor on bridge records completion. These are the highest-risk class and require explicit pipeline definition.

## 5. Risk Levels

Every automation instance is assigned a risk level before it is approved. The risk level determines the approval gate in §6.

### Level 1 — Low Risk
- Read-only or internal-write only
- Reversible — outputs can be deleted or undone
- No external system effects
- No financial, infrastructure, or publishing dimension

**Examples:** Reading a file from local disk, writing a task record to operational memory, generating a draft that is not published, filing a research finding to canonical memory, running a health check, fetching a public web page for research.

### Level 2 — Medium Risk
- Writes to persistent internal state or posts to internal communication channels
- May invoke model inference
- May involve cross-agent handoffs
- Outputs are retained and may influence future automation

**Examples:** Posting a digest to Telegram, submitting a memory promotion packet to Archivist, creating a new cron job definition, updating a task record status, handoff from bridge to forge, generating a report that will be reviewed.

### Level 3 — High Risk
- Writes to external systems, platforms, or services
- Publishes content that others may see
- Modifies system configuration or agent definitions
- Actions that are difficult but not impossible to reverse

**Examples:** Publishing a post to an external platform, calling an external API with side effects, modifying an OpenClaw agent definition, updating a DHCP reservation, posting to a public channel or storefront, adding a new model to the inference roster.

### Level 4 — Critical Risk
- Irreversible or financially consequential
- Infrastructure changes with system-wide impact
- Mass operations (bulk delete, bulk publish, bulk send)
- Financial transactions of any kind
- Changes to secrets, credentials, or access controls

**Examples:** Processing a financial transaction, deleting a canonical memory store, bulk-publishing listings, changing API keys, modifying firewall rules, any action where the undo cost is high or impossible.

## 6. Approval Model

### 6.1 Approval Levels by Risk

| Risk Level | Approval Required | Who Approves | Recurring After Approval |
|---|---|---|---|
| Level 1 — Low | Task class approval at phase entry | Human approves the task type when phase begins | Yes — runs without re-approval until scope changes |
| Level 2 — Medium | Task packet approval | Human approves the specific task packet before first run of that type | Yes — re-runs of the same type within same scope do not need re-approval |
| Level 3 — High | Explicit per-run approval OR pre-authorized template | Human approves each run, OR a named template with fixed inputs is pre-authorized | Template pre-auth: yes within template. Per-run: no — each run requires approval |
| Level 4 — Critical | Explicit per-action human approval | Human approves each individual action | Never — each action always requires approval regardless of prior history |

### 6.2 Autonomy Maturity and Automation Approval

Approval requirements connect to the Autonomy Maturity Ladder defined in D1.4 §8 and D1.1 §7. An automation type's maturity level gates what kind of approval model applies:

| Maturity Level | Automation Behavior |
|---|---|
| 0 — Manual | No automation approved; human acts directly |
| 1 — Assisted | Agent produces a draft or plan; human executes the action |
| 2 — Supervised | Agent executes; human reviews every result before next run |
| 3 — Delegated | Agent executes; human reviews spot-check on schedule; proof required |
| 4 — Trusted | Agent executes; human reviews only flagged items; escalation triggers active |
| 5 — Autonomous | Agent executes, self-corrects within scope; human reviews periodic digest |

**Current system maturity: Level 0 for all task types.** No automation is running unattended at Phase 1 start. Advancing levels requires track record: defined number of successful supervised runs before advancing to the next level. Track record counts are defined per task type in D3.x delegation specs, not in this document.

### 6.3 Scope Change Re-approval

A previously approved automation requires re-approval when any of the following change:
- The output target (where results are written or sent)
- The external services it contacts
- The input sources it reads from
- The machine it runs on
- The agent or role that executes it
- Any parameter that affects reversibility or external impact

Changes to scheduling cadence alone do not require re-approval if the action itself is unchanged.

### 6.4 Implicit Approval by Document

Certain automation types are implicitly approved by the existence of a completed policy or delegation spec document. Creating D1.8 does not activate any automation — it defines the policy under which automations may be approved. Explicit approval happens at phase entry checkpoints or via task packet dispatch.

### 6.5 Self-Managed Execution of Approved Work

Once a task class is approved at the appropriate maturity level, agents are expected to pull and execute eligible work without waiting for a human to prompt each pickup. The human approves the class, the risk level, and the rules. The swarm manages execution within those rules.

**This means:**
- Bridge (Conductor) scans the Mission Control backlog on a defined cadence and picks up eligible tasks
- Agents do not wait for the human to manually assign each eligible task; they pull from the approved queue
- Backlog hygiene — flagging stale, blocked, or malformed tasks — is an ongoing Conductor responsibility, not a human chore
- When a task class has an approved workflow in D5.6-WORKFLOW_LIBRARY.md, that workflow is used; improvising a new approach for a known pattern is not permitted
- Recurring patterns without an approved workflow are surfaced as workflow candidates for human promotion

**The self-management boundary is fixed:**
Agents may self-manage within approved scope. They may not:
- Pull tasks above their current autonomy level
- Expand scope mid-execution based on what they discover
- Initiate a new task class that has not been approved
- Promote a workflow to the library without human sign-off

Self-managed, but not self-authorizing beyond policy. (Source: D1.1 §3 principle 15, D1.3 BRG-SWM-01–04, SWM-AUTH-01)

## 7. Unattended Execution Policy

### 7.1 When Unattended Execution Is Permitted

An automation may run to completion without the human being present or monitoring if all of the following are true:

1. The automation is at Maturity Level 2 or higher for its task type
2. It has been approved at the appropriate approval gate for its risk level
3. Its scope, inputs, output targets, and proof requirements are fully declared in its task packet or cron definition
4. No escalation trigger condition (§8) is currently active
5. The target machine is within its resource budget (§12)

### 7.2 What "Run to Completion" Means

An automation running to completion means:
- It executes all declared steps
- It writes proof to the defined output location
- It posts a completion notice to the defined notification channel (Telegram)
- It updates the task record in Mission Control to `closed`
- It does NOT request mid-task input from the human unless an escalation trigger fires

The human is not required to be present. The human is required to exist — they will review the completion notice and proof artifact at their next active session.

### 7.3 What Unattended Execution Does NOT Permit

Unattended execution approval does not grant:
- Permission to expand scope mid-task based on what the agent discovers
- Permission to initiate new automations from within an automation without a declared child task boundary
- Permission to take Level 3 or Level 4 actions not declared in the original task packet
- Permission to contact external services not declared in the original task packet
- Permission to silence or suppress escalation triggers

### 7.4 Overnight and Multi-Hour Runs

Automations that are expected to run for more than 2 hours must:
- Have a declared expected duration range in their task packet or cron definition
- Post an in-progress notice at defined intervals (minimum: every 2 hours) to Telegram
- Have an explicit maximum wall-clock time after which the automation is paused and an alert is posted

## 8. Escalation Triggers

The following conditions always halt an active automation and require human input before the automation continues or restarts. These are hardcoded. No task packet, no approval level, and no agent may suppress or override them.

| # | Trigger | Required Response |
|---|---|---|
| 1 | **Scope boundary reached** — automation would need to exceed its declared scope to continue | Halt. Post alert to Telegram. Wait for human re-scoping or closure instruction. |
| 2 | **Unexpected external system response** — external API returns an unexpected error, auth failure, or response indicating the action may have had unintended effect | Halt. Post alert with response details. Do not retry automatically. |
| 3 | **Resource budget exceeded** — machine RAM, VRAM, or CPU exceeds defined budget during automation run | Halt inference or task. Post DEGRADED alert. Do not queue additional tasks until budget is restored. (Source: D1.6 §11) |
| 4 | **Financial dimension detected** — any automation step encounters pricing, payment, billing, or transaction data not declared in the original task packet | Halt immediately. Post alert. Level 4 approval required for any financial action. |
| 5 | **Irreversible action about to execute that was not declared** — agent is about to take an action it cannot undo that was not explicitly listed in the approved task packet | Halt before action. Post alert with description of the action. Wait for explicit human approval. |
| 6 | **Repeated failure** — same step fails 3 consecutive times | Halt. Post failure report. Do not continue. Human must review and re-dispatch. |
| 7 | **Communication system unreachable** — Telegram or the defined notification channel is unavailable for more than 15 minutes during an active automation | Halt new task dispatches. Continue current in-flight tasks to their next safe checkpoint. Post recovery notice when channel is restored. |
| 8 | **Credentials or secrets invalid** — authentication fails for any system the automation needs to contact | Halt. Post masked alert (no credential values in alert). Human must update credentials before automation can continue. (Source: D1.14-SECRETS_POLICY.md) |
| 9 | **Canonical memory unavailable during write phase** — automation was expected to commit to canonical memory but Desktop is unreachable | Queue the commit in operational memory. Post a DEGRADED alert. Do not retry writing to canonical indefinitely. |

When a trigger fires, the automation records the trigger event in operational memory with the trigger ID, timestamp, and context. It does not attempt to self-resolve escalation triggers.

## 9. Scheduled Automation Policy

### 9.1 Registration Requirement

Every scheduled (Class 1) automation must be registered before its first run. Registration requires:
- A cron definition with schedule, owner agent, task type, and output target
- A declared risk level
- A notation in Mission Control's task register (not just in OpenClaw's cron config)
- Approval at the appropriate gate for its risk level

An unregistered scheduled job is not an approved automation. It is an unauthorized loop.

### 9.2 Heartbeat and Liveness Checks

Heartbeat and liveness checks are not automations in the sense of this policy. They:
- Do not invoke model inference (Source: D1.5 §7)
- Do not write task records to memory (Source: D1.7 §6.2)
- Do not post to Telegram unless a threshold is crossed (failure or DEGRADED state)
- Are not registered in Mission Control's task register — they are registered only in OpenClaw's cron config

A heartbeat that triggers an alert is NOT a heartbeat that writes a task record. The alert is the output. The event that caused the alert (e.g. machine unreachable) generates the operational memory entry.

### 9.3 Schedules Must Be Bounded

Every scheduled automation must define:
- A maximum consecutive failure count (default: 3) before the schedule is paused and an alert is posted
- A maximum run duration (where applicable)
- A skip behavior: does it skip missed runs or does it catch up?

**Default skip behavior:** Skip missed runs. Do not catch up on missed cron windows unless the task type explicitly requires catch-up (and that is declared in the registration). Catching up on missed runs can cause cascading load on the host machine.

### 9.4 Schedule Density Limits

No more than 3 scheduled automations may be configured to fire within the same 5-minute window on Mac Mini. This protects gateway uptime. Schedules must be staggered. (Source: D1.5 §7 heartbeat rules, D1.6 §11 resource budget)

## 10. Child Prompt / Subagent Policy

### 10.1 When Agents May Generate Child Tasks

An agent may generate a child task or sub-task only when all of the following are true:
1. The parent task packet explicitly permits child task generation (field: `allows_children: true`)
2. The child task type is within the same approved automation class as the parent, or is a lower-risk class
3. The child task has its own task ID, owner, scope, output, and proof target — it is not an anonymous step
4. The depth limit is respected (see §10.2)

### 10.2 Depth Limit

The maximum delegation depth is 2 levels:
```
Level 0: Human → task packet → bridge (Conductor)
Level 1: Bridge → child task packet → brain or forge
Level 2: Brain or forge → atomic sub-step (no further child generation)
```

An agent at Level 2 does not generate further task packets. It completes its atomic work and returns a result. This limit is hardcoded and may not be extended by a task packet declaration.

### 10.3 Child Task Scope Inheritance

A child task may not inherit a wider scope than its parent. If the parent task is approved for internal-only actions (Level 1 risk), a child task generated by that parent may not contact external services (Level 3 risk). The child task inherits the parent's scope ceiling, not the child agent's general capability.

### 10.4 Unbounded Generation Is Prohibited

An agent may not generate an arbitrary number of child tasks in a loop based on what it discovers during execution. If a task is likely to generate a variable number of children (e.g. a research sweep that may find 1 or 50 articles), the maximum child count must be declared in the parent task packet. If the count would be exceeded, the agent stops at the declared limit and reports the cutoff.

## 11. Cross-Agent Workflow Policy

### 11.1 Bridge Owns the Task Dispatch Chain

Bridge (Conductor) is the orchestrator for all cross-agent workflows. Brain and forge receive tasks from bridge and return results to bridge. Brain and forge do not initiate cross-agent task dispatches to each other without bridge's knowledge. (Source: D1.4 §7)

### 11.2 Handoff Requirements

Every cross-agent handoff must include:
- The originating task ID (no orphan tasks)
- The receiving agent
- The declared scope for the receiving agent's contribution
- The output format and handoff target for the receiving agent's result
- A timeout: how long bridge waits for the result before escalating

A handoff that does not include all five fields is an incomplete task packet. The receiving agent must reject it and notify bridge.

### 11.3 No Silent Cross-Agent Communication

Agents do not communicate with each other outside of defined task packets. There is no side-channel, no shared memory that one agent writes and another reads without a declared handoff, and no peer-to-peer task initiation. All cross-agent communication routes through bridge or through the defined memory promotion pipeline (D1.7 §9).

### 11.4 Task Lineage is Traceable

Every task — parent and child — has a lineage: the chain of task IDs from the original human directive to the final result. This chain is recorded in each task's memory entry. An agent may not execute a task without a task ID that it can trace to a human-approved directive.

## 12. Machine Capacity / Parallelism Policy

### 12.1 Mac Mini Parallelism Limits

| Resource | Limit | Behavior When Exceeded |
|---|---|---|
| Concurrent inference tasks | 1 | New inference tasks queue; existing task finishes first |
| Concurrent non-inference bridge tasks | Up to gateway CPU budget | New tasks queue when CPU budget is reached |
| Scheduled jobs firing simultaneously | Maximum 3 per 5-minute window | Excess jobs are staggered by bridge |
| Active task records in operational memory | No hard limit, but daily summary trigger fires if count exceeds a high-water mark (TBD) | Alert posted; human reviews backlog |

(Source: D1.6 §11 Mac Mini resource budget)

### 12.2 Desktop Parallelism Limits

Desktop parallelism limits are TBD until hardware is confirmed. Design assumption: Desktop runs at most 1 Local-High inference task and 1 Local-Mid inference task simultaneously, within its RAM budget. Brain (Archivist) does not queue unlimited work against Desktop — bridge is responsible for pacing dispatch to brain. Specific limits are set after Desktop hardware survey and recorded in D1.6 §11.

### 12.3 GPU3060 Parallelism Limits

| Resource | Limit | Behavior When Exceeded |
|---|---|---|
| Concurrent GPU-accelerated inference | 1 | New GPU tasks queue |
| Concurrent CPU inference (parallel to GPU) | 1 if RAM allows | Queue if RAM budget would be exceeded |
| Active working directories | No hard limit, but size threshold triggers alert | Forge posts WARNING; human reviews |

(Source: D1.6 §11 GPU3060 resource budget)

### 12.4 No Cross-Machine Oversubscription

Bridge does not dispatch more tasks to brain or forge than those machines can hold within their resource budgets. Bridge is responsible for tracking what is in flight on each machine and pacing dispatch accordingly. If a machine's queue is full, bridge holds the task in its own operational queue rather than dumping it to the target machine.

## 13. External Action Policy

An external action is any action that writes to, reads from, or communicates with a system outside the organization's controlled environment. This includes: public web APIs, external platforms, publishing channels, storefronts, financial systems, external messaging services, and cloud services not operated by the organization.

### 13.1 Reading External Resources

Reading public information (web pages, public APIs, RSS feeds) for research purposes is **Level 1–2 risk** depending on whether it involves authentication. It does not require per-run approval if the task class has been approved and the source domains are declared in the task packet.

**Constraint:** External reads may not circumvent access controls, scrape at a rate that constitutes abuse, or access systems requiring credentials that are not registered in D1.14-SECRETS_POLICY.md.

### 13.2 Writing to External Services

Any write, publish, post, submit, or upload action targeting an external service is **Level 3 minimum**. It requires either:
- Explicit per-run approval, OR
- A pre-authorized template with fixed output targets and declared scope

**Before any external write is activated for an automation:** The target service, the type of content, the account or API key used, and the approval level must all be declared. Undeclared external writes are treated as escalation trigger #2.

### 13.3 Publishing and Content Distribution

Publishing content to an external platform (social media, blog, marketplace, storefront) is **Level 3 risk** minimum. The first publication of any new content type to any new platform requires explicit per-run approval. Pre-authorized templates may apply to subsequent runs of an identical content type to the same platform once a track record is established.

Automated mass publishing is **Level 4 risk** and requires explicit per-action human approval regardless of prior track record. Mass is defined as more than 10 items in a single automation run.

### 13.4 Financial Systems

Any automation that contacts a financial system — payment processor, bank API, marketplace payment system, billing platform — is **Level 4 risk**. No financial transaction may be executed by an automation without explicit per-action human approval. This applies regardless of transaction size.

Monitoring financial systems (reading balances, reading transaction history) is **Level 2 risk** and follows the standard approval model for that level.

### 13.5 Infrastructure and Configuration

Automation that modifies infrastructure — network configuration, DNS, firewall rules, machine configuration, server provisioning, access control lists — is **Level 4 risk**. Human approval required per action.

## 14. Failure / Recovery Policy

### 14.1 Failure Defined

An automation step has failed if it:
- Returns an error response that prevents the step from producing its declared output
- Exceeds its declared maximum wall-clock time
- Triggers an escalation trigger from §8
- Produces output that fails a declared validation check

### 14.2 Retry Rules

| Risk Level | Automatic Retry Permitted | Max Retries | Delay |
|---|---|---|---|
| Level 1 | Yes | 3 | 30 seconds between retries |
| Level 2 | Yes | 2 | 60 seconds between retries |
| Level 3 | No — alert and wait for human | 0 | — |
| Level 4 | No | 0 | — |

After the maximum retry count is reached, the automation is halted and an alert is posted to Telegram (ops-channel). It does not restart automatically.

### 14.3 Partial Completion Handling

If an automation completes some steps but fails partway through, it must:
1. Record which steps succeeded and which failed in the task's operational memory entry
2. Not silently retry only the failed steps as if the successful steps did not happen — the full context of partial completion must be included in the failure report
3. Not assume that successful steps can be rolled back — it must report what succeeded and what did not, and let the human decide whether to re-run, continue from the failure point, or close the task

### 14.4 Circuit Breaker

If the same automation class fails 5 times within a 24-hour window, a circuit breaker activates:
- No further runs of that automation class are started
- A DEGRADED alert is posted to Telegram (ops-channel)
- The circuit breaker remains active until the human reviews and explicitly resets it
- The human decides whether to fix the root cause, re-approve with modified scope, or retire the automation

### 14.5 No Self-Healing That Expands Scope

An automation may not attempt to self-heal by acquiring new permissions, contacting new external services, or expanding its declared scope. If the declared scope is insufficient to recover from a failure, the automation halts and escalates. Self-healing is limited to retrying the same action within the same scope.

## 15. Audit / Proof Policy

### 15.1 Proof Is Required for Every Automation Run

Every automation run must produce at least one proof artifact before it is considered complete. A proof artifact is a file, log, or structured result that confirms the automation ran, what it did, and what it produced. (Source: D1.1 §3 principle 12)

Proof artifacts are not optional. A run with no proof artifact is an unconfirmed run, regardless of what the agent reports in Telegram.

### 15.2 What Counts as Proof

| Automation Type | Acceptable Proof |
|---|---|
| Research task | Written summary file at a declared path, OR a canonical memory entry filed by Archivist |
| Build task | Build output at declared path, OR test pass/fail log with result |
| Memory archival | Confirmation from Archivist with canonical ID of committed entry |
| Scheduled digest | The digest file itself at the declared output path, PLUS the Telegram message reference |
| Cross-agent handoff | Task completion record in operational memory with receiving agent's output reference |
| External read | Extracted data file or structured summary at a declared path |
| External publish | Platform confirmation (post ID, listing ID, or URL) recorded in the task entry |

### 15.3 Proof Location

Proof artifacts are stored:
- For short-term reference: in the operational memory task record on Mac Mini (path reference, not the artifact itself unless it is small)
- For long-term reference: as part of the canonical memory task entry when promoted

### 15.4 Audit Capability

The human operator must be able to answer the following questions for any automation run within the past 30 days using Mission Control and operational memory alone:
- What ran?
- When did it run?
- What was its input?
- What was its output?
- Did it succeed or fail?
- What proof artifact does it reference?

If these questions cannot be answered from Mission Control, the automation's proof and audit trail is insufficient.

3. **Define the first three approved automation types** — Write D3.1 (research delegation), D3.2 (build delegation), and D3.3 (memory delegation) before activating any automation in those classes. Each delegation spec must include the maturity advancement track record requirement. (Source: §6.2)

5. **Establish the Mission Control automation register** — Create a register in Mission Control listing every approved automation: class, risk level, approval status, and current maturity level. This is the source of truth for what is authorized to run. (Source: §3 principle 9)

9. **Review and classify all legacy automations** — Before re-activating any n8n workflow or prior automation, classify it using the risk levels in §5 and the automation class in §4.
