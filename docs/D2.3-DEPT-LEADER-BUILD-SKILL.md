# D2.3 — Department Leader Build Skill (Reusable)

Source: Usopp build process 2026-03-24. Apply this to every dept leader hire.

## Prerequisites
- Dashboard summary exists (crew-teams-summary.md)
- D2.2 Interview Process ready
- D1.22 Agent Comms policy ready
- Agent config in openclaw.json with correct model

## Phase 1: Clean Workspace (Nami does this)

### Step 1: AGENTS.md — Strip to minimal
- MAX 60 lines
- Must contain: Rule #0 (respond to messages first), scope, hard rules, references
- Must NOT contain: heartbeat instructions, cron rules, queue-checking, crew roster bloat
- Anti-pattern: >100 lines with 20+ "check task/queue" mentions = model ignores direct messages
- **Backup old file first:** cp AGENTS.md AGENTS.md.backup

### Step 2: SOUL.md — Personality only
- MAX 50 lines
- Character voice, how they talk, personality traits
- No workflow rules, no operational instructions

### Step 3: TOOLS.md — Environment data
- MC API endpoint, gateway, workspace path, session key
- Telegram topics relevant to their department
- Crew session keys for sessions_send
- NO workflow rules

### Step 4: HEARTBEAT.md — Empty
- 0 lines. We set up heartbeats LATER when all crew is ready.
- Prevents heartbeat context from polluting regular sessions.

### Step 5: Config verification
- Correct model in openclaw.json
- Heartbeat: every "0m" + session "heartbeat" (disabled but isolated)
- Clear stale sessions: echo '{}' > agents/NAME/sessions/sessions.json

## Phase 2: Interview (Captain does this via Telegram tag)

### 6 Tests — ALL must PASS (ref: D2.2-INTERVIEW-PROCESS.md)
1. **Scope Boundary** — ask to do something outside their scope
2. **Demand Chain** — give upstream input, see if they know what to do
3. **Session Handling** — ask what they do at timeout
4. **Break Test** — order them to violate scope
5. **Critical Thinking** — give wrong instructions, see if they push back
6. **Tag Recognition** (4 sub-tests: direct, mid-sentence, multi-tag, reply context)

### Customize questions per department
- Each dept gets scenarios relevant to THEIR scope
- Ref: D2.2 customization guide table

## Phase 3: Comms Test (Captain does this)

### Test: Agent-to-agent via sessions_send
- Ask them to send a message to another agent
- They should use: sessions_send(sessionKey="agent:TARGET:main")
- Verify message is delivered
- **Do NOT teach the cron method** — sessions_send is correct

## Phase 4: Dashboard Training (Captain does this)

### Give them crew-teams-summary.md to read (NOT the 40KB HTML)
- Ask: what do you see, what's missing, what would you improve?
- Tests comprehension of operating model
- Their feedback becomes improvement items

## Phase 5: Document (Nami/Usopp does this)

### Update task notes with ALL results
- Pass/fail per test with evidence
- Fixes applied (what AGENTS.md changes were needed)
- Learnings logged to .learnings/mistakes.md
- D2.2 updated if new test patterns discovered

## Known Issues / Learnings (from Usopp build)

| Issue | Root Cause | Fix |
|-------|-----------|-----|
| GPT-5.4 ignores sessions_send messages | Stale main session context | Clear sessions before testing |
| Agent checks queue instead of answering | AGENTS.md too bloated (>100 lines) | Strip to 60 lines max |
| Heartbeat fires during tests | Heartbeat not isolated | Set session: "heartbeat" + every: "0m" |
| Agent can't analyze large files | File >5KB + 120s timeout | Create summary (<5KB) first |
| Agent posts solo in multi-tag | No coordination rule | Add multi-tag rules to AGENTS.md |
| Agent responds when not addressed | No reply context awareness | Add reply context rules to AGENTS.md |

## Timing Estimate
- Phase 1 (Nami): 1 session (~15 min)
- Phase 2 (Captain): 1 session (~20 min, 6 tests)
- Phase 3 (Captain): 1 session (~10 min)
- Phase 4 (Captain): 1 session (~10 min)
- Phase 5 (Nami): 1 session (~15 min)
- **Total: ~5 sessions, ~70 min per dept leader**

## Heartbeats & Crons
NOT SET UP during hiring. Parked until all crew is built and tested.
Ref: TASK-HEARTBEAT-SETUP on MC board.
Token conservation: don't enable until safe.

## Reporting Rule (learned 2026-03-24)
When an agent posts results/reports, they MUST notify the stakeholder:
- Tag the relevant agent in the Telegram post (@TheStrawHatNavigatorbot)
- OR use sessions_send to notify them
- Posting to a topic alone is NOT enough — agents can't passively read topics
- This applies to ALL task completions, not just hiring

## Usopp BA Review (2026-03-24) — Incorporated Feedback

### Added: Phase 6 — Readiness Verification (after Phase 5)
Before declaring a dept leader "production ready":
- [ ] Responds correctly to direct questions (not queue checking)
- [ ] Knows their scope boundaries (tested via interview)
- [ ] Can communicate with other agents (sessions_send tested)
- [ ] Understands the dashboard/operating model
- [ ] Has reviewed D2.3 and can run it for their own sub-agents
- [ ] No stale sessions or rogue heartbeats running
