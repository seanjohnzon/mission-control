# D5.5 — WORKFLOW MIGRATION PLAN
**Autonomous AI Organization — Migration Path from Prior Automation Stack to New Architecture**
_Last updated: 2026-03-13 | Status: Blocked (Desktop offline) | Authority: D1.1-ROADMAP.md §2, §9_

---

## 1. Purpose

This document defines how prior workflows, automations, and infrastructure are migrated into the new organizational architecture. Migration is deliberate — every prior asset is evaluated, classified, and either migrated, adapted, or retired with documented rationale.

---

## 2. Migration Principles

1. **Inventory before migration.** No workflow is migrated until it is inventoried in D5.3.
2. **Classify before acting.** Every asset gets a Keep/Adapt/Retire classification before any work begins.
3. **Migrate before rebuilding.** If a prior workflow works, migrate it. Don't rebuild from scratch without evidence that the prior approach is broken.
4. **Execution substrates are subordinate.** n8n, scripts, browser automation, and external workflow tools serve Mission Control — they don't replace it. (D1.1 Principle 10)
5. **One migration at a time.** Complete each migration before starting the next. Partial migrations create debt.

---

## 3. Migration Framework

### Phase 1: Inventory (D5.3)
- Survey all machines for prior workflows, scripts, configs, prompts
- File each in D5.3-LEGACY-ASSET-REGISTER.md with classification
- Human reviews and approves classifications

### Phase 2: Assessment
For each **Keep** or **Adapt** asset:
- Document current function (what does it do?)
- Document current dependencies (what does it need?)
- Identify target execution layer in new architecture
- Estimate migration effort (hours, complexity)
- Identify risks (data loss, broken dependencies, missing credentials)

### Phase 3: Migration Execution
For each approved migration:
1. Create a TASK entry in TASK-REGISTER
2. Create a CHG entry if the migration changes operational state
3. Execute migration in the target execution layer
4. Test in controlled context
5. Verify output matches or exceeds prior workflow
6. Human accepts or requests changes
7. Close task and update D5.3

### Phase 4: Retirement
For each **Retire** asset:
- Archive the asset (do not delete — archive for reference)
- Document retirement rationale in D5.3
- Remove from active execution paths

## 4. Execution Layer Mapping

When migrating workflows, the target execution layer depends on what the workflow does:

| Workflow Type | Target Layer | Notes |
|---|---|---|
| Scheduling / cron jobs | OpenClaw cron | Native capability |
| Message routing / notifications | OpenClaw broker (Telegram) | Native capability |
| Web research / data retrieval | Research Lane (forge) | May use Perplexity if CAP-006 activated |
| Commerce automation | brain (Merchant) + execution substrate | Phase 2+ |
| Content generation | forge (Builder) | Phase 3+ |
| Dashboard / monitoring | Mission Control visual layer | Phase 2+ |
| File management / archiving | brain (Archivist) | Phase 2+ |
| External API integrations | OpenClaw skills or execution substrate | Evaluated per integration |
