## A. Purpose

This Activation Pack is the single document that collects every prerequisite, credential, decision, and sequencing rule needed to execute Phase 1 activation from start to finish — without mid-session interruptions, without guessing, and without needing to hunt across 15 policy documents to find one answer.

**Why this document exists:** The Phase 1 runbooks (RB-01 through RB-05 in D2.0) assume that the human has already gathered everything they need. If you start RB-01 without completing this pack first, you will hit a missing credential, an unmade decision, or a dependency gap at the worst possible moment. This document is the pre-flight checklist that prevents that.

**What completing this pack means:** When you finish Section N (the final go/no-go checklist), you have confirmed that every required input is available, every required decision has been made, and OpenClaw has the information it needs to execute RB-01 through RB-05 in sequence without stopping to ask questions.

**Authoritative source:** This document is authoritative for Phase 1 activation prerequisites. If this document and a runbook (D2.0) conflict, resolve the conflict by updating whichever is wrong before proceeding.

## C. Execution Surfaces

Understanding which surface does what prevents confusion during activation.

### C.1 Cursor (Planning and Documentation Surface)

Cursor runs on the MacBook Air (and may be open on the Mac Mini as a secondary surface). During activation, Cursor is used for:
- Reading and referencing policy documents (D1.x series)
- Reading and updating this Activation Pack and runbooks (D2.x)
- Drafting commit messages and change records
- Consulting Claude via the Cursor IDE for reasoning, debugging, and policy interpretation

Cursor does **not** execute commands on the Mac Mini. Cursor does not run agent workflows. Cursor does not store secrets. Cursor is a **supervised/manual** engineering surface — it is not the swarm's autonomous runtime (OpenClaw is). In future phases, Cursor Claude chat may be operated indirectly through device-mediated control (Perplexity or OpenClaw navigating the UI) without requiring direct API integration — see D1.5 §15.5.

### C.2 OpenClaw on the Mac Mini (Execution Surface)

OpenClaw is the agent runtime. All Phase 1 execution happens through OpenClaw on the Mac Mini. OpenClaw is used for:
- Running the bridge agent
- Executing RB-01 through RB-05 step by step
- Testing Telegram, the local LLM, and the cron scheduler
- Writing task state updates to the Mission Control task register
- Posting to Telegram once SEC-002 is active

OpenClaw does **not** make policy decisions. OpenClaw does not approve new capabilities. OpenClaw does not proceed past a STOP marker without human confirmation.

### C.3 Claude (Primary Reasoning and Debug Lane)

Claude (via Cursor or OpenClaw's model routing) is the primary lane for:
- Interpreting ambiguous policy or runbook steps
- Diagnosing errors during activation
- Drafting and reviewing documents
- Answering questions during execution

Claude is not used to approve capabilities, make activation decisions, or bypass human confirmation requirements.

### C.4 Telegram (Communication Channel — Activated During Phase 1)

Telegram does not exist as a surface at the start of Phase 1. It is created and confirmed during Phase 1 (RB-02). Once active, it becomes the human operator's primary window into the running system.

### C.5 MacBook Air (Human Authoring Surface)

The MacBook Air is where the human reads and writes. It is not a worker node, not a server, and the organization does not depend on it being online. If you are completing this document on the MacBook Air, that is correct and expected.

## D. Access Minimization Rules

These rules govern how Phase 1 activation approaches access decisions. They are policy (D1.1 Principle 17, D1.5 §15) stated here as operational rules.

**Rule 1: Prefer the simplest stable path.** If OpenClaw has a built-in capability that satisfies a need, use it. If a stable API integration satisfies it without a device dependency, use that. Only use a device-mediated or computer-use approach as a last resort, and only if it is in an approved task packet.

**Rule 2: Do not activate capabilities you do not need right now.** Phase 1 requires CAP-001 (Anthropic inference) and CAP-002 (Telegram bot). It does not require CAP-006 (Perplexity), CAP-007 (device-mediated access), xAI, Discord, or any other capability. Do not obtain credentials for capabilities not needed in Phase 1.

**Rule 3: Use the keychain, not flat files.** Secrets go in the Mac Mini OS keychain. They do not go in `.env` files, in notes, in Cursor files, in git-tracked files, or anywhere in the Mission Control workspace.

**Rule 4: Claude before API before device.** For any task during activation:
1. Can Claude (built into OpenClaw) answer or resolve it? → Use Claude.
2. Is there an approved API (Anthropic, Telegram) that handles it? → Use the API.
3. Is there a device-mediated GUI automation approach available? → Only if the above two fail, and only if it is in an approved task packet.

**Rule 5: Do not expand scope mid-activation.** If you notice during activation that something "would be nice to set up," write it down for after Phase 1. Do not activate, configure, or test anything that is not required by D1.3 §14, D2.0 RB-01 through RB-05, or this Activation Pack.

## H. OpenClaw Startup Responsibilities

This section tells OpenClaw exactly what its responsibilities are at the start of activation and during the sequence. OpenClaw reads this section before acting.

### H.1 Before Accepting Any Task

1. **Confirm gateway is healthy.** OpenClaw gateway must be responding on port 18789. If it is not responding, stop and surface the error — do not proceed.

2. **Confirm Mission Control is accessible.** The path `~/Cursor Projects/mission-control/` must exist and be readable. If it is not, stop and surface the error.

3. **Read this document (D2.1) completely.** Do not start the activation sequence without reading all sections. If any prerequisite in Section E is `MISSING` and the human has not confirmed it is resolved, stop and surface the gap before proceeding.

4. **Do not create secrets.** OpenClaw does not generate API keys or bot tokens. It uses values the human has already placed in the keychain.

5. **Do not activate capabilities not listed in Section E.4.** If you encounter a step that suggests activating something not listed, stop and surface it to the human before proceeding.

### H.2 During Activation

1. **Record every step outcome.** For each runbook step you complete, write the outcome (success / failure / partial) to the task record in Mission Control. Do not rely on memory — write it down.

2. **Stop at STOP markers.** Every STOP marker in D2.0 requires human confirmation before proceeding. This is not optional. A STOP marker is not a suggestion.

3. **Surface errors immediately.** If a step fails, stop the sequence, describe what failed (step number, error message, current state), and wait for human direction. Do not attempt to work around a failure without human input.

4. **Do not invent alternate steps.** If a runbook step is unclear, surface the ambiguity via the active communication channel (Telegram once active, or direct to the human in session before Telegram is active). Do not substitute a different approach.

5. **Update SECRET-REGISTRY.md after each secret activation.** When a secret transitions from `registered` to `active`, update the `activation_status`, `approval_date`, and `last_rotated` fields in `registry/secrets/SECRET-REGISTRY.md` and commit the change.

### H.3 Proof of Completion

For each runbook completed (RB-01 through RB-05), OpenClaw must record:
- Runbook ID and name
- Date and time of completion
- Verification step outcome (what was tested and what the result was)
- Any deviations from the runbook (none expected — document if any occurred)
- Commit hash of the Mission Control state after completion

These records are the proof artifacts for Phase 1 exit gate (D1.3 §14).

## I. Bootstrap Mode Clarification

Bootstrap mode is the permitted operating state for early Phase 1 before the local LLM is running.

**What bootstrap mode means:** Bridge runs inference through the Anthropic API (SEC-001, CAP-001) instead of the local LLM. This is allowed for Phase 1 activation because the local LLM model has not yet been selected and validated.

**What bootstrap mode does not mean:** It does not mean Phase 1 is complete without a local LLM. It does not mean Anthropic cloud inference is the target steady state. It does not mean you can skip local LLM setup indefinitely.

**Bootstrap mode exits when:** The local LLM is selected, installed, running, and confirmed working by RB-04. After RB-04 succeeds, bridge routes inference to the local model per D1.5 model routing policy, and cloud inference becomes the fallback lane (not the primary lane).

**Policy references:**
- CHG-002 established the bootstrap mode allowance
- D1.5 §9 governs inference routing and the local-first principle
- D1.3 BRG-INF-01 establishes local inference as required before Phase 1 exit

**Risk during bootstrap mode:** Using cloud inference exclusively during early Phase 1 creates Anthropic API cost exposure and a cloud dependency. Keep the bootstrap period short. Select and install the local LLM before running any sustained automated workloads.

## K. Stop Conditions and Escalation Conditions

These are the conditions under which OpenClaw must stop the activation sequence and wait for human input. OpenClaw does not work around these — it stops, describes the situation, and waits.

| Condition | What OpenClaw Does |
|---|---|
| Any `MISSING` prerequisite is still missing at the step that requires it | Stop sequence at that step. Surface the missing item: what it is, which step needs it, how to provide it. Wait. |
| Any secret is not loadable from the keychain | Stop at Step 3 or any step that uses that secret. Describe the load failure. Human must re-store the secret. |
| Any API call (Anthropic or Telegram) fails more than once | Stop. Log the error. Surface to human with the exact error message. Do not retry more than once without human direction. |
| Any required decision is still `DECISION PENDING` at a step that needs it | Stop. Surface the decision required. Wait for human to make it. |
| A step produces unexpected output or an error not described in the runbook | Stop. Describe what happened vs. what was expected. Wait for human direction. |
| Any attempt to activate a capability not listed in Section E.4 | Stop. Surface the capability request and why it arose. Human decides whether to approve it through proper channels (D1.12). |
| MAC-MINI-BASELINE verification check fails | Stop. Identify which check failed and the likely cause. Surface to human. |
| Phase 1 exit gate (Step 9) reveals an unmet criterion | Stop. List unmet criteria. Do not declare Phase 1 complete. Human decides next steps. |

**What OpenClaw does while stopped:** It does not guess, retry with a different approach, or proceed past the stop. It waits with the situation described clearly. If Telegram is operational at the time of the stop, the description is posted to the ops-channel. If Telegram is not yet operational, it surfaces the issue in the current session.
