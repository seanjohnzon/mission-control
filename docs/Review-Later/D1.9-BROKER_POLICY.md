## 1. Purpose

This document defines the broker policy for the organization. It governs how messages are classified, routed, delivered, and logged across every channel the system uses: Telegram (human-facing, primary), internal task packets (agent-facing), and eventually LAN-based inter-agent communication.

**This document resolves:**
- BRG-OCL-07 (D1.3) — broker behavior and LAN auth policy deferred to D1.9 and D1.14
- The cross-agent communication protocol required by D1.8 §11 before Phase 2 agents are activated

**What this document governs:**
- What the Broker role inside bridge is responsible for
- How each message class is classified and routed
- What human-facing communication must and must not look like
- Escalation routing path and timing
- Telegram primary channel policy (§12)
- Discord optional channel policy (§11)
- LAN-based inter-agent routing design (Phase 2+)
- Message logging requirements

**What this document does not cover:**
- Discord bot setup steps — D2.1-BRIDGE-RUNBOOK.md
- Secrets and tokens for Discord, Telegram, or LAN endpoints — D1.14-SECRETS_POLICY.md

## 3. Policy Principles

1. **Bridge is the single broker.** All messages between the human and agents, and all messages between agents, route through bridge. No agent sends messages to the human directly. No agent sends task packets to another agent without bridge's knowledge and involvement.

2. **Every message has a class and a route.** Before any message is sent or dispatched, it is classified by type and assigned a route. Unclassified messages are not silently dropped — they are held and flagged.

3. **Human-facing output is clean.** The human should not receive raw logs, stack traces, model output dumps, or internal technical state. Every human-facing message is formatted to be clear, calm, and action-oriented. If nothing meaningful has happened, no message is sent.

4. **Silence is a valid state.** Agents do not generate messages to confirm they are alive, to report routine progress on long-running tasks, or to fill quiet periods. A message is sent when there is something for the human to know or do — not to demonstrate activity.

5. **Directives always receive acknowledgement.** Every human directive delivered via Telegram receives an acknowledgement from bridge within the defined timeout. Directives that are not acknowledged are considered undelivered.

6. **Escalations go to the human — not to other agents.** When an automation trigger fires or an agent cannot continue, the escalation routes to the human via Telegram. Agents do not escalate to each other. There is no agent-to-agent override or escalation bypass.

7. **No peer-to-peer side channels.** Agents do not communicate with each other outside of task packets routed through bridge. No shared file that one agent writes and another reads without a declared handoff. No LAN socket connections between brain and forge that bypass bridge.

8. **All routing is logged.** Every message routed by the broker is logged with timestamp, class, source, destination, task ID (where applicable), and delivery status. Message logs live in operational memory subject to D1.7 retention rules.

9. **LAN routing is future-facing.** Phase 1 has no LAN-connected agents. Bridge does not attempt LAN routing until Desktop and GPU3060 are online, surveyed, and authorized. Designing for LAN now does not mean activating it now.

10. **One primary human channel.** The human receives operational communication through one primary Telegram channel. Multiple overlapping channels for the same message class create noise and fragment attention.

## 4. Broker Responsibilities

The Broker role lives inside the bridge agent on Mac Mini. It is one of two logical roles bridge carries (Broker and Conductor). Conductor manages task dispatch and cross-agent orchestration. Broker manages the communication layer.

**Broker is responsible for:**

1. **Inbound classification** — receiving all inbound messages from the human via Telegram (primary), classifying each by message class (§5), and routing to the appropriate internal handler or Conductor. Also receives internally-initiated messages from Conductor (self-managed task pickups, backlog notices, workflow candidate alerts) and routes them appropriately — not all routed messages originate from the human.
2. **Outbound formatting** — receiving results, alerts, and updates from Conductor and agents, formatting them for human consumption, and delivering to the appropriate channel
3. **Acknowledgement** — posting an acknowledgement for every inbound directive before processing begins
4. **Escalation routing** — converting escalation trigger events (D1.8 §8) into human-facing alerts with the correct urgency, format, and channel
5. **Digest assembly** — assembling scheduled digests from operational memory entries and posting them to the human channel on schedule
6. **Message logging** — writing a log entry for every message sent or received through any channel
7. **Degraded mode management** — detecting channel failures and entering defined degraded behavior (§15)

**Broker is NOT responsible for:**
- Deciding what work to do — that is Conductor's domain
- Writing to canonical memory directly — that is Archivist's domain
- Approving or rejecting automation risk — that is the human's domain enforced by policy
- Making routing decisions about whether a task should go to brain or forge — Conductor decides; Broker delivers

**Note on Perplexity outputs:** When agents invoke Perplexity (CAP-006, when activated) during task execution, the research outputs and retrieved content are task results — not channel messages. They travel through the normal task result → Conductor → Broker → formatted completion notice pipeline. Perplexity does not post to human-facing channels directly. Perplexity outputs are formatted by Broker before human delivery, filed to Mission Control with a proof reference, and subject to D1.10 §8 validation before any action is taken. (Source: D1.4 §11 Perplexity callable capability note)

## 5. Message Classes

Every message handled by the broker is classified as one of the following types. Classification determines routing, formatting, and logging behavior.

| Class | Direction | Description |
|---|---|---|
| `directive` | Human → Bridge | A human instruction that requires agent action. May be a task dispatch, a policy update instruction, or a system command. |
| `query` | Human → Bridge | A human request for information or status. Requires a response but does not dispatch a task to an agent. |
| `feedback` | Human → Bridge | A human response to a prior agent output: approval, rejection, correction, or modification instruction. Linked to a prior task record. |
| `acknowledgement` | Bridge → Human | Confirmation that a directive or escalation has been received and is being acted on. Not a result — purely a receipt. |
| `progress` | Bridge → Human | An in-progress status update for a long-running task. Sent only at defined intervals for tasks that declare them. Never sent for short tasks. |
| `completion` | Bridge → Human | Notification that a task has closed. Includes the result summary and proof artifact reference. |
| `alert` | Bridge → Human | A system-generated warning or failure notice. Does not require human action unless it is an escalation. |
| `escalation` | Bridge → Human | A halt-and-wait message. Automation has stopped and requires human input before it can continue. Requires explicit acknowledgement. |
| `digest` | Bridge → Human | A scheduled summary of a defined period's events. Sent on a declared schedule. Never duplicates individual completion notices. |
| `handoff` | Bridge ↔ Agent (internal) | An internal task packet moving work from bridge to an agent, or a result packet returning from an agent to bridge. Not human-facing. |
| `memory-commit` | Agent → Bridge → Archivist (internal) | A promotion packet carrying a draft canonical memory entry. Handled by the memory promotion pipeline (D1.7 §9). Not human-facing. |

## 6. Routing Rules by Message Class

| Class | Source | First Stop | Routing Logic | Final Destination |
|---|---|---|---|---|
| `directive` | Human (Telegram) | Broker | Classify → acknowledge → pass to Conductor | Conductor → task dispatch |
| `query` | Human (Telegram) | Broker | Read current state from operational memory or MC → format response | Human (Telegram) |
| `feedback` | Human (Telegram) | Broker | Link to task record → update status in operational memory → notify Conductor if task is in-flight | Task record + Conductor if needed |
| `acknowledgement` | Broker | Telegram | Immediate — no routing delay | Human (Telegram) |
| `progress` | Conductor | Broker | Format → post only if task has declared progress intervals | Human (Telegram) |
| `completion` | Conductor | Broker | Format → post to ops channel → update task record in MC | Human (Telegram) + MC |
| `alert` | Bridge (any role) | Broker | Classify severity → format → post to appropriate channel | Human (Telegram) |
| `escalation` | Bridge (automation trigger) | Broker | Format with trigger ID → post with urgency marker → start acknowledgement timer | Human (Telegram) — awaits response |
| `digest` | Cron (bridge) | Broker | Assemble from operational memory → format → post on schedule | Human (Telegram) |
| `handoff` | Bridge / Agent | Conductor | Task packet validation → dispatch to target agent → await result | Target agent (internal only) |
| `memory-commit` | Agent / Bridge | Archivist (via Conductor) | Promotion pipeline (D1.7 §9) | Canonical memory on Desktop |

## 7. Human-Facing Channel Policy

### 7.1 One Primary Channel

The human receives all operational communication in one primary Telegram channel (group or private chat). This channel carries: completions, alerts, non-critical notices, and digests. It does not carry: raw logs, task packets, internal state dumps, or heartbeat confirmations.

Channel identity and bot token are governed by D1.14-SECRETS_POLICY.md (SEC-002). The logical label here is `ops-channel`.

### 7.2 Escalation Delivery

Escalations are delivered in the ops-channel with a distinct visual marker (e.g. a prefix or label that makes them immediately identifiable). If Telegram supports reply threads, escalations may be threaded to allow focused discussion. The escalation remains visible in the main channel regardless of whether threading is used.

The human does not receive escalations through a separate channel by default. A separate escalation channel is a Phase 2+ option if the ops-channel volume warrants it.

### 7.3 Message Formatting Rules

Every human-facing message from bridge must follow these rules:

1. **Lead with what happened or what is needed.** Not what the agent was doing, not which model processed the task, not internal state.
2. **Maximum one outbound message per event.** If the result content is longer than one message (~1800 characters), the full content goes to Mission Control or an output file. The outbound message contains a summary + reference path.
3. **No stack traces or raw error objects.** If a failure occurred, the message says what failed, the impact, and what the human must do — not the raw exception.
4. **No model names or inference details in routine messages.** Which model processed the task is not human-facing information unless the human specifically queried for it.
5. **Tense: past for completions, present for alerts, imperative for escalations.** "Task completed." / "Memory write is failing." / "Approve or reject the following before the automation can continue."
6. **Escalations include the required human action.** Not just "something went wrong" — the escalation message states exactly what decision the human needs to make.

### 7.4 Message Volume Control

Bridge does not produce more than N messages per hour in the ops-channel during normal operation. This limit is TBD (see §16) and is set based on what volume the human finds useful vs noisy. If an operational period would generate more messages than the limit, non-critical messages are batched into the next scheduled digest rather than sent individually.

Exceptions to the volume limit: escalations and critical alerts are always sent immediately regardless of volume.

### 7.5 What Bridge Never Posts to Human-Facing Channels

- Heartbeat confirmations ("Mac Mini is online")
- Intermediate reasoning steps or draft outputs
- Inference call results before they have been processed into a task result
- Raw memory entries or file contents
- Agent-to-agent handoff packet contents
- Any content that was already delivered in a prior message for the same event

## 8. Agent-Facing Routing Policy

### 8.1 Task Packet Flow

Task packets originate from two sources: a human directive, or a Conductor self-managed pickup of an eligible backlog task (D1.8 §6.5, D1.4 §9). Both produce identical packet structures and follow the same routing chain.

**Human-directed flow:**
```
Human directive (Telegram)
    → Broker (classify + acknowledge)
    → Conductor (task packet creation)
    → Agent (brain or forge)
    → Agent returns result to Conductor
    → Conductor updates task record
    → Broker formats + delivers completion to human (Telegram)
```

**Self-managed flow (Conductor-initiated):**
```
Conductor scans backlog → identifies eligible task
    → Conductor creates or validates task packet
    → Agent (brain or forge)
    → Agent returns result to Conductor
    → Conductor updates task record
    → Broker formats + delivers completion notice to human
```

In the self-managed flow, Broker does not post an acknowledgement (there is no inbound directive to acknowledge). Broker does post the completion notice. The completion notice must indicate the task was agent-initiated, not human-directed, so the human can distinguish self-managed work from assigned work. (Source: D1.3 MC-TSK-05)

No step in either flow is optional. An agent that receives a task directly from another agent without bridge in the chain is not operating under policy.

### 8.2 Agent Response Protocol

When an agent completes a task, it returns a result packet to bridge (Conductor). The result packet must include:
- The originating task ID
- Status: `success` | `failure` | `partial`
- Output reference: path or content
- Proof artifact reference
- Any child task IDs generated during execution

Bridge does not accept a result packet that is missing any of these fields. An incomplete result packet is logged as a protocol error and the task remains `active` pending a corrected result.

### 8.3 No Direct Agent-to-Human Messages

Agents (brain and forge) do not post to Telegram or any human-facing channel. They do not hold communication channel credentials. All human-facing output is formatted and posted by bridge (Broker). This ensures human communication is consistently formatted, consistently routed, and consistently logged.

The exception is bridge itself: bridge posts under its own Telegram bot identity (SEC-002) as the agent-facing face of the system.

## 9. Escalation Routing Policy

### 9.1 Escalation Source

Escalations originate from two sources:
1. **Automation triggers** (D1.8 §8) — conditions that halt an active automation
2. **Agent-generated escalations** — an agent determines it cannot proceed within its declared scope and signals bridge to escalate

In both cases, the escalation travels to bridge (Conductor first, then Broker). Broker formats and delivers it to the human.

### 9.2 Escalation Message Format

Every escalation message must contain:
- **Trigger ID** (from D1.8 §8) or `agent-escalation` for agent-generated halts
- **What was running** — task ID and task type
- **What happened** — one sentence description of the condition
- **Current state** — what has been halted, what is still in flight
- **Required human action** — exactly what the human must do to unblock: approve, reject, provide information, or close the task
- **Acknowledgement instruction** — how the human signals they have received and acted on the escalation

### 9.3 Escalation Is Not a Notification

A notification informs. An escalation halts and waits. Bridge treats these differently:
- Notifications do not require acknowledgement
- Escalations require explicit acknowledgement before the halted automation may continue
- Bridge tracks escalation state: `pending` (sent, awaiting ack) → `acknowledged` (human responded) → `resolved` (automation resumed or task closed)

### 9.4 Escalation Acknowledgement

The human acknowledges an escalation by responding in Telegram. Bridge (Broker) detects the response and transitions the escalation from `pending` to `acknowledged`. The response must address the required human action — a generic "ok" without a decision is not a sufficient acknowledgement for escalations that require a decision.

Acknowledgement format is defined in the escalation message itself. Bridge does not make the human guess how to respond.

### 9.5 Re-Send Policy

If an escalation has been in `pending` state for longer than the defined timeout (§10.3), bridge re-sends the escalation message. It does not send a new escalation — it resends the original with a visible re-send marker. Re-send continues at the defined interval until:
- The human acknowledges, or
- The maximum re-send count is reached, at which point the escalation is logged as `unresolved` and the affected automation remains halted indefinitely

Maximum re-send count and interval are defined in §10.3.

## 10. Timeout and Acknowledgement Policy

### 10.1 Directive Acknowledgement Timeout

| Event | Timeout | Action if Exceeded |
|---|---|---|
| Human directive received | Bridge acknowledges within 30 seconds | If bridge cannot acknowledge within 30 seconds, an error is logged. Human may assume the directive was not received and re-send. |
| Query received | Bridge responds within 60 seconds | If query requires checking a large operational memory state, bridge sends an interim "looking up" message within 30 seconds and the full response within 60 seconds total. |

### 10.2 Agent Task Completion Timeout

| Handoff | Default Timeout | Action if Exceeded |
|---|---|---|
| Bridge → Agent (task dispatched) | TBD per task type in D3.x | Bridge marks task BLOCKED; escalates to human if task is on the critical path |
| Agent → Bridge (result expected) | TBD per task type in D3.x | Bridge flags task as potentially stalled; posts progress alert to human |

Default timeouts are defined per task type in D3.x delegation specs, not in this document. Until D3.x specs exist, timeouts are set at Phase 1 entry and recorded in Mission Control's automation register.

### 10.3 Escalation Re-Send

| Escalation Re-Send | Value |
|---|---|
| Time before first re-send | 30 minutes |
| Re-send interval | 30 minutes |
| Maximum re-sends before marking `unresolved` | 3 re-sends (total window: ~2 hours) |
| Action when `unresolved` | Log as unresolved; automation remains halted; human must manually review MC to find and act on the pending escalation |

### 10.4 Digest Timing

Digests are sent on a declared schedule registered in Mission Control. If a digest run produces no meaningful content (no tasks completed, no alerts, no notable events), the digest is suppressed rather than sent as an empty notice. The suppression is logged but not reported to the human.

## 11. Discord Policy (Optional — Activation Decision Required)

### 11.1 Activation Status at Phase 1

Discord is an **optional** channel for this organization. It is not automatically activated at Phase 1. Before any Discord credentials are obtained, registered, or configured, the human must make an explicit activation decision (TASK-010, D1.14 §16 Priority 5): activate Discord in Phase 1, or defer to Phase 2 or later. Obtaining a Discord bot token before this decision is made is a policy violation (D1.14 §16 Priority 5).

The activation decision must be made after Telegram is confirmed operational as the primary channel (TASK-004 complete, SEC-002 active). TASK-010 is a P1 task sequenced after Telegram is confirmed.

### 11.2 If Activated: Channel Structure

If the human activates Discord (TASK-010 outcome: activate), the minimum required Discord setup is:

| Channel | Purpose | Who Posts |
|---|---|---|
| `#ops` (or equivalent) | Operational messages as a secondary supplement to Telegram | Bridge only |

Discord is a secondary supplement to the Telegram ops-channel. It does not replace Telegram as the primary channel. A single ops channel is sufficient if Discord is activated. Additional Discord channels are added only if message volume or topic separation genuinely requires it.

### 11.3 Bridge Bot Identity (If Activated)

If Discord is activated, bridge posts to Discord under a dedicated bot identity. The bot name, avatar, and token are managed per D1.14-SECRETS_POLICY.md. The bot does not impersonate the human. The bot does not post under a human account.

### 11.4 Message Retention

Discord message history is not the source of truth for task state or organizational memory. Task state lives in Mission Control. Completed task records live in canonical memory. Discord messages are reference-only and may be lost or pruned without data loss to the organization. This holds whether or not Discord is activated.

### 11.5 What Discord Is Not

- Not the task register — that lives in Mission Control
- Not the memory store — that lives in D1.7 tiers
- Not the approval record — approvals are logged in Mission Control
- Not a debugging console — no technical dumps or raw states
- Not a persistent log — do not design any system to depend on Discord history for state reconstruction
- Not the primary channel — Telegram is the Phase 1 primary communication channel (§12)

### 11.6 Rate Limiting and Burst Control (If Activated)

If Discord is activated, bridge must not post more than Discord's API rate limits allow. If a burst event would exceed rate limits, bridge queues outbound messages and sends them with a short delay between each.

## 12. Telegram Channel Policy (Primary)

### 12.1 Status at Phase 1

Telegram is the **primary operational communication channel** for Phase 1. The Telegram bot (SEC-002, CAP-002) is the required Phase 1 human-agent communication interface. Bridge posts all human-facing output — directive acknowledgements, completions, alerts, escalations, and digests — to the Telegram ops-channel.

Telegram bot setup is a Phase 1 activation prerequisite (TASK-004, D2.0 RB-02). Phase 1 is not operational without Telegram active. This is established in D1.11 §3 Principle 1, D1.14 §14, and the Phase 1 exit gate (D1.3 §14).

### 12.2 Scope at Phase 1

The Telegram ops-channel carries all human-facing message classes from §5:
- Inbound human directives to bridge
- Acknowledgements for all directives
- Completion notices for completed tasks
- Progress updates for long-running tasks (where declared)
- Alerts (non-critical and critical)
- Escalations (halt-and-wait, requiring human response)
- Scheduled digests

Telegram is the full operational channel at Phase 1 — not a mobile urgency layer only. All message classes from §6 that route to the human route through Telegram.

### 12.3 Bot Configuration

The Telegram bot token is registered as SEC-002 in D1.14-SECRETS_POLICY.md and stored in the Mac Mini keychain. The bot identity is bridge's communication face. Bridge is the only entity that posts to the ops-channel. The human operator sends directives, reviews results, and acknowledges escalations through the Telegram ops-channel.

### 12.4 Relationship to Discord

Discord is an optional secondary channel requiring an explicit activation decision (TASK-010, D1.14 §16 Priority 5). Telegram does not depend on Discord being activated. Phase 1 is fully operational with Telegram only. If Discord is activated (TASK-010 outcome), it supplements Telegram as a secondary surface — it does not replace Telegram as the primary channel.

### 12.5 Other Channels

No other channel (email, SMS, Slack, Signal, etc.) is considered at any phase without this document being updated to define its scope and activation gate. Adding communication channels without policy increases noise and operational complexity without justification.

## 13. LAN / Cross-Agent Broker Policy

### 13.1 Phase 1: No LAN Cross-Agent Routing

At Phase 1, Desktop and GPU3060 are not reachable. Bridge does not attempt to route task packets to brain or forge over the network. Any reference to LAN routing in this section is design policy for Phase 2+.

### 13.2 LAN Routing Architecture

When Desktop and GPU3060 are online, bridge routes task packets to brain and forge over the local area network. The architecture:

```
Bridge (Mac Mini)
    │
    ├── Brain (Desktop) — task packets over LAN
    └── Forge (GPU3060) — task packets over LAN

Brain and Forge do not communicate directly.
All LAN traffic passes through or is initiated by Bridge.
```

### 13.3 LAN Message Format

Task packets sent over LAN use a structured format (JSON or structured markdown) that carries all required task packet fields (D1.8 §2). The specific serialization format is defined in the first D3.x delegation spec that covers cross-agent tasks. Until that spec is written, no LAN routing is activated.

### 13.4 LAN Authentication

Authentication for LAN inference endpoints and agent-to-agent connections is defined in D1.14-SECRETS_POLICY.md. No LAN connection between agents is opened without a registered credential or authentication mechanism.

Permitted LAN connection models:
- Shared secret / API key (acceptable for Phase 2 internal-only LAN)
- Mutual TLS (preferred for Phase 3+ when forge serves inference over LAN)

Unauthenticated LAN connections between agents are not permitted.

### 13.5 LAN Failure Behavior

If bridge cannot reach brain or forge over LAN:
- Tasks addressed to the unreachable agent are queued by bridge (not rerouted to another agent)
- Bridge posts an alert to the ops-channel (Telegram): which agent is unreachable, which tasks are queued
- Bridge retries the LAN connection at a defined interval before declaring the agent offline
- Retry count and interval are defined in D4.2-RECOVERY-PLAYBOOK.md (TBD)
- Tasks that have been queued longer than the task-type timeout are escalated to the human

### 13.6 No Peer-to-Peer Exceptions

Brain and forge do not establish direct connections to each other under any circumstances, including:
- Memory reads (Brain reading Forge working memory is done by Brain over LAN to Forge's file system — this is a read, not a message routing exception, and is still logged by bridge)
- Result forwarding (Forge does not forward results to Brain directly; it returns results to Bridge, which routes them)
- Inference requests (Forge does not solicit inference from Brain's model stack)

All message routing exceptions must be documented in this policy before they are implemented.

## 14. Message Logging and Traceability

### 14.1 Log Requirement

Every message routed through the broker — inbound or outbound, human-facing or agent-facing — must generate a log entry. Silent routing is not permitted.

### 14.2 Log Entry Fields

| Field | Required | Description |
|---|---|---|
| `timestamp` | Yes | UTC datetime of routing event |
| `class` | Yes | Message class from §5 |
| `source` | Yes | Originating entity: `human`, `bridge`, `brain`, `forge`, `cron` |
| `destination` | Yes | Target entity or channel |
| `task_id` | If applicable | The task ID this message is associated with (null for queries with no task) |
| `delivery_status` | Yes | `delivered`, `queued`, `failed`, `suppressed` |
| `channel` | Yes | `telegram-ops`, `discord-ops` (if activated), `lan-brain`, `lan-forge`, `internal` |

### 14.3 Log Location

Message logs live in operational memory on Mac Mini (D1.7 §6). They are not a separate logging system. Retention follows D1.7 §10.1 rules for the entry types they map to.

Cross-agent handoff logs include the full task lineage chain (parent task ID → child task ID → result task ID). This chain is what makes any task traceable from human directive to final output.

### 14.4 Undelivered Message Handling

If a message fails to deliver (Telegram API error, Discord API error if activated, agent unreachable, channel unavailable):
- The log entry records `delivery_status: failed` with the error reason
- The message is placed in a retry queue for appropriate message classes (completions, escalations)
- Heartbeat-type messages that fail are not retried — they are logged and dropped
- If a queued message has not been delivered within 1 hour, an alert is posted via any available channel and the message is logged as undelivered

## 15. Broker Failure / Degraded Mode Policy

### 15.1 Primary Channel (Telegram) Unreachable

**Detection:** Broker cannot post to or receive from Telegram.
**Behavior:**
- Bridge enters DEGRADED state and logs the condition
- In-flight agent tasks may continue to completion — their results are held in the outbound message queue
- No new task dispatches from bridge until Telegram is confirmed reachable or the human acts via an alternate path
- When Telegram reconnects: queued outbound messages are sent in chronological order with a note indicating delayed delivery
- A recovery notice is posted first: "Bridge was in DEGRADED state from [time] to [time]. The following messages were queued during that period:"

If Discord is activated (TASK-010) and Telegram is unreachable, bridge may post a degraded-state alert to Discord as a fallback notification. This is notification only — no task dispatches resume until Telegram is restored or the human issues an explicit directive via Mission Control.

### 15.2 Bridge Process Crash

**Detection:** OpenClaw restarts bridge via launchd; broker state is lost from in-memory queue.
**Behavior:**
- On restart, Broker reads operational memory to reconstruct pending outbound messages
- Messages that were `queued` at crash time and have not been delivered are re-queued
- Broker posts a recovery notice to Telegram when the first successful post is confirmed
- Conductor simultaneously applies the reboot recovery procedure from D1.7 §6.4 (interrupted task detection)

### 15.3 Unclassified Inbound Message

**Detection:** An inbound message from the human does not match any class in §5.
**Behavior:**
- Broker holds the message for up to 5 minutes while attempting classification with a second pass
- If still unclassified after 5 minutes: Broker posts to ops-channel: "Received a message I could not classify. Please rephrase or clarify."
- The original message is logged with `class: unclassified`
- No action is taken on the content of the unclassified message

### 15.4 Volume Surge

**Detection:** Outbound message queue grows beyond a defined depth (TBD — see §16) faster than messages can be sent within Discord rate limits.
**Behavior:**
- Non-critical messages (progress updates, minor alerts) are batched into the next digest
- Escalations and critical alerts are always sent first regardless of queue depth
- Bridge logs a volume surge event in operational memory
- If the surge persists, Broker posts a single batching notice: "High activity period — non-critical updates will be included in the next digest."
