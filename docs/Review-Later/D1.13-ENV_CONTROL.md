# D1.13 — ENV_CONTROL
**Autonomous AI Organization — Environment Control, Configuration Governance, and Drift Prevention**
_Last updated: 2026-03-11 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document defines how environments are controlled across the organization. An environment is the totality of configuration, state, services, and constraints on a machine or within a context that determines how work is executed there.

Without deliberate environment control, the swarm accumulates invisible assumptions. Agents write one-off scripts that become de facto infrastructure. Config changes made during debugging persist silently. Credentials leak across boundaries. A method that worked yesterday fails today because something changed without a record. None of this shows up as an error until it does — loudly, in production.

Environment control exists to make the swarm's execution consistent, reproducible, and auditable. A workflow that works in production today should work in production next month — because the environment is defined, not improvised.

**This document governs:**
- What environment types exist and what each permits
- Which machines participate in which environments
- What is authoritative vs. local-only state
- How configuration is layered and constrained
- What must be version-controlled
- How changes move from experiment to production
- What counts as a healthy machine baseline
- What agents may change autonomously and what they may not
- How reusable methods and workflows become environment assets
- How drift is detected and corrected

**What this document does not cover:**
- Specific secrets, tokens, and credentials — D1.14-SECRETS_POLICY.md
- How to set up or operate machines — D2.x runbooks
- Capability approval and registration — D1.12-CAPABILITY_ACQUISITION_POLICY.md
- Workflow definition and promotion — D5.6-WORKFLOW_LIBRARY.md
- Agent task routing and message formats — D1.9-BROKER_POLICY.md

## 3. Environment Control Principles

1. **Consistency beats convenience.** An approach that works once is not worth adopting unless it can be made repeatable. Shortcuts that bypass defined config layers are not conveniences — they are undeclared dependencies.

2. **Production is boring on purpose.** Production environments change slowly, deliberately, and with proof. Novelty belongs in experimental environments. Production's job is to be reliable, not interesting.

3. **Experiment first, promote later.** No new capability, config change, or method goes directly from idea to production. It passes through an experimental or sandbox environment and earns promotion through demonstrated stability. The risk of getting this wrong is not worth the time saved by skipping it.

4. **One authoritative source per state type.** For any given kind of state — policies, canonical memory, operational task state, capability approvals — exactly one location is authoritative. Local copies and caches are derivatives. When there is a conflict, the authoritative source wins.

5. **Reusable methods are environment assets.** A method, prompt template, workflow, or build pattern that has been used successfully is an organizational asset. It belongs in a registered location (D5.6 or D5.7), not only in the memory of the agent that created it. Environment control is what makes method reuse work reliably across runs, machines, and time.

6. **No silent drift.** A change to the environment that is not recorded did not happen in an auditable sense. If an agent updates a config value, installs a dependency, or enables a service, that change is logged. If it is not loggable, it is not within the agent's authority to make.

7. **Self-managed does not mean self-redefining.** Agents pull and execute approved work without per-task human prompting. They do not use that autonomy to redefine their own environment boundaries, expand their own access scope, or declare a new environment type. Environment changes are proposed, approved, and applied through the change process. (Source: D1.1 §3 principle 15, D1.8 §3)

8. **Recovery must be possible from documentation alone.** Any environment that cannot be rebuilt from version-controlled documentation and approved config is fragile. If the only record of how something was configured is the running machine, the organization has an unacceptable dependency on that machine's continuous health.

9. **Config debt is technical debt.** Undocumented configuration, unregistered capabilities, and ad-hoc overrides accumulate silently. Each one is a future failure point. Environment control is how the organization avoids this class of debt.

## 4. Environment Types

The organization operates four environment types. Each has defined permitted and forbidden activities. All work happens in one of these types — there is no undefined fifth environment.

**Note on external managed services:** Third-party external services (e.g. Perplexity, Telegram, Anthropic API) are not environment types — they are callable capabilities governed by D1.12 and registered in D5.7. These services run in the provider's environment; the organization's environment boundary applies to how credentials are managed (D1.14) and how outputs are handled within the org's systems. Invoking an external service from within a production task does not change the environment type of that task. The capability must be approved and registered before production use; this is a capability governance requirement, not an environment control question.

### 4.1 Production

**What it is:** The live operating environment where the swarm runs real tasks, manages real state, and produces real outputs. The organization's canonical record, operational task queue, and active agent services run here.

**Permitted:**
- Approved, registered capabilities executed within their declared scope
- Tasks at or above Autonomy Level 2 (per D1.8 §4)
- Changes applied through the promotion path (§9 below)
- Operational registry updates within bridge's declared write authority (D1.12 §8.1)

**Forbidden:**
- Experimental or untested configuration
- Test credentials or placeholder secrets
- Any change not traceable to an approved workflow, capability, or human directive
- Changes that have not passed through a sandbox or experimental review
- Silent dependency installations

**Governing documents:** D1.8, D1.9, D1.12, D1.14, and all D1.x policy docs

### 4.2 Sandbox

**What it is:** A controlled environment that closely mirrors production but is isolated from live state and real credentials. Used for validating changes before production promotion. Inputs and outputs are bounded — sandbox results do not affect real external systems.

**Permitted:**
- Validating promoted candidates from experimental (see §9)
- Integration testing of new capabilities or config changes
- Rehearsal of workflows before production activation
- Model evaluation with production-equivalent (but not real) workloads

**Forbidden:**
- Production secrets or real credentials
- Connection to live external services (read-only access may be approved with explicit scope)
- Permanent state — sandbox state is expected to be cleared between sessions

**Governing documents:** D1.8, D1.12, D5.4

### 4.3 Dev

**What it is:** A local environment for writing, editing, and testing new code, documents, agent config, or workflow definitions. Dev work is authored by the human or by agents operating under explicit dev-context task scope. No real agent tasks run here by default.

**Permitted:**
- Document authoring and editing
- Code writing, linting, and unit testing
- Agent config drafting before sandbox validation
- Prompt template drafting before registration as CAP-WFM

**Forbidden:**
- Real agent task execution
- Connection to production registries or canonical memory
- Production credentials

**Governing documents:** D1.15-CHANGE_CONTROL.md (to be written), D1.8

### 4.4 Experimental

**What it is:** An unconstrained workspace for trying new capabilities, models, methods, and approaches. Outputs are ephemeral until adoption is decided. Failure is expected and acceptable. Experimental work runs on GPU3060 by default. It does not affect production state.

**Permitted:**
- Any new capability, model, or method that has not yet been evaluated
- Experiments registered in D5.4
- Novel approaches to known tasks, with results compared against existing methods
- Agent behavior under unusual inputs (stress, edge cases, adversarial prompts)

**Forbidden:**
- Real credentials or production secrets
- Adoption into production without passing through the promotion path (§9)
- Running experimental code directly on production machines

**Governing documents:** D5.4-EXPERIMENT-REGISTRY.md, D1.12 §6

## 6. Authoritative Sources of State

For every class of state, exactly one location is authoritative. Local copies are derivatives. In any conflict, the authoritative source wins.

| State Class | Authoritative Location | Local Cache Policy |
|---|---|---|
| Org policies (D1.x) | Mission Control: `docs/` on Mac Mini | No local cache — always read from source |
| Mission Control host | Mac Mini: `~/Cursor Projects/mission-control/` | MacBook Air has working copy; Mac Mini copy is authoritative |
| Canonical memory | Desktop (brain) — path TBD in Phase 2 | Agents may cache read results in working memory; cannot write to canonical memory directly |
| Operational memory (task state) | Mac Mini (bridge) — `~/.openclaw/` and Mission Control task register | Other agents read via bridge; no direct writes |
| Working memory | GPU3060 (forge) — local to forge | Local only; ephemeral; not promoted to canonical without explicit submission to Archivist |
| Capability register (D5.7) | Mission Control: `registry/D5.7-CAPABILITY-REGISTER.md` | Bridge may cache for read; status field updates must write back to source |
| Workflow library (D5.6) | Mission Control: `docs/D5.6-WORKFLOW_LIBRARY.md` | Agents read from source; no local forks |
| Delegation status | Mission Control: autonomy register (D1.8 §9.1) | Bridge reads; human sets; no agent promotes its own delegation level |
| Secrets and credentials | D1.14 — registered secret store (path TBD) | Loaded into process memory at runtime; never written to task records, logs, or registries |
| Agent config | Per-machine OpenClaw workspace + Mission Control config reference | Local override only within declared agent-local layer (see §7) |

**Local cache rule:** A local cache is always a read-through derivative of the authoritative source. Agents may cache for performance but must validate against source on divergence. A cache that becomes the de facto source of record — because the authoritative source was not updated — is a silent drift condition.

## 7. Configuration Layering Policy

Configuration is layered. Higher layers constrain lower layers. Lower layers may customize within the space allowed by the layer above — they may never override or violate a higher-layer constraint.

```
Layer 1 — Org-wide policy
    Defined in D1.x documents and Mission Control registries.
    Governs all machines, all agents, all tasks.
    Human sets. Agents read. Agents may not redefine.

Layer 2 — Machine-local config
    Settings specific to a single machine: LAN IP, model path, service startup list,
    local inference parameters, resource caps.
    Set during machine setup (D2.x runbooks). Recorded in Mission Control.
    Bridge may update within approved scope; changes logged.

Layer 3 — Agent-local config
    Settings specific to one agent's runtime: workspace path, model routing preference,
    task class defaults, tool access list.
    Defined in agent's OpenClaw workspace config.
    Must not exceed what Layer 1 and Layer 2 permit.
    Agent may tune within approved parameters; changes logged.

Layer 4 — Task-local overrides
    Per-task adjustments: specific model override, timeout extension, resource reservation.
    Applied at task packet level. Must be declared in the task record.
    May not violate Layer 1 policy (risk class, financial controls, escalation triggers).
    Ephemeral — do not persist beyond task completion.
```

**Rules for all layers:**
- Higher layers constrain lower. A task-local override cannot circumvent an org-wide policy.
- Agents may not invent undeclared config layers. There are four. Adding a fifth requires a change to this document, approved by the human.
- A config value with no declared layer assignment belongs at Layer 1 until the human classifies it.
- Layer conflicts are resolved upward: if Layer 4 and Layer 1 disagree, Layer 1 wins and bridge escalates the conflict.

## 8. Version Control Policy

### 8.1 Mission Control as the Tracked Workspace

Mission Control at `~/Cursor Projects/mission-control/` on the Mac Mini is a git repository. All documents, registries, workflow definitions, and tracked config files in that workspace are version-controlled. The commit history is the organization's operational audit trail.

### 8.2 What Must Be Committed

The following must be committed to version control and must not exist only as untracked files:

- All D1.x–D5.x documents
- Task register entries (as they are created or updated)
- Capability register entries (D5.7)
- Workflow library entries (D5.6)
- Delegation status register
- Machine baseline declarations (see §10)
- Automation cron definitions (D1.8 §9.1)
- Proof artifact references (not the artifacts themselves — the references to them)
- Any approved configuration change that affects environment scope

### 8.3 What Must Never Be Committed

- Secrets, API keys, tokens, or credentials of any kind
- Temporary debug output or raw model responses
- Personal identifiers or private human data
- Large binary files (model weights, databases — these are referenced, not stored)
- Local scratch work that has not been reviewed and promoted (working notes go in working memory, not in tracked docs)

Secrets committed to the repo — even briefly — are treated as compromised and must be rotated. History rewrites to remove secrets from git history are performed before the branch is merged, not after.

### 8.4 Config Changes Require Document Update First

If a config change affects the declared behavior of any D-series document, the document is updated first and committed before the config change is applied. The commit message references the document section being implemented. This prevents the running environment from diverging silently from its documented state. (Source: D1.1 §3 principle 1, D1.2 guardrail 6)

### 8.5 Reusable Methods Must Be Tracked

A prompt template, evaluation rubric, or workflow method that has been used successfully is not informally remembered — it is registered (as CAP-WFM in D5.7 or as a workflow in D5.6) and committed. Methods that exist only in agent working memory or in a chat thread are not organization assets. They are personal knowledge that will be lost. (Source: D1.12 §10)

## 9. Environment Promotion Policy

Work moves through environments in order. Riskier changes require more steps. No change skips the sequence without explicit human approval and documented justification.

### 9.1 Promotion Path

```
Experimental → Sandbox → Production

For low-risk changes (Risk Class 1, no external dependencies):
    Experimental → Production (abbreviated path, human acknowledgement required)

For medium-risk changes (Risk Class 2):
    Experimental → Sandbox → Production (full path, explicit approval required)

For high-risk and critical changes (Risk Class 3–4):
    Experimental → Sandbox → 48h hold → Production
    (written approval + experiment record + human sign-off on each transition)
```

### 9.2 Promotion Requirements Per Transition

**Experimental → Sandbox:**
- Experiment record exists in D5.4
- Experiment result documents that the approach works as intended
- Risk class has been assigned
- No production credentials used

**Sandbox → Production:**
- Sandbox validation confirmed and documented
- Adoption proposal submitted and approved per D1.12 §7 (if a new capability)
- Any new config changes committed to version control
- Rollback path identified (what is done if the production change fails)
- For Risk Class 3+: 48-hour hold period completed, human has explicitly signed off

### 9.3 Rollback Expectation

Every production change must have a declared rollback path before it is applied. "We will figure it out if something breaks" is not a rollback path. The rollback is documented in the task record and the capability/workflow entry before the change is applied.

### 9.4 No Direct Experiment-to-Production for Risk Class 2+

An experiment that produces a working result is a candidate for adoption — not an approved production change. The result triggers the adoption proposal process (D1.12 §6). The proposal is approved. The sandbox run is conducted. Production activation follows. The excitement of a successful experiment is not a reason to skip the path.

## 11. Agent Change Authority Within Environments

Agents operate within declared authority boundaries. What is not explicitly permitted is not permitted.

### 11.1 What Agents May Change Autonomously

Bridge (Conductor) may update the following without human approval:
- Task status fields in the Mission Control task register (in-progress, blocked, completed, escalation reason)
- Operational memory (task session logs, routing history)
- Capability register fields within the write scope defined in D1.12 §8.1 (`status`, `last_reviewed`, `review_due`, `notes`, cost usage tracking)
- Workflow library entries marked `draft` — draft creation only; activation requires human approval
- Cron schedule execution (not cron schedule definition — definition requires human approval)

Forge may update the following without human approval:
- Working memory in its own workspace
- Experiment records in D5.4 (filing results — not approving outcomes)
- Capability candidate stubs (creating `candidate` status entries in D5.7 — not approving)

Brain (Archivist) may update the following without human approval:
- Canonical memory via the promotion process (D1.7 §9)
- Memory archive entries (receiving and writing promoted entries)

### 11.2 What Agents May Not Change

No agent may change the following without human approval:
- Environment type definitions or machine-to-environment mapping (§4, §5)
- Config Layer 1 (org-wide policy) values
- Capability approval status (`approved_by`, `approval_date`, `risk_class`, scope fields)
- Autonomy level for any task class
- Escalation trigger thresholds (D1.8 §8)
- Secrets or credential entries (D1.14)
- Any D-series policy document

No agent may silently change machine-local configuration (Layer 2) without logging the change and referencing it in the next daily digest. Silent config changes are drift. (Source: D1.8 §6.5, D1.1 §3 principle 15)

### 11.3 Proposed Changes

When an agent identifies a configuration change that would require human approval, it:
1. Documents the proposed change in the task record (what, why, expected effect)
2. Surfaces it in the next daily digest as a pending proposal
3. Does not apply the change until the human approves

The agent does not delay current work to wait for approval on a non-blocking proposal. It completes what is approvable and surfaces the proposal asynchronously.

## 12. Reusable Methods and Workflow Consistency

This section defines how the organization converts successful ad-hoc approaches into repeatable, version-controlled, environment-consistent methods.

### 12.1 Methods and Workflows Are Consistency Controls

A workflow or method is not just a convenience — it is a consistency enforcement mechanism. When an agent follows an approved workflow, it produces the same class of output in the same way, regardless of which agent instance runs it or when. This is the precondition for reliable reuse, reliable measurement, and reliable improvement.

Environment consistency is what makes workflow reuse produce reliable results. If the same workflow runs in different environments with different dependencies, different credentials, or different tool versions, the outputs will diverge. A method that depends on a specific environment state is not a reusable method — it is a one-time procedure.

### 12.2 When a Method Should Be Registered

A method becomes a candidate for registration when:
- The same approach has been used successfully 3 or more times for the same task class
- The approach depends on specific environment state (tool versions, paths, config values) that must be consistent
- The approach produces a testable output that can be validated
- The method's removal or modification would affect multiple workflows or agents

Registration path:
- Workflow execution sequence → D5.6 (workflow draft, promoted per §5.4 of that document)
- Reusable technique or template → D5.7 as CAP-WFM (per D1.12 §10)

### 12.3 Prefer Approved Methods Over Improvisation

When an approved method or workflow exists that covers a task, it is used. Improvising a different approach for the same task class is a deviation. Deviations are flagged in the task record and surfaced in the next digest. This is not bureaucracy — it is how the organization detects when an approved method needs updating. (Source: D5.6 §4.1, D1.12 §3 principle 2)

### 12.4 Environment Consistency as a Method Requirement

When a method is registered, its environment requirements are declared:
- Which environment it runs in (Production / Sandbox / Experimental)
- Which machine it runs on
- Which tool versions or dependencies it requires
- Which config layer values it reads

If the environment changes and a previously approved method no longer works as declared, the method is flagged as inconsistent. The running environment is not silently adjusted to match the method — the method is reviewed and updated through the standard change process.

### 12.5 Sprint-Like Execution Within Approved Boundaries

The swarm executes approved work in sprint-like cycles: agents pull eligible backlog tasks, sequence them by priority, execute them, and close them without requiring human re-prompting for each step. Workflow consistency is what makes this work — a consistent environment means a consistent result means a reliable track record. (Source: D1.1 §3 principle 15, D1.11 §10.3)

Agents do not need to reinvent the approach for every task. The backlog defines what to do. The workflow defines how. The environment defines the substrate. The agent executes.

## 13. Drift Detection and Correction Policy

### 13.1 What Counts as Drift

Drift is any departure from a declared baseline or approved configuration that was not explicitly authorized and recorded. Categories:

| Drift Type | Examples |
|---|---|
| Config drift | A Layer 2 or Layer 3 config value differs from its declared value with no update record |
| Version drift | A tool, service, or dependency is at a different version than declared in baseline |
| Service drift | A required service is not running, or an undeclared service is running |
| Capability drift | A capability is in use but its registered scope, status, or version no longer matches its actual usage |
| Method drift | An approved workflow is being executed differently from its D5.6 definition |
| Credential drift | A credential is in use that is not registered in D1.14, or is beyond its rotation window |

### 13.2 How Drift Is Detected

- Bridge performs a baseline check against declared machine state on a defined cadence (cadence TBD — registered in Mission Control cron register after Phase 1)
- Agents flag any departure from declared state they encounter during task execution
- Version checks on startup: agent validates its own environment against its declared baseline before accepting tasks
- Capability register review: bridge compares active capability usage against registered scope during daily hygiene checks

### 13.3 What May Be Auto-Corrected

Auto-correction is permitted for:
- Restarting a declared required service that has stopped (with the restart logged)
- Re-validating a baseline check that failed transiently (network timeout, etc.) — retry once before escalating
- Clearing ephemeral working memory that has exceeded its retention window (D1.7)

Auto-correction is not permitted for:
- Config value changes (even to restore a prior declared value — must be human-approved or logged)
- Reinstallation or upgrade of any tool or service
- Any change that affects environment scope, agent authority, or production state

### 13.4 Escalation for Drift

Drift that cannot be auto-corrected is:
1. Logged in the task record and the capability/method record
2. Surfaced in the next daily digest for non-critical drift
3. Escalated immediately (D1.9 §10.3 format) for drift that is affecting live production tasks

Bridge does not attempt to "fix" unknown drift by self-directed environment reconstruction. It reports the drift and halts affected tasks until the human provides direction. (Source: D1.8 §8 escalation trigger 1 — scope boundary)

## 14. Recovery and Rebuild Expectations

### 14.1 The Core Standard

Any production environment must be rebuildable from version-controlled documentation and approved configuration. If a machine is lost or fully corrupted, the organization must be able to reconstruct its environment to a known-good baseline by following recorded documentation — not by trying to remember what was done.

This standard is not aspirational. It is the minimum requirement for any production environment.

### 14.2 What Rebuild Requires

For a rebuild to be possible:
- All configuration decisions must be committed to version control before they are applied (§8.4)
- All installed tools and services must have their setup path documented in D2.x runbooks
- All secrets must be registered in D1.14 with enough information to re-generate or retrieve them
- All baseline state declarations (§10) must be current and committed
- All approved capabilities must have their activation requirements recorded in D5.7

A machine rebuilt from these sources should produce an environment that is functionally identical to the one it replaced — not a best-effort approximation.

### 14.3 No Critical Dependency on Machine State Alone

No critical operational behavior may depend on state that lives only in a running machine's memory, an untracked local file, or a chat history. This includes:
- Workflow methods known only to a single agent's context
- API endpoint configurations not committed to version control
- Routing decisions not documented in D1.9
- Capability activations not registered in D5.7

### 14.4 Recovery Restores Declared Baseline, Not Improvised State

When a machine is recovered, the goal is to restore its declared baseline (§10) — not to reconstruct what it happened to be running at the moment of failure. If the declared baseline is insufficient for current needs, the baseline is updated through the change process before the rebuild. Rebuilding to a different state than the declared baseline without updating the declaration is a form of configuration drift. (Source: D1.2 guardrail 5)

## 16. Immediate Environment Priorities

These are the concrete, sequenced actions required before environment control can be considered operational.

### Before Phase 1 Exits

1. **Initialize git repository in Mission Control workspace** — `git init` in `~/Cursor Projects/mission-control/`. Create an initial commit with all existing D-series documents. Establish `.gitignore` that explicitly excludes secrets, credentials, and any path that D1.14 will designate as the secrets store. This must exist before any agent writes to Mission Control. (Source: §8.1)

2. **Write D1.14-SECRETS_POLICY.md** — Environment control is incomplete without a defined secrets management policy. D1.14 must exist before any credential is registered, any bot token is configured, or any API key is set. D1.14 is the next document in the sequence. (Source: §6, §8.3, §10)

3. **Declare Mac Mini baseline in committed form** — Write the Mac Mini baseline declaration as a committed file in Mission Control (not just as prose in this document). The baseline file must be referenced by bridge as the authoritative check target. This becomes the first drift detection anchor. (Source: §10.1)

4. **Decide on Mission Control git remote** — Before Phase 1 operation begins, decide: does the Mac Mini git repo have a remote backup? Options are a private GitHub repo, a local NAS mirror, or mirroring to another basement machine once Phase 2 is live. Document the decision. A single-point git repo is an organizational risk. (Source: §15, D1.2 §13)

5. **Confirm bridge write access scope is implemented** — Before bridge operates on any registry, verify that its write authority (D1.12 §8.1, §11.1 of this document) is implemented as config-level permissions, not just as documented policy. Trust but verify. (Source: §11.1)

### Phase 2 Setup (When Cluster Comes Online)

6. **Survey Desktop before touching it** — Before any configuration change on the Desktop, inventory its existing state: OS, running services, n8n status, local ports, active legacy processes. File results in D5.3. Nothing runs until classified. (Source: D1.1 §2, D1.2 §3, §10.2)

7. **Establish sandbox isolation path on Desktop** — Once Desktop OS is known, define the specific directory path and access controls that constitute the sandbox environment on Desktop. Commit the declaration before any sandbox work begins. (Source: §4.2, §5)

8. **Commit GPU3060 and Desktop baseline declarations** — After Phase 2 integration, write and commit baseline declarations for both machines (§10.2, §10.3). Bridge's drift detection cannot cover these machines until their baselines exist. (Source: §13.2)
