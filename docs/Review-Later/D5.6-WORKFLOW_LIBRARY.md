# D5.6 — WORKFLOW_LIBRARY
**Autonomous AI Organization — Reusable Company Process Definitions**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document is the Workflow Library for the organization. It stores approved, reusable definitions for recurring work that agents have been authorized to execute without improvising a new approach each time.

A workflow is not a runbook (that is D2.x). It is not a delegation spec (that is D3.x). A workflow is a repeatable execution pattern — the step sequence, inputs, outputs, handoffs, and proof requirements for a specific class of work that happens often enough to warrant standardization.

**Why this matters:**
- Repeated patterns executed ad hoc are inconsistent, slow to improve, and hard to audit
- Approved workflows allow bridge (Conductor) to execute known work reliably without human re-explanation
- Workflow reuse is part of the swarm's self-managed operating responsibility (D1.8 §6.5)
- When the same pattern recurs without an approved workflow, that is a signal to create one

**What this document contains:**
- The workflow registry (list of approved workflows by ID)
- The format every workflow definition must follow
- The promotion process for converting ad-hoc patterns into approved workflows
- The deprecation process for workflows that are no longer valid

**What this document does not contain:**
- Runbook steps for setting up systems — D2.x
- Delegation specs defining task class authorization levels — D3.x
- Policy on what may be automated — D1.8-AUTOMATION_POLICY.md
- Experiment records — D5.4-EXPERIMENT-REGISTRY.md

## 3. Workflow Definition Format

Every workflow in the library must conform to this format. Incomplete workflow definitions are not approved for agent-initiated execution.

```
---
id: WF-<sequential number, e.g. WF-001>
name: <Short human-readable name>
version: <integer, starting at 1>
status: draft | active | deprecated
owner_agent: bridge | brain | forge
automation_class: <Class 1–5 from D1.8 §4>
risk_level: <Level 1–4 from D1.8 §5>
autonomy_level_required: <0–5 from D1.4 §10>
phase_available: <phase number>
approved_by: human
approval_date: YYYY-MM-DD
last_reviewed: YYYY-MM-DD
source_task_pattern: <description of the recurring pattern this was derived from>
delegation_spec_ref: <D3.x reference if applicable, or null>
---

## Inputs
<What the workflow requires to start: data, references, task packet fields>

## Steps
<Numbered sequence of steps. Each step names the executing role, the action, and the expected output.>

1. [Role] — [Action] → [Output]
2. [Role] — [Action] → [Output]
...

## Decision Points
<Any step where the workflow branches. Describe each condition and which branch is taken.>

## Outputs
<What the workflow produces: proof artifact, memory entry, notification, or downstream task.>

## Handoffs
<Who receives the output and what they do with it.>

## Failure Handling
<What happens at each step if it fails. Reference escalation triggers from D1.8 §8 where applicable.>

## Proof Requirement
<What artifact confirms the workflow completed. This satisfies D1.8 §15.>

## Change Notes
<Version history: what changed and why.>
```

## 4. Self-Management Rules for Workflows

### 4.1 When Agents Use a Workflow

Bridge (Conductor) checks the workflow registry before executing any recurring task. If an active workflow exists for the task class, that workflow is used. The task record must reference the workflow ID. (Source: D1.3 BRG-SWM-03, D1.8 §6.5)

### 4.2 When No Workflow Exists

If no workflow exists for a task class being executed:
- The agent documents the approach used in the task record under `approach: ad-hoc`
- If the same pattern occurs 3 or more times, Conductor surfaces a workflow candidate proposal to the human (see §5.3)
- The human decides whether to promote it or leave it ad-hoc

Agents do not self-promote workflows to `active` status. Human approval is required. Self-managed, but not self-authorizing beyond policy. (Source: D1.1 §3 principle 15)

### 4.3 Workflow Execution Is Not Modification

Executing an approved workflow does not grant permission to modify it. If an agent encounters a step that no longer works as defined, it halts at that step, records the issue in the task record, and surfaces it in the next digest. The workflow is updated only through the change process in §5.4.

## 5. Workflow Lifecycle

### 5.1 Origin: Pattern Recognition

A workflow candidate originates when the same task pattern recurs. The trigger is either:
- Human notices a pattern and requests a workflow be created
- Conductor detects 3+ ad-hoc executions of the same task type within a 30-day window and surfaces a candidate

### 5.2 Draft Creation

When a candidate is surfaced, a workflow draft is created:
- Conductor (or the human) writes the draft using the format in §3
- Status is set to `draft`
- The draft is filed in this document under the registry
- Conductor does not execute the draft as an approved workflow — it remains ad-hoc until promoted

### 5.3 Candidate Surface Format

When Conductor surfaces a workflow candidate, it posts to the ops-channel in this format:

```
Workflow candidate: [task name]
Pattern observed: [N] executions in last [N] days
Consistent approach used: yes / no
Draft available: yes (WF-XXX draft in D5.6) / no
Action: Review and approve, modify draft, or decline
```

### 5.4 Promotion to Active

A workflow draft is promoted to `active` when:
1. The human reviews the draft and approves it (Telegram or Mission Control record)
2. The `status` field is updated to `active`
3. The `approved_by` and `approval_date` fields are populated
4. The workflow is added to the registry table in §2

Promotion may not be initiated by an agent.

### 5.5 Versioning

When an active workflow is modified:
- The version number increments
- The `change_notes` section records what changed and why
- The prior version is not deleted — it is retained in the `change_notes` for audit
- If the change is significant (different risk level, different output target, different steps), re-approval is required before the new version is executed

### 5.6 Deprecation

A workflow is deprecated when:
- The task class it covers is no longer approved or has been superseded by a new approach
- The workflow is consistently failing and cannot be corrected within scope
- The human explicitly retires it

Deprecated workflows are marked `deprecated` in the registry. They are not deleted. A deprecated workflow may not be executed by an agent.

Confirm this access path is functional when the broader Mission Control access policy is resolved (D1.3 MC-ACC-03). (Source: D1.3 BRG-SWM-03, BRG-SWM-04)
