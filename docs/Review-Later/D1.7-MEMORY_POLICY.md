# D1.7 — MEMORY_POLICY
**Autonomous AI Organization — What Gets Stored, How, Retention, and Retrieval**
_Last updated: 2026-03-11 | Status: Active | Parent: D1.1-ROADMAP.md_

## 1. Purpose

This document defines the memory policy for the organization. It governs what counts as organizational memory, where each kind lives, who may write it, how long it is kept, when it is promoted or retired, and how it is retrieved.

**This document resolves:**
- BRN-MEM-03 — Memory technology decision (deferred from D1.3)
- MEM-CAN-04 — Retention policy (deferred from D1.3)

**What this document does not cover:**
- File paths and directory structures — those live in D1.2-ARCHITECTURE.md and are updated when machines are surveyed
- Secrets and credentials — D1.14-SECRETS_POLICY.md
- Model context windows and in-flight reasoning state — those are inference concerns covered by D1.5-MODEL_POLICY.md
- Research intake and experiment records — D5.1-INTAKE-LANES.md and D5.4-EXPERIMENT-REGISTRY.md
- Runbook steps for setting up memory infrastructure — D2.x

## 2. Scope

Memory in this organization spans four layers, each bound to specific machines and phases.

| Layer | Machine | Phase Required | Status |
|---|---|---|---|
| Canonical | Desktop (brain) | Phase 2 | Planned |
| Operational | Mac Mini (bridge) | Phase 1 | Required now |
| Working | GPU3060 (forge) | Phase 2 | Planned |
| Edge receipts | Pixel / Seeker (edge) | Phase 5 | Far future |

MacBook Air is excluded. It is the human control hub and does not store organizational memory.

This document specifies policy for all four layers. Only the operational layer on Mac Mini is active at Phase 1. Canonical and working layers are specified now so Phase 2 can build to a stable policy rather than invent one under pressure.

## 3. Policy Principles

1. **Files are the source of truth.** Organizational memory lives in structured files on local disk. Indexes, vector stores, and caches are derived from files and can be rebuilt. Files cannot be rebuilt from indexes.

2. **Human-readable by default.** Every persistent memory entry must be readable by the human operator without specialized tooling. Markdown with structured front matter is the required format.

3. **One canonical home.** Long-term organizational knowledge has one authoritative location: Desktop. No other machine may hold an authoritative duplicate of the canonical store.

4. **Brain (Archivist) is the sole writer to canonical memory.** Bridge and forge do not write directly to canonical memory. They submit entries for Archivist to validate, format, and commit. This protects the integrity and quality of the canonical store.

5. **Memory is written at task boundaries.** Agents write memory when tasks start, complete, or fail — not during in-flight reasoning, not during heartbeats, not in response to chat.

6. **Promotion is explicit, not automatic.** An entry does not move from working to operational, or from operational to canonical, unless a defined promotion rule is satisfied. Memory does not self-elevate.

7. **Retention is explicit, not indefinite.** Every memory layer has a defined retention window and an explicit policy for what happens when the window ends. Memory does not accumulate forever by default.

8. **Retrieval must not require the human.** Bridge must be able to read relevant canonical memory from Desktop without the human transferring it manually. The mechanism is specified in §11.

9. **Quality over volume.** A small set of high-signal entries serves the organization better than a large pile of low-value logs. The quality policy in §12 defines what qualifies as a memory entry.

## 4. Memory Layers

The organization maintains four memory layers. They differ in lifespan, ownership, write authority, and intended use.

### 4.1 Canonical Memory
**What it is:** The long-term organizational knowledge base. Completed task records, decisions, research findings, organizational knowledge, and milestones that the organization needs to remember across phases, reboots, and agent restarts.

**What it is not:** A diary, a chat log, an activity feed, or a record of every inference call. It does not record that something was tried — it records what was learned, decided, or accomplished.

**Owner:** Brain (Archivist role on Desktop)
**Write authority:** Archivist only — no other agent writes directly to canonical memory
**Lifespan:** Long — permanent until the retention policy triggers archival or compression
**Phase:** Phase 2

### 4.2 Operational Memory
**What it is:** The active state and short-term context the bridge agent needs to run tasks: task queue state, active task context, scheduled job records, recent error logs, and daily operational summaries.

**What it is not:** A substitute for canonical memory. Operational memory is what is happening now and recently — not what the organization knows.

**Owner:** Bridge (Conductor role on Mac Mini)
**Write authority:** Bridge only
**Lifespan:** Short — defined retention windows per entry type (see §10)
**Phase:** Phase 1

### 4.3 Working Memory
**What it is:** Transient in-progress artifacts generated during forge task execution: research scratch, build intermediates, QA states, partial outputs. These exist while a task is active.

**What it is not:** Persistent storage. Working memory is temporary. Valuable outputs must be extracted and promoted before task close or they are lost.

**Owner:** Forge (Builder, Critic, Researcher roles on GPU3060)
**Write authority:** Forge only within its own working directory
**Lifespan:** Temporary — cleaned up within 24 hours of task close
**Phase:** Phase 2

### 4.4 Edge Receipts
**What they are:** Lightweight records of completed edge actions emitted by Pixel/Seeker nodes. Receipts confirm what was done and when, without storing the full context locally. They are forwarded to bridge for inclusion in the operational log.

**What they are not:** Full memory entries. Edge nodes do not hold organizational memory.

**Owner:** Edge node (emits); Bridge (receives and logs)
**Phase:** Phase 5

## 5. Canonical Memory Policy

### 5.1 What Belongs in Canonical Memory

An entry belongs in canonical memory if it satisfies at least one of the following:

| Qualifies | Examples |
|---|---|
| A task has been completed and its outcome has lasting organizational value | Research brief filed by Researcher, build completion with proof artifact path |
| A policy, architectural, or strategic decision has been made | Memory technology choice, model routing decision |
| An observation or finding has been extracted that would inform future agent or human behavior | Repeated failure pattern, market signal, capability insight |
| A phase milestone has been achieved | Phase 1 exit gate satisfied |
| A periodic summary is due | Weekly digest committed by Brain |

An entry does NOT belong in canonical memory if it is:
- A chat reply or informal response
- A heartbeat or ping record
- A failed attempt with no learning extracted
- An intermediate reasoning step or model output that was not validated
- A duplicate of an existing entry with no new information

### 5.2 Entry Format

All canonical memory entries use structured markdown with YAML front matter:

```
---
id: <unique identifier>
date: YYYY-MM-DD
type: task | decision | knowledge | observation | summary | error | milestone
owner: bridge | brain | forge | human
task_id: <task identifier or null>
phase: 0 | 1 | 2 | 3 | 4 | 5
tags: [tag, ...]
status: active | closed | archived | superseded
---

# <Title — one sentence>

## Summary
<What this entry records. 2–5 sentences.>

## Content
<Type-specific content — see §5.3>

## Source
<Where this came from: task ID, agent, handoff chain if applicable>
```

### 5.3 Content Requirements by Entry Type

| Type | Required Content |
|---|---|
| `task` | Input received, steps taken, output produced, proof artifact path or reference, handoff target |
| `decision` | Decision statement, options considered, rationale, source requirement IDs |
| `knowledge` | Source, extraction date, key findings, relevance tags, confidence (low / medium / high) |
| `observation` | What was observed, context, potential implications, severity |
| `summary` | Period covered, events, completions, failures, current system state |
| `error` | Error description, trigger, impact, recovery action taken, escalation (yes/no) |
| `milestone` | Milestone statement, phase, evidence or proof artifact reference |

### 5.4 Superseded Decisions

When a policy decision is changed or overruled, the old entry is not deleted. The `status` field is set to `superseded` and a `superseded_by` field is added with the ID and date of the replacement. Both entries remain in the canonical store.

### 5.5 Write Authority Enforcement

Archivist is the only role authorized to commit entries to the canonical store. Agents that have entries to submit — bridge completing a task, forge extracting a research finding — package them as handoff task packets addressed to Archivist (see §9 Promotion Policy). Archivist validates, formats if needed, assigns a canonical ID, and commits the file. If validation fails, the entry is returned with a reason and the submitting agent retains it in operational or working memory until corrected.

## 6. Operational Memory Policy

### 6.1 What Belongs in Operational Memory

Operational memory on Mac Mini holds what bridge needs to run the organization day-to-day:

| Entry type | Purpose | Written when |
|---|---|---|
| Task record (active) | Tracks an in-flight task — owner, scope, status, partial state | Task dispatched by Conductor |
| Task record (closed) | Completed task before promotion window expires | Task confirmed complete |
| Error log | Failure events, escalation records | Failure or escalation occurs |
| Cron/schedule records | What recurring jobs are defined and when they last ran | Job definition or run event |
| Daily summary draft | Day's events, pending for promotion to canonical memory | End of day (on schedule or trigger) |
| Edge receipts received | Forwarded records from edge nodes (Phase 5) | Receipt received from edge |

### 6.2 What Does NOT Belong in Operational Memory

- Heartbeat pings and liveness checks — these are not memory events
- Every model inference call or routing decision — inference logs are for the inference stack, not the memory layer
- Chat messages and Discord replies that do not advance a task
- Duplicate status updates for the same task event

### 6.3 Persistence Requirement

Operational memory must survive:
- Mac Mini clean reboot
- Unplanned power loss
- OpenClaw process crash and restart

Operational memory paths must be on primary local disk. No tmpfs or volatile paths. (Source: MEM-OPS-01)

### 6.4 Reboot Recovery Procedure

After an unplanned Mac Mini restart: (Source: MEM-OPS-02)
1. OpenClaw restarts via launchd
2. Bridge reads operational memory — tasks marked `active` at crash time are flagged `interrupted`
3. An `error` entry is written for each interrupted task
4. Bridge posts a recovery notice to Discord listing interrupted tasks with their IDs
5. Human reviews and re-dispatches or manually closes interrupted tasks
6. No silent replays — each re-dispatched task creates a new task record

Model context from before the crash is not recoverable. Recovery depends entirely on what was written to operational memory before the crash occurred.

## 7. Working Memory Policy

### 7.1 Purpose

Working memory is the scratch space forge uses during active task execution. It exists so forge can produce incremental outputs without losing state between inference calls or tool uses. It is not intended to be preserved.

### 7.2 Scope of Working Memory

Working memory includes:
- Research scratch (partially processed source material, URL notes, partial summaries)
- Build intermediates (generated code files before QA, test scaffolds)
- QA states (test run logs, comparison outputs)
- Task-local context files that Researcher, Builder, or Critic write during task execution

### 7.3 Structure

Each active forge task has its own working directory. Bridge includes the working directory path in the task packet when dispatching to forge. Forge writes exclusively within that directory.

### 7.4 Cleanup Rules

| Trigger | Action |
|---|---|
| Task closes successfully | Valuable outputs extracted and submitted as promotion handoff to Archivist. Working directory deleted within 24 hours. |
| Task closes with failure | Error summary extracted if applicable. Working directory deleted within 24 hours. |
| Task abandoned (no close after 72 hours) | Working directory flagged for cleanup. Forge posts a WARNING to Discord. Bridge records an abandoned task entry in operational memory. Human reviews. |
| Working directory size threshold exceeded | Forge posts a WARNING to Discord. Human may authorize manual cleanup of oldest abandoned directories. |

Forge does not accumulate working memory across tasks. Clean-up is not optional. (Source: MEM-WRK-01)

### 7.5 Brain May Read Forge Working Memory

Archivist may read a forge working directory to extract a valuable finding directly without waiting for forge to submit a formal handoff. This is permitted as a read-only shortcut. Forge retains write authority within its working directory; Brain has no write access to forge's machine. (Source: MEM-WRK-02)

## 8. Human-Readable Memory Surfaces

### 8.1 Requirement

The human operator must be able to read what the organization has remembered without querying a database, writing code, or asking an agent. This is a hard requirement, not a nice-to-have. (Source: MEM-HUM-01, MC-MEM-01)

### 8.2 Day View

The memory surface in Mission Control must present all memory entries indexed by date. The default view is the current day. Navigation supports moving backward and forward by day. (Source: MC-MEM-02, MEM-HUM-01)

This view covers:
- Operational entries for today (active tasks, completions, errors) — sourced from Mac Mini
- Canonical entries dated today (completed task records, decisions, knowledge) — sourced from Desktop once online

### 8.3 Long-Term vs Daily Distinction

Long-term canonical memory entries must be visually distinguishable from daily operational entries. The human must be able to tell at a glance whether an entry represents a persistent organizational record (canonical) or a current operational event (operational). (Source: MC-MEM-03, MEM-HUM-02)

### 8.4 Agent Access via Same Structure

The memory architecture must support both human browsing and agent read access via the same underlying files. There is no separate export format for the human view. Mission Control reads the same structured files that agents read. (Source: MC-MEM-04)

### 8.5 Phase 1 Memory Surface Scope

At Phase 1, the memory surface displays operational memory from Mac Mini only. Canonical entries are not present until Desktop is online and the promotion pipeline is active. The day view is useful from Phase 1 for operational entries alone.

### 8.6 Legacy Asset Evaluation

Before building any new memory surface in Mission Control, previously built operational assets that already satisfy the memory screen requirements must be evaluated. Reimplementation without evaluation is waste. (Source: LEG-MIG-04)

## 9. Memory Promotion Policy

Promotion is the process by which memory moves from a lower tier to a higher, longer-lived tier. Promotion is always explicit, gated, and agent-authored — not automatic.

### 9.1 Working Memory → Operational

Forge submits a promotion request to bridge when a task closes and there are findings worth preserving beyond the working directory's cleanup window.

**Promotion packet contents:**
- Submitting agent: forge
- Entry draft (markdown with front matter, type: `task` or `knowledge`)
- Task ID that generated it
- Proof artifact path (file, log, or structured result — see D1.1 §3 principle 12)

Bridge (Conductor) receives the packet, adds it to the operational task record, and queues it for promotion to canonical memory.

**What is not promoted:** Scratch, failed intermediates, incomplete outputs. The submitting agent determines what has value. If nothing has value, nothing is submitted.

### 9.2 Operational Memory → Canonical

Bridge submits promotion requests to Brain (Archivist) at task close (for task records) and on the daily summary schedule (for the day's summary draft).

**Promotion packet contents:**
- Submitting agent: bridge
- Entry draft (markdown with front matter)
- Promotion basis: task close | daily summary | error escalation | milestone

Archivist validates against the entry type requirements in §5.3. If valid: assigns a canonical ID, commits the file to the canonical store. If invalid: returns the entry to bridge with a rejection reason. Bridge retains it in operational memory until corrected.

### 9.3 What Triggers Promotion

| Trigger | Entry Type | Destination |
|---|---|---|
| Task close confirmed (bridge completes dispatch loop) | `task` | Canonical |
| Forge extracts research finding | `knowledge` | Operational first, then canonical |
| Decision made (policy doc created or updated) | `decision` | Canonical |
| Daily summary scheduled | `summary` | Canonical |
| Error escalation (severity: critical) | `error` | Canonical |
| Phase milestone achieved | `milestone` | Canonical |

### 9.4 What Is Never Promoted

- Heartbeat and cron execution logs
- Routine Discord replies with no task outcome
- Duplicate entries for the same event
- Draft entries that fail Archivist validation and are not corrected within the retention window

### 9.5 Proof Requirement

Every task entry submitted for promotion must include a reference to a proof artifact: a file path, log reference, or structured result. Entries without proof can only be promoted as `observation` type, not `task`. (Source: D1.1 §3 principle 12)

## 10. Memory Retention Policy

Retention windows define how long each entry type lives in each tier before it is archived, compressed, or deleted. This resolves MEM-CAN-04.

### 10.1 Operational Memory Retention (Mac Mini)

| Entry Type | Retention Window | End-of-Window Action |
|---|---|---|
| Active task record | Until task is closed | Transition to closed; apply closed task window |
| Closed task record | 30 days from close | If promoted to canonical: delete local copy. If not promoted: compress to one-line summary, retain 30 additional days, then delete. |
| Error record (non-critical) | 60 days | Delete |
| Error record (critical / escalated) | Promoted to canonical via escalation trigger | Delete local copy after canonical commit confirmed |
| Cron / job execution records | 7 days rolling | Rotate and delete — not promoted |
| Daily summary draft | 7 days | Promoted to canonical. If promotion fails within 7 days: delete and log the missed day as a gap. |
| Edge receipts | 14 days | Delete — not promoted unless contents warrant a canonical `observation` entry |

### 10.2 Working Memory Retention (GPU3060)

| Entry Type | Retention Window | End-of-Window Action |
|---|---|---|
| Active task working directory | Until task closes | Cleaned up within 24 hours of task close |
| Abandoned working directory | 72 hours from last activity | Warning posted; human reviews; directory deleted |
| Extracted outputs awaiting handoff | 48 hours after task close | Promoted or discarded; not retained longer |

### 10.3 Canonical Memory Retention (Desktop)

| Entry Type | Retention Window | End-of-Window Action |
|---|---|---|
| Task records | 12 months from close | Move to `/archive` subdirectory; compress to summary if original is bulky |
| Decision records | Permanent | Never deleted; superseded entries marked `status: superseded` |
| Knowledge entries | 12 months from creation | Marked `stale` at 12 months; archived if not refreshed within 90 additional days |
| Error records (critical) | 24 months | Archive; never permanently deleted |
| Summary entries | 6 months | Archive after 6 months; originals compressed into periodic digest |
| Milestone entries | Permanent | Never deleted |

**Archival:** Archived entries move to a designated archive subdirectory within the canonical store. They remain human-readable and retrievable. They are not deleted unless the human explicitly authorizes a purge.

## 11. Retrieval Policy

### 11.1 Bridge Retrieving from Canonical Memory (Phase 2+)

Bridge must retrieve relevant canonical entries without human involvement. (Source: BRN-MEM-02, MEM-CAN-02)

**Baseline mechanism (Phase 2):** Network-readable path from Mac Mini to Desktop canonical directory. Bridge reads files directly using text search over YAML front matter fields and file names.

**Enhanced mechanism (conditional, Phase 2+):** If a vector index or full-text search index is activated on Desktop, Bridge may query it via a local HTTP endpoint served by Brain. This is a performance enhancement, not a Phase 2 requirement. The baseline file-read path must remain functional as fallback. See §14 for the activation gate.

**What Bridge may retrieve:**
- Task records filtered by tag, date range, or task type
- Decisions filtered by policy domain or requirement ID
- Knowledge entries filtered by topic or tag
- Milestones filtered by phase

**What Bridge may not do via retrieval:** Write to canonical memory, trigger inference on Desktop as a side effect of a read request, or bypass the Archivist for writes.

### 11.2 Forge Retrieving from Working Memory

Forge reads its own working directory directly via local file system. No retrieval abstraction is needed.

### 11.3 Forge Retrieving from Canonical Memory (Phase 2+)

Forge may read canonical entries via the same mechanism as bridge (network read path to Desktop). This supports the Researcher role — retrieving prior findings before beginning a new research task to avoid duplication.

### 11.4 Human Retrieval

The human operator retrieves memory through Mission Control (see §13). No direct file access is required for day-to-day review.

### 11.5 Offline Fallback

If Desktop is unreachable, Bridge and Forge fall back to operational memory only. Tasks that require specific canonical knowledge they cannot retrieve are marked BLOCKED and queued until Desktop is reachable again. No canonical memory tasks are silently rerouted or skipped.

## 12. Memory Quality Policy

This section defines what prevents memory from becoming a clutter pile or a diary. Quality is enforced at the point of entry, not retroactively.

### 12.1 Entry Qualification Rules

An entry must satisfy all of the following to be accepted as a canonical memory write:

1. **It records something that happened** — not something that might happen, not a plan, not an opinion
2. **It has a defined type** — one of the seven types in §5.3; miscategorized entries are returned
3. **It has a task ID or explicit source reference** — orphan entries without provenance are rejected
4. **It adds information not already in the store** — duplicates are rejected; updates require explicit `superseded_by` linkage
5. **It passes the "would this matter in six months?" test** — if the entry would be irrelevant to a future agent reading the canonical store, it belongs in operational or working memory at most, not canonical

### 12.2 What Is Always Rejected from Canonical Memory

The following are never accepted as canonical memory entries, regardless of how they are submitted:

- Heartbeat logs and liveness pings
- Unvalidated model outputs (entries where the content was generated by a model but not reviewed by an agent or human before submission)
- Chat transcripts and Discord message logs
- Duplicate entries for the same event
- Entries with empty or placeholder content fields
- Entries submitted while a task is still `active` (task entries require task close)

### 12.3 Archivist's Review Responsibility

Archivist does not blindly commit what is submitted. It validates against the rules above and returns rejections with a reason. This is not optional — the Archivist role is the quality gate for the canonical store.

### 12.4 Operational Memory Quality

Bridge applies a lighter version of the same quality rules to operational memory:
- No duplicate task records for the same task ID
- No entries without a triggering task event
- Heartbeats and cron pings are never written as task records

## 13. Memory and Mission Control

Mission Control's memory screen is a read-only view over the memory store. It does not have its own data layer. It reads the same structured files that agents use.

### 13.1 Screen Requirements

The Mission Control memory screen must satisfy:

| Requirement | Source | Implementation Direction |
|---|---|---|
| Human-readable memory view | MC-MEM-01, MEM-HUM-01 | Formatted display of markdown entries from operational and canonical tiers |
| Day-indexed browsing | MC-MEM-02, MEM-HUM-01 | Default view: today. Navigation: day-by-day backward and forward |
| Long-term vs daily distinction | MC-MEM-03, MEM-HUM-02 | Visual separation of canonical entries (long-term) from operational entries (daily) |
| Same structure for human and agent | MC-MEM-04 | MC reads the same files agents read — no secondary export format |
| Expandable storage support | MC-MEM-05 | Architecture uses local disk per machine; adding storage or expanding to NAS or external drive is a path addition, not a schema change |

### 13.2 Phase 1 Screen Behavior

At Phase 1, the memory screen shows Mac Mini operational entries only. Canonical entries (Desktop) are not displayed until Desktop is online and the promotion pipeline is active. The screen is not empty at Phase 1 — task records, error entries, and daily summaries from bridge are visible immediately.

### 13.3 What Mission Control Does Not Do

- Mission Control does not write memory — it reads only
- Mission Control does not run queries against a vector store at Phase 1 — plain file reads are sufficient
- Mission Control does not serve as the canonical memory store — the files on Desktop are authoritative

2. **Verify reboot recovery behavior** — Trigger a controlled Mac Mini restart with an active task record in place. Verify interrupted task detection fires and the Discord recovery notice is posted. (Source: MEM-OPS-02)

4. **Define the daily summary schedule** — Decide what time bridge generates the daily summary draft and what events trigger early summary generation (e.g. task threshold reached). This feeds the retention window in §10.1.

5. **Stub the memory screen in Mission Control** — Minimal implementation: a list of today's operational entries, one entry per task record. Day navigation can come later. Human-readable is the requirement; visual completeness is not. (Source: MC-MEM-01, MEM-HUM-01)

### Phase 2 — after switch uplink and Desktop online

6. **Survey Desktop disk capacity** — Confirm available storage before sizing the canonical memory reserve. Record in D1.2.

7. **Confirm network read path from Mac Mini to Desktop** — Test that bridge can read a file from the Desktop canonical directory over LAN without human involvement. (Source: MEM-CAN-02)

8. **Run first end-to-end promotion test** — Bridge completes a task → submits a promotion packet → Archivist validates and commits → Human reads the resulting canonical entry in Mission Control. (Source: MEM-CAN-03)

9. **Write D3.3-DELEGATION-SPEC-MEMORY.md** — The Archivist handoff protocol needs a full delegation spec before the promotion pipeline goes to production. The design placeholder in §9 is not sufficient for implementation.

10. **Confirm working memory cleanup on GPU3060** — After Forge hardware survey, verify cleanup automation triggers correctly after task close, and define the working directory size threshold. (Source: MEM-WRK-01)
