# D1.16 — MEMORY_RND
**Autonomous AI Organization — Memory Research and Development Agenda**
_Last updated: 2026-03-11 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document defines the organization's research agenda for memory systems. It does not replace or modify D1.7-MEMORY_POLICY.md, which remains the authoritative operational policy for memory as it exists today. This document exists to systematically drive improvement in how the organization stores, retrieves, evolves, and acts on what it knows — and to ensure that improvements are governed, evidence-backed, and promoted through a controlled path before they affect production behavior.

Memory is a foundational constraint on organizational capability. An agent that cannot reliably access prior decisions, prior outputs, prior context, or prior errors is forced to rediscover and repeat. The productivity ceiling of a self-managed swarm scales with memory quality. This agenda treats memory improvement as a first-class organizational priority — one that requires research rigor, not just feature enthusiasm.

**This document governs:**
- What memory research questions are active and why
- What candidate memory evolutions are being studied
- What classes of experiments are permitted and at what scope
- How experiment results are evaluated
- What gates must be passed before any memory change enters operational policy
- How this research agenda relates to capability acquisition, workflow development, and self-managed swarm operation

**What this document does not cover:**
- Canonical memory structure and edit authority (D1.7 §5)
- Operational memory definitions and tier structure (D1.7 §6)
- Working memory policy and session boundaries (D1.7 §7)
- Memory promotion gates from working to canonical (D1.7 §9)
- Retention windows and expiry policy (D1.7 §10)
- Any currently active memory method, tool, or storage location (D1.7 governs those)

## 2. Scope

This research agenda applies to all potential changes to how the organization's agents store, retrieve, index, retrieve, summarize, search, or act on information across any memory tier. It covers:

| Research Domain | Examples |
|---|---|
| Long-term memory architecture | Vector databases, hybrid retrieval, embedding strategies |
| Cross-session continuity | How context is preserved across agent session boundaries |
| Working memory management | Compression, summarization, prioritization within context |
| Canonical memory governance | How policies and decisions remain accessible and searchable |
| Memory retrieval quality | Relevance scoring, staleness detection, confidence signals |
| Multi-agent memory sharing | How two agents access the same ground truth without conflict |
| Memory-task feedback loops | How completed task outcomes update reusable knowledge |
| Forgetting and expiry | Structured deletion, expiry policies, noise reduction over time |

**Out of scope for this document:**
- Prompt engineering improvements that do not change memory storage or retrieval mechanisms
- Improvements to task register structure that do not affect memory policy
- Model capability improvements (covered by D1.10 / D1.12 CAP-MOD)

## 3. Research Principles

1. **Operational pain justifies research; novelty alone does not.** A research question earns priority when it is grounded in a real limitation observed in production operation. New technology is interesting. A documented gap with evidence is a research mandate.

2. **Experiments must be bounded and isolated.** No memory research experiment may modify production memory state, overwrite canonical memory, or affect agent behavior outside a declared experimental scope. Experiments run in Experimental or Dev environments (D1.13 §4) unless explicitly approved otherwise.

3. **A successful experiment does not authorize production adoption.** Positive results from an experiment change the evidence base, not the policy. Production adoption requires passing the promotion gates in §10 and receiving explicit human approval. Experiment success is one input to the approval decision, not the decision itself.

4. **Local-first where practical.** Memory research should prefer solutions that operate within the organization's machine environment before requiring third-party API dependencies. External memory APIs introduce credential surface, reliability risk, and cost exposure. These are acceptable when the operational gain clearly justifies them — not as a default starting point.

5. **Research conclusions must be legible.** Every completed experiment produces a brief written summary: what was tested, what conditions applied, what was observed, what was concluded, and what the recommended next step is. Verbal or implicit conclusions do not count.

6. **Repeated successful methods surface as workflow or capability candidates.** If an experiment produces a memory method that works well and is applied more than twice, it becomes a candidate for CAP-WFM (D5.7) or a workflow entry (D5.6). Repeatedly re-deriving the same approach is waste; research should compound into reusable organizational knowledge.

7. **This agenda does not grant authority.** Being listed here does not authorize an agent to acquire new capabilities, integrate new tools, or change memory storage for production. All such changes follow D1.12 (capability acquisition) and D1.15 (change control).

## 4. Current Operational Boundary

This section documents where memory capability stands today, as of D1.7. It is the baseline that research is meant to improve.

| Memory Tier | Current State | Known Limitation |
|---|---|---|
| Working memory | In-context only; bounded by context window | No cross-session continuity; large tasks degrade near context limit |
| Canonical memory (D1.7 §5) | **Not yet active as a dedicated tier.** Phase 1 durable corpus: D1.x–D5.x policy documents and operational records in Mission Control git on Mac Mini, read by known file path only. Canonical memory store (Desktop / brain) is a Phase 2 target | Phase 2 store does not yet exist; Phase 1 corpus is accessible only by exact path — no indexing, no semantic retrieval, no agent-initiated write |
| Operational memory | Task registers and status fields in D5.x files | Flat markdown; no structured query capability |
| Episodic / audit memory | Change records (D1.15), capability register (D5.7), communication logs | Partially structured; no replay or summary capability |
| Declarative policy memory | D1.x documents | Well-structured; accessible only by reading whole documents; no indexed retrieval |

**Current agent memory access model:**
- Agents receive working memory at session start via context injection
- Agents read the Phase 1 durable corpus by reading specific files at known paths; this is not the canonical memory tier defined in D1.7 §5
- Agents cannot query across documents or across sessions without human assistance
- There is no vector index, embedding store, or semantic retrieval layer

**Current reliability gaps:**
- Context window saturation during long tasks leads to silent context loss
- Agents re-discover prior decisions because there is no efficient retrieval path
- Canonical memory updates require human-directed commits; agents cannot self-service corrections
- No signal when working memory is stale relative to canonical state

## 5. Research Themes

The organization's memory research is organized into six active themes. Themes define the territory; §6 defines the specific questions within each.

| Theme ID | Theme | Rationale |
|---|---|---|
| MEM-T1 | Retrieval and Search | Agents cannot efficiently query across the document corpus; most retrieval is manual navigation |
| MEM-T2 | Cross-Session Continuity | Each session starts from near-zero; prior work is only accessible if explicitly loaded into context |
| MEM-T3 | Working Memory Management | Long tasks saturate context; no compression or prioritization mechanism exists |
| MEM-T4 | Multi-Agent Consistency | When multiple agents work concurrently, there is no shared ground truth layer below the markdown files |
| MEM-T5 | Memory Hygiene and Expiry | Stale, contradicted, or superseded information accumulates without structured removal |
| MEM-T6 | Feedback and Learning Loops | Completed tasks and resolved incidents do not reliably update reusable organizational knowledge |
| MEM-T7 | External Retrieval Augmentation | Agents have no access to current information; all retrieval is from static model weights or files at known paths. Perplexity (D5.7 CAP-006) is a candidate external retrieval capability that may address the real-time information gap — evaluated under D1.10 §9.0 and TASK-013 |

## 6. Research Questions by Theme

### MEM-T1: Retrieval and Search

- What is the minimum index structure that allows an agent to find a relevant policy section without reading the whole document?
- Is embedding-based search over the D1.x–D5.x corpus practical at current scale, and what is the maintenance cost?
- What retrieval quality is achievable with BM25 or structured keyword search versus vector search over this corpus?
- When should retrieval return a document excerpt versus a synthesized summary?
- What is the right granularity for chunking policy documents — section, paragraph, or heading level?
- Can retrieval be made deterministic enough that two agents asking the same question get consistent answers?

### MEM-T2: Cross-Session Continuity

- What is the minimum context injection needed at session start to give an agent meaningful continuity?
- Is a structured session summary (prior decisions, open items, last known state) sufficient, or is retrieval-on-demand necessary?
- What session state is worth persisting versus discarding?
- How do session handoffs between agents degrade context, and what can mitigate that degradation?
- Is there a light-weight journaling approach that does not require a vector database but still provides better-than-zero continuity?

### MEM-T3: Working Memory Management

- At what point in a long task does context saturation measurably degrade output quality?
- What compression strategies (summary generation, attention pruning, hierarchical compression) are viable within a single session?
- Can a "rolling summary" approach preserve task fidelity across context boundaries without human intervention?
- What is the right trigger for a compression event — token count, task phase boundary, or explicit signal?
- Does structured working memory (typed sections: decisions, open questions, completed steps) outperform unstructured working memory at scale?

### MEM-T4: Multi-Agent Consistency

- When two agents write to the same canonical file without coordination, what are the failure modes?
- Is a lightweight write-lock mechanism sufficient, or is a merge/conflict model needed?
- How can agents detect that their local view of canonical memory is stale?
- What coordination primitive (file-based, message-based, or register-based) minimizes conflict without introducing bottlenecks?
- Does a "canonical memory changelog" (an append-only log of recent updates) allow agents to resync without full re-reads?

### MEM-T5: Memory Hygiene and Expiry

- What classes of memory entries should have explicit expiry (task-context entries, experiment drafts, stale baselines)?
- How can superseded information be flagged without deleting it — and when does deletion become the right answer?
- What is the right review cadence for operational memory that has not been read or referenced in a defined period?
- Can a lightweight "memory audit" be operationalized as a recurring task without becoming expensive?
- How does expiry policy interact with the audit trail requirement — some stale records must be retained for compliance?

### MEM-T6: Feedback and Learning Loops

- What information from a completed task is worth generalizing into reusable knowledge?
- What prevents the current post-task synthesis from producing durable organizational learning?
- Is there a structured "task retrospective" format that produces a usable memory artifact rather than a closed task entry?
- How can repeated patterns in task completion surface as CAP-WFM or workflow candidates automatically?
- What is the cost-benefit threshold for committing a post-task synthesis artifact versus leaving the task record as the only trace?

## 8. Experiment Classes

Not all experiments carry the same risk. The class determines what environment is required, what authorization is needed, and what review the results must receive.

| Class | Description | Permitted Environment | Authorization Required | Risk Level |
|---|---|---|---|---|
| Class A — Read-only observation | Studying existing memory behavior without changing state | Any | Approved task entry; no additional authorization | Low |
| Class B — Isolated structure experiment | Trying a new memory structure on synthetic or dummy data | Dev or Experimental | Approved task entry; no additional authorization | Low |
| Class C — Live-data dev experiment | Using real but non-production data in a bounded Dev environment | Dev only | Approved task entry + human acknowledgement | Medium |
| Class D — Sandbox integration experiment | Running a candidate evolution in Sandbox with real data and real agents | Sandbox only | Human approval per D1.15 Medium change flow | Medium-High |
| Class E — Production pilot | Running a promoted evolution in Production on a bounded surface | Production | Full change record + human approval per D1.15 High change flow | High |

Class D and Class E experiments require a pre-declared rollback plan per D1.15 §8.3. Class E requires a pre-execution review per D1.15 §7.1.

## 9. Evaluation Criteria

An experiment does not conclude until it has been evaluated against all relevant criteria for its class. Partial evaluation is not sufficient for a promotion recommendation.

**Effectiveness criteria:**
- Does the candidate evolution measurably reduce the operational gap it was designed to address?
- Is the improvement durable across multiple test cases, or does it work only on the specific scenario it was designed for?
- Does the improvement hold when the agent is operating under realistic task load, not idealized conditions?

**Safety and governance criteria:**
- Does the candidate evolution introduce any new write surface not previously governed?
- Does it require any new credential, API key, or external dependency?
- Does it change how agents interact with canonical memory in ways that could produce silent corruption?
- Is the new behavior fully observable — can a human reviewing the outputs tell what memory operations occurred?

**Operational fit criteria:**
- Is the candidate evolution compatible with the existing D1.7 operational policy without requiring a D1.7 rewrite?
- Does it introduce net new maintenance cost that the organization can sustain at current scale?
- If it fails at runtime, does it fail safely — does the agent fall back to prior behavior without data loss?

**Adoption criteria:**
- Has it been applied successfully at least three times before being considered for operational policy promotion?
- Is there a clear rollback path if the promoted method turns out to be wrong?
- Has the evaluation summary been written in a form that is legible to a human reviewing it six months later?

## 10. Promotion Gates Into Production Policy

A memory evolution may not enter D1.7 or any other operational policy document until all of the following gates have been passed. Gate satisfaction must be documented in the experiment record, not assumed.

| Gate | Condition | Authority |
|---|---|---|
| G1 | Experiment classification and scope were declared before the experiment started | Change record per D1.15 |
| G2 | Experiment was conducted in the required environment class (§8) | Verifiable from experiment record |
| G3 | Evaluation criteria from §9 were applied and documented | Experiment summary written |
| G4 | No governance gap was introduced that is not addressed by a proposed D1.x update | Human review |
| G5 | Capability acquisition completed per D1.12 if any new tool or API is required | D1.12 approval record |
| G6 | Change record created per D1.15 for the D1.7 policy update | D1.15 change record |
| G7 | Human explicitly approved the D1.7 update | Telegram approval + MC record per D1.15 §7.1 |

Gates must be passed in order. G7 is the production gate. An agent may not apply a memory evolution to production behavior in advance of G7 on the grounds that it passed G1–G6.

## 11. Risks and Failure Modes

| Risk | Description | Mitigation |
|---|---|---|
| Silent canonical memory corruption | An agent writes a memory evolution output to a canonical path without authorization | D1.13 §11.1 write constraints; change records required for all canonical writes |
| Context poisoning | Injected session-start memory contains stale or incorrect state; agent acts on it | Review freshness of injected content; include canonical update timestamp |
| Runaway experiment scope | A Class B experiment is applied beyond its declared boundary | Explicit scope declaration required; Conductor enforces boundary |
| Promotion pressure from sunk cost | Substantial experiment investment creates bias toward promotion regardless of evaluation results | Evaluation criteria are applied independently; human gate is final |
| Memory layer proliferation | Multiple partial memory systems accumulate without clear authority | One authoritative location per state class (D1.13 §6); new memory systems must displace, not supplement |
| Cross-agent inconsistency | Two agents operating on different views of canonical memory produce contradictory outputs | Canonical memory changelog (MEM-E6) mitigates; multi-agent writes require coordination before any Class D+ experiment |
| Credential surface expansion | Memory evolution requires external embedding API, introducing new credential dependency | Local-first principle (§3.4); external API requires D1.12 + D1.14 compliance before experiment |
| Memory research consuming agent capacity disproportionately | R&D tasks crowd out operational priorities | Research tasks are backlog items subject to the same prioritization as all others; human declares priority |

## 12. Relationship to Workflows, Capabilities, and Self-Managed Swarm

**Relationship to D5.6 (Workflow Library):**
When a memory method or experiment protocol has been applied successfully three or more times, Conductor surfaces it as a workflow candidate per D5.6 §5.3. This includes memory experiment setup procedures, evaluation templates, and context injection patterns. Memory R&D is itself operationalizable — the research process should compound into reusable methods.

**Relationship to D5.7 (Capability Register):**
If a memory evolution requires a new tool, library, API, or model class, it must be registered in D5.7 as a capability prior to any Class C or higher experiment. Experiments do not bypass D1.12 capability governance. Memory research and capability governance are complementary, not competing.

**Relationship to CAP-WFM (Reusable Methods):**
Memory methods that become stable and generalizable — for example, a standard context injection template, a canonical write protocol, or a post-task retrospective format — are CAP-WFM candidates. They are registered in D5.7, not in D1.16. This document tracks the research path; D5.7 and D5.6 hold the outcomes.

**Relationship to self-managed swarm operations:**
Memory quality is a direct enabler of swarm autonomy. An agent that can reliably retrieve prior decisions, access relevant policies, and avoid repeating already-resolved problems requires less human re-direction. Memory R&D is therefore aligned with, not in tension with, the goal of reducing the human re-prompting load. However, the governance boundary applies equally to swarm agents: they cannot self-authorize a memory evolution, self-promote an experiment result, or write to canonical memory outside their declared write authority. The research agenda is set by this document; the execution follows the standard task and change governance.

**What the swarm may do in support of memory R&D:**
- Execute Class A and Class B experiments under approved task entries
- Write experiment draft summaries into working memory or the scratch notes area
- Surface CAP-WFM and D5.6 candidates based on repeated successful memory methods
- Propose changes to this research agenda as task proposals

**What the swarm may not do:**
- Self-authorize promotion of any experiment result into D1.7
- Integrate a new memory tool or API without D1.12 capability approval
- Modify canonical memory as part of an experiment without an approved change record
- Reclassify an experiment from a higher class to a lower class to bypass the required environment constraint

## 14. Immediate Priorities

**Phase 1 — Before any memory experiment begins:**

1. **Instrument the current operational baseline.** Before researching improvements, measure what exists: average task length before context saturation, frequency of re-discovery events (agent asks a question already answered in a prior session), canonical memory read frequency per task type. These baselines make subsequent experiment results meaningful.

2. **Define the canonical path for memory experiment records.** Where are experiment summaries stored? What file format? What naming convention? This must be decided before the first experiment is declared complete. Candidate: `mission-control/research/memory/experiments/MEM-E{N}-{slug}.md`.

3. **Evaluate MEM-E1 (structured context injection template) as the first Class B experiment.** This is the lowest-risk, highest-potential-return experiment: it requires no new tools, no new credentials, no external dependencies, and directly addresses the cross-session continuity gap. Draft the template, test it in three sessions, evaluate against §9 criteria, write the summary.

4. **Evaluate MEM-E2 (BM25 full-text index) as the first retrieval experiment.** Local, no external API required, directly addresses the policy retrieval gap. Scope: index D1.x and D5.x documents; measure recall on a defined set of test queries.

5. **Connect memory R&D findings to the CAP-WFM and D5.6 surfacing loop.** After the first two experiments, determine whether either result should produce a workflow candidate. The goal is that memory research produces compound organizational value, not one-off improvements.

**Phase 2 and beyond:**
- Proceed to Class C experiments after Phase 1 baseline and Phase 1 experiments are documented
- Evaluate embedding-based retrieval (MEM-E4) only after MEM-E2 results are complete and a capability approval path for the embedding model is confirmed
- Multi-agent memory coordination (MEM-T4) becomes a priority when concurrent agent operations are operational
