# D1.1 — ROADMAP
**Autonomous AI Organization — Master Execution Spine**
_Last updated: 2026-03-13 | Status: Active_

---

## 1. Mission

Build and operate an autonomous organization of AI agents that produces real value 24 hours a day without requiring constant human attention. The human role is architect, director, and judge — not operator.

The system must be:
- **Durable** — survives reboots, outages, and drift without manual recovery
- **Legible** — every agent, role, and decision can be read, traced, and audited
- **Incrementally delegated** — autonomy is earned per task type, not granted wholesale
- **Self-improving** — the system gets better over time through deliberate research and integration

Beautiful UI is deferred. Legibility and accuracy come first. D1.2 ARCHITECTURE.md defines where Mission Control lives; this scope defines what it must contain before Phase 1 exits.

## 3. Planning Principles

1. **Document before building.** Every phase starts with a written spec, not code.
2. **One working thing beats three half-built things.** Complete each layer before adding the next.
3. **Real agents are few. Logical roles are many.** Three runtime agents carry all named roles internally. Do not create a new agent for every new capability.
4. **Earn autonomy.** New task types start supervised. Delegation expands only when track record justifies it.
5. **Local inference first.** Every worker machine runs local models. Cloud inference is a fallback, not a dependency.
6. **Avoid heartbeat noise.** Agents report on events and outcomes, not on existence. Silence is a valid state.
7. **The source of truth is Mission Control.** Telegram is the primary coordination channel. Logs are evidence. Code is implementation.
8. **Improvement is continuous but deliberate.** Articles, X posts, and experiments feed the system through defined intake lanes, not ad hoc sprints.
9. **Migrate before rebuilding.** Existing workflows, prompts, automations, docs, and infrastructure from prior systems must be inventoried before replacement work begins. Assume prior work has value until proven otherwise.
10. **Execution layers are subordinate.** Tools like n8n, scripts, browser automation, and external workflow systems are execution substrates — not the strategic brain and not the source of truth. They serve Mission Control; they do not replace it.
11. **Work moves through packets, not vibes.** Meaningful work must be described as a task packet with owner, scope, route, output, and handoff target. Informal instructions that skip this structure are not delegated work — they are chat.
12. **Every delegated task leaves proof.** Delegated execution must emit evidence — a log, a file, a structured result — not just a chat reply. Proof is what allows review, audit, and improvement.
13. **Agents may generate child prompts only within scope.** Prompt generation is allowed as part of delegated execution, but only inside an approved task class, with bounded depth and defined handoff rules. Unbounded prompt generation is not delegation — it is drift.
14. **Scale must respect reality.** The system is intended to overcome human execution bottlenecks, but must remain grounded in real hardware limits, model limits, and operational reliability. Ambition that ignores constraints produces technical debt, not capability.

15. **The swarm is self-managed within approved boundaries.** Once a goal, backlog, policy, workflow, or task class is approved, agents are expected to identify eligible work, pull tasks, sequence, execute, hand off, and close work on their own — without the human needing to push each step. Backlog hygiene, task pickup, estimation, sequencing, and workflow reuse are part of the swarm's operating responsibility. The human sets mission, priorities, constraints, approvals, and escalation thresholds. Agents manage execution. Self-managed, but not self-authorizing beyond policy.

16. **Tool lanes are self-managed within approved policy.** Three designated tool lanes serve specific task classes: OpenClaw (orchestration, scheduling, governance), Cursor (engineering, code, QA, document authoring), Perplexity (live web research, current-information retrieval, browser/computer-use tasks). Once tool lane assignments are approved in policy, agents select the correct lane automatically for each task without per-task human instruction. Tool lane policy changes go through D1.15 change control. Within approved policy, lane selection is an execution decision, not an approval gate.

17. **Access minimization is an operating principle.** Agents prefer access paths that are stable, low-maintenance, and do not require ongoing human credential management or device interaction. API access is not always the preferred path — if a built-in OpenClaw capability or a stable local integration can fulfill the same need without an external dependency, that path is preferred. Device-mediated or computer-use fallback lanes are a last resort for access tasks that cannot be satisfied by stable paths, and are governed as a candidate capability class (not an ad-hoc improvisation). The goal is to minimize the number of access surfaces the organization must maintain, not to maximize what agents can reach.

## 4. Real Runtime Shape

### The Three Agents

```
bridge   →  Mac Mini       Routing, scheduling, comms brokering, gateway management
brain    →  Desktop        Memory, planning, commerce, knowledge synthesis
forge    →  GPU3060        Build, QA, research, local inference workloads
```

### Logical Roles (hosted inside agents, not separate processes)

| Role | Lives In | Responsibility |
|---|---|---|
| Conductor | bridge | Coordinates task flow across agents |
| Broker | bridge | Routes messages, manages channel I/O |
| Archivist | brain | Maintains memory, indexes knowledge |
| Planner | brain | Produces plans, tracks goal state |
| Merchant | brain | Commerce logic, value tracking |
| Builder | forge | Code generation and execution |
| Critic | forge | QA, testing, evaluation |
| Researcher | forge | Reads sources, extracts signal |

These roles are logical labels, not separate OpenClaw agents. They describe what a given agent prompt context is doing at a given moment.

### Tool Lanes

Three designated tool lanes serve distinct task classes. Agents select the appropriate lane automatically within approved policy (D1.1 Principle 16):

| Tool Lane | Task Classes | Status |
|---|---|---|
| OpenClaw | Orchestration, scheduling, cron, inter-agent dispatch, gateway management | Active — Phase 1 |
| Cursor | Engineering, code generation, QA, document authoring, configuration | Active — Phase 1 |
| Perplexity | Live web research, current-information retrieval, browser tasks, computer-use | Candidate — see D5.7 CAP-006 |

Perplexity is a callable capability invoked by agents, not a runtime agent. It does not hold task state, does not route messages, and does not replace OpenClaw or Cursor. Tool lane activation is governed by D1.12 and D1.15.

### Control Flow (intended)

```
Human (MacBook Air)
    ↓ intent / directive
Mission Control (source of truth)
    ↓ task assignment
bridge (Mac Mini) — dispatches, schedules, monitors
    ├── brain (Desktop) — thinks, remembers, plans
    └── forge (GPU3060) — builds, tests, researches
    ↓ results
Telegram (primary channel) / Discord (optional) → Human review
```

## 7. Autonomy Maturity Ladder

Each task type is rated independently on this scale. The overall system maturity is determined by the lowest-rated critical task type.

| Level | Name | Description |
|---|---|---|
| 0 | Manual | Human does the work directly |
| 1 | Assisted | Agent drafts, human executes |
| 2 | Supervised | Agent executes, human reviews every result |
| 3 | Delegated | Agent executes, human spot-checks on schedule |
| 4 | Trusted | Agent executes, human reviews only flagged items |
| 5 | Autonomous | Agent executes and self-corrects, human reviews digests |

**Current system maturity: Level 0.**
Target end of Phase 3: Level 2 for core task types.
Target end of Phase 4: Level 3 for core task types, Level 2 for new task types.

## 9. Improvement Lanes

These are the defined channels through which new capabilities, knowledge, and methods enter the system. Intake happens deliberately, not reactively.

### Research Lane
- Source: X/Grok posts, articles, YouTube, arXiv, blogs
- Owner: forge (Researcher)
- Output: brief filed to brain (Archivist), summary posted to Telegram (ops-channel)
- Cadence: on trigger or scheduled sweep

### Experiment Lane
- Source: human-initiated experiments with new tools, models, or methods
- Owner: human + forge
- Output: result doc in D5.x or docs, pass/fail verdict, adoption decision

### Skill/Plugin Lane
- Source: OpenClaw skills marketplace, community, custom builds
- Owner: human approves, bridge installs
- Output: new capability registered in D5.2 SKILL-REGISTRY.md

### Model Lane
- Source: new local or API models
- Owner: human evaluates, forge benchmarks
- Output: model added to inference roster or rejected with reason

### Environment/Change Lane
- Source: macOS updates, OpenClaw updates, hardware changes
- Owner: bridge monitors, human approves changes
- Output: updated runbooks, config committed to docs

### Cost / Usage Lane
- Source: model usage logs, token burn rates, provider billing, recurring automation load
- Owner: brain (Merchant / finance-facing logic) with bridge reporting usage data
- Output: visibility into what the system costs to run, where usage is concentrated, and where optimization is warranted
- Cadence: reviewed per phase and when new models or automations are introduced

### Research Intake Progression

Research intake matures in stages as the system gains capacity:

- **Stage 1 — Manual Intake:** Human reads source, pastes summary or link into Telegram. forge (Researcher) processes on demand.
- **Stage 2 — Structured Packetization:** Human uses a defined intake template. forge files, tags, and routes automatically. Brief goes to brain (Archivist).
- **Stage 3 — Automated Intake:** Monitored feeds (RSS, X lists, YouTube channels) are swept on a schedule. forge filters, summarizes, and files without human input. Human reviews digest.

Move to the next stage only when the current stage is stable and producing signal, not noise.

### Workflow Substrate Note

n8n, custom scripts, browser automation, and external workflow tools are **execution substrates**. They are reusable infrastructure that carries out instructions. They are not the strategic brain, not the memory system, and not the source of truth. These tools plug into Mission Control as subordinate layers — they receive tasks, execute them, and return results. Prior n8n workflows that have value are worth migrating into this execution layer rather than discarding. See D5.5 WORKFLOW-MIGRATION-PLAN.md.

## 9.5 Strategic Outcome Tracks

These are the five long-term value streams the organization is building toward. Each track is a distinct domain of work that agents will eventually own and operate. They are not phases — they run in parallel once the foundation is stable.

### Track 1 — Personal Operating System / Jarvis Track
The system manages the human's schedule, communications, research intake, task queue, and ambient awareness. The human receives digests, makes decisions, and delegates execution. The goal is to eliminate low-value cognitive overhead from the human's day entirely.

### Track 2 — Content and Traffic Engine Track
The system produces written, visual, or structured content from research and directives. It publishes, distributes, and monitors performance. Over time it learns what produces reach and engagement and improves its own output quality. Managed by forge (Builder, Researcher) with brain (Archivist) providing memory.

### Track 3 — Commerce and Sales Engine Track
The system manages product listings, pricing logic, fulfillment triggers, and sales channel activity. It monitors inventory states, flags anomalies, and executes routine commerce operations without human input. brain (Merchant) owns this track. Prior commerce automation is the migration baseline.

### Track 4 — Financial Operating System Track
The system tracks revenue, costs, and margins across all active operations. It produces financial digests, flags threshold breaches, and maintains a live picture of the organization's economic state. In early phases, this track does not make financial decisions. It provides accurate, timely financial visibility and recommendations until stronger controls, auditability, and approval rules are in place.

### Track 5 — Experimentation and Intelligence Engine Track
The system continuously runs small experiments — new models, new tools, new methods, new prompts — and files results into D5.4 EXPERIMENT-REGISTRY.md. It surfaces what works and retires what doesn't. This track is what keeps the organization from stagnating. forge owns execution; brain owns the knowledge base; human approves adoption decisions.

## 10. Completion Standard Per Phase

A phase is complete when:
1. All exit criteria are checked off
2. All phase documents are written and accurate
3. At least one human review session has verified the system behaves as documented
4. No open critical failures exist
5. The next phase's first document has been drafted

A phase is **not** complete because the code works. It is complete when the system is documented, verified, and stable.
