# D1.12 — CAPABILITY_ACQUISITION_POLICY
**Autonomous AI Organization — How New Capabilities Are Evaluated, Approved, Registered, and Governed**
_Last updated: 2026-03-13 | Status: Active | Parent: D1.1-ROADMAP.md_

---

## 1. Purpose

This document defines how the organization acquires new capabilities. A capability is anything that extends what the swarm can do: a model, a skill, a plugin, a tool, a third-party service, a connected platform, a reusable workflow method, or an automation component not already in production.

Capabilities are not adopted on impulse. They must be evaluated, approved by the human, registered, scoped, and versioned before they enter production use. This prevents capability sprawl, undeclared dependencies, unreviewed risk surface, and the drift that comes from each agent improvising new tools without organizational awareness.

**Why this matters:**
- An unapproved capability can introduce security exposure, cost obligations, data routing risks, or reliability assumptions that were never reviewed
- A capability in use but not registered cannot be maintained, deprecated, or governed
- Repeated improvisation of the same task pattern is a signal to evaluate whether a capability should be formally acquired
- Agents cannot acquire capabilities on their own authority — they may identify and recommend; the human decides

**This document governs:**
- What counts as a capability subject to this policy
- How capabilities are discovered, proposed, and evaluated
- The approval gates required before production adoption
- How capabilities are registered, versioned, and scoped
- How risky capabilities receive stronger gates
- How legacy and improvised capabilities are brought into compliance
- How capabilities are retired

**What this document does not cover:**
- Workflow definitions (reusable task execution patterns) — D5.6-WORKFLOW_LIBRARY.md
- Secrets and credential management for capabilities — D1.14-SECRETS_POLICY.md
- Automation class and risk level for an individual task — D1.8-AUTOMATION_POLICY.md
- Specific experiment records — D5.4-EXPERIMENT-REGISTRY.md

## 2. Scope

This policy applies to all capability additions, whether initiated by the human or surfaced by agents. It covers:

| Capability Type | Examples |
|---|---|
| AI models | Language models, embedding models, image models, specialized inference models |
| Agent skills / plugins | Tool-use plugins, function call schemas, capability extensions added to an agent's available toolkit |
| Automation tools | Scripts, cron jobs, API wrappers, programmatic integrations built by the swarm |
| Third-party services | External APIs, SaaS platforms, data providers, notification services |
| Connected platforms | Social channels, e-commerce platforms, knowledge bases, data pipelines |
| Reusable workflow methods | Prompt templates, evaluation frameworks, data transformation patterns used repeatedly across tasks |
| Infrastructure components | Databases, queues, file systems, containers used as capability substrate |

A capability is subject to this policy if adopting it would:
- Change what the swarm can do
- Create an external dependency
- Introduce a cost obligation (actual or potential)
- Expose the organization's data or credentials to a new surface
- Require configuration, maintenance, or operational attention

## 3. Policy Principles

1. **Agents recommend; humans decide.** No agent may adopt a capability into production use on its own authority. Agents may identify candidates, draft proposals, run experiments, and recommend adoption — but the human approves before any capability is used in production.

2. **Prefer approved capabilities over improvisation.** When an approved capability can fulfill a task requirement, it is used. Improvising a new approach when an approved one exists is a deviation from policy and must be flagged. (Source: D5.6 §4.1)

3. **Every production capability is registered.** A capability in use that is not in the capability register is non-compliant. Agents are responsible for flagging unregistered capabilities they discover in active use.

4. **Scope is declared, not inferred.** Every registered capability has a declared scope: which agents may use it, for which task classes, under what conditions. Using a capability outside its declared scope requires a scope update, which is treated as a new acquisition decision.

5. **Risky capabilities require stronger gates.** Capabilities that touch financial data, credentials, external write access, or user data are Risk Class 3 or higher. They do not enter production on the same path as low-risk utilities.

6. **Acquisition is reversible; operational dependency is not.** Evaluate and register before use. Once a capability is embedded in live workflows, reversing it is costly. The right time to apply scrutiny is before adoption, not after.

7. **Capability sprawl is a failure mode.** More is not better. Each new capability adds maintenance surface, potential failure points, and organizational complexity. Acquisitions are justified by demonstrated need, not speculative future use.

8. **Cost is always surfaced.** Any capability with a variable, usage-based, or subscription cost must have that cost declared at acquisition time and monitored as defined in D1.8 §16 (financial track). No capability with unknown or uncapped cost enters production without an explicit cost ceiling approved by the human.

9. **Experiments are not production.** Running a capability in an experiment (D5.4) is not the same as adopting it for production use. An experiment that succeeds must still pass through the acquisition gates in §7 before production deployment.

10. **Legacy capabilities are not exempt.** Capabilities that predate this policy are subject to retroactive registration and review. Until reviewed, they are treated as pending approval — agents should flag them and the human should schedule review.

## 4. Capability Categories

Capabilities are classified by their primary function. This classification determines which evaluation checklist applies and what the default risk class is.

| Category | Description | Default Risk Class |
|---|---|---|
| CAP-MOD | AI model — inference, embedding, classification | 2 (Medium) |
| CAP-SKL | Agent skill or plugin — extends an agent's tool palette | 2 (Medium) |
| CAP-AUT | Automation tool — script, API wrapper, cron component | 1–3 (based on access scope) |
| CAP-SVC | Third-party service — external API or SaaS dependency | 2–3 (based on data exposure) |
| CAP-PLT | Connected platform — social, commerce, data pipeline | 3 (High) — external write access assumed |
| CAP-WFM | Workflow method — prompt template, evaluation framework, reusable pattern | 1 (Low) |
| CAP-INF | Infrastructure component — DB, queue, storage, container | 2–3 (based on data sensitivity) |

Risk class may be elevated based on evaluation findings. It may not be lowered below the default for its category without explicit justification in the capability record.

## 5. Acquisition Sources

Capabilities may be surfaced through any of the following sources:

### 5.1 X Research Pipeline
The X research pipeline (D1.10) may identify capabilities worth evaluating. Signal types `capability` and `architecture` in D1.10 §7 are the primary capability-discovery signals. A research finding that reaches the output stage (D1.10 §10) becomes a capability candidate, not an automatic acquisition.

### 5.2 Experiment Results
An experiment in D5.4 may demonstrate the value of a new capability and recommend adoption. Successful experiment results create a capability candidate. The experiment record is linked from the capability record at acquisition time.

### 5.3 Human Intuition and Directive
The human may identify a capability worth evaluating based on external awareness, industry exposure, or organizational need. Human-initiated candidates skip the candidate surfacing stage and go directly to evaluation.

### 5.4 Repeated Work Patterns (Internal Signal)
When the swarm repeatedly improvises the same capability (using an unapproved tool for the same class of work 3+ times), this is a signal that a formal acquisition may be warranted. Conductor surfaces this alongside the workflow candidate signal in §5.3 of D5.6. The human decides whether to acquire the capability or leave it ad-hoc.

### 5.5 Legacy Asset Discovery
Capabilities already in use but not registered are discovered through legacy asset review (D5.x). These are surfaced to the human for retroactive evaluation and either registered or retired.

## 6. Evaluation Stages

Every capability candidate passes through these stages before approval is requested.

### 6.1 Stage 1 — Candidate Identification
- Source is identified (§5 above)
- Capability category assigned (§4)
- Initial risk class assigned
- A capability proposal stub is created in the capability register (status: `candidate`)

### 6.2 Stage 2 — Feasibility and Fit Assessment
- Does this capability solve a real, demonstrated need? (not speculative)
- Does an approved capability already cover this need? (if yes, stop — use the existing one)
- Is the source credible? (if CAP-MOD or CAP-SVC: is the provider established? is the API stable?)
- What is the cost structure? (free tier, usage-based, subscription — with ceiling estimate)

### 6.3 Stage 3 — Risk and Dependency Analysis
- What external dependencies does this introduce?
- What data is shared with the capability (credentials, user data, task content)?
- What is the failure mode if this capability becomes unavailable?
- Does this create a single point of failure in any workflow?
- Does this touch financial, credential, or user-facing systems? (elevates risk class)

### 6.4 Stage 4 — Experiment (Optional but Recommended)
For CAP-MOD, CAP-SKL, CAP-SVC, and CAP-PLT: a bounded experiment is recommended before approval. The experiment is filed in D5.4. Results are linked to the capability proposal.

For CAP-WFM and low-risk CAP-AUT: a brief trial may substitute for a formal experiment.

### 6.5 Stage 5 — Adoption Proposal
A formal adoption proposal is drafted and submitted to the human. Format defined in §7.2.

## 7. Approval Gates

### 7.1 Risk-Gated Approval Requirements

| Risk Class | Approval Required | Experiment Required | Notes |
|---|---|---|---|
| 1 (Low) | Human acknowledgement | No | Proposal posted in Telegram; human confirms |
| 2 (Medium) | Human explicit approval | Recommended | Proposal in Mission Control; human responds in Telegram or MC |
| 3 (High) | Human explicit approval + review period | Yes | Minimum 48-hour review window before activation |
| 4 (Critical) | Human explicit approval + documented risk acceptance | Yes | Written acknowledgement in Mission Control required; no time pressure |

Risk Class 4 is reserved for capabilities with financial write access, credential management scope, or direct data exfiltration risk. These are rare and treated as organizational-level decisions, not operational ones.

### 7.2 Adoption Proposal Format

Every proposal submitted for approval must include:

```
---
id: CAP-<sequential number>
date: YYYY-MM-DD
proposed_by: human | bridge | brain | forge
category: CAP-MOD | CAP-SKL | CAP-AUT | CAP-SVC | CAP-PLT | CAP-WFM | CAP-INF
name: <short human-readable name>
risk_class: 1 | 2 | 3 | 4
source: X-research | experiment | human | internal-pattern | legacy
experiment_ref: <D5.4 experiment ID or null>
---

## What This Is
<1–2 sentence description of the capability>

## Why It Is Needed
<Demonstrated need — reference the task class or pattern that surfaced this>

## What It Can Do
<Scope of function — what it can do AND what it explicitly cannot do in this organization>

## What It Costs
<Cost structure: free / usage-based (with ceiling estimate) / subscription (rate) / infrastructure cost>

## Dependencies Introduced
<External services, credentials, or infrastructure it depends on>

## Data Exposure
<What data the capability touches, where it is sent, and under what conditions>

## Failure Mode
<What breaks if this capability becomes unavailable>

## Alternatives Considered
<Why existing approved capabilities do not cover this need>

## Proposed Scope
<Which agents may use it, for which task classes, under what conditions>

## Recommended Activation Gate
<What must be true before this is used in production (credentials registered, test passed, etc.)>
```

### 7.3 Approval Decision

The human responds with one of:
- **Approve** — capability may be registered and activated per the declared scope
- **Approve with conditions** — capability may be registered with modified scope or additional gate
- **Defer** — not now; re-surface at a defined milestone
- **Decline** — not for this organization; record the reason

The decision is recorded in the capability register. Declined capabilities retain their record with status `declined` — the reasoning is preserved for future reference.

## 8. Registration and Versioning Policy

### 8.1 Capability Register

The capability register is maintained in Mission Control at **D5.7-CAPABILITY-REGISTER.md** (see §16 Priority 1). Every production capability has an entry. The register is the source of truth for what the organization is currently running.

**Relationship to D5.2-SKILL-REGISTRY.md:** D5.2 is a specialized index for agent skills and plugins (CAP-SKL category only). D5.7 is the master register covering all capability categories. When a skill or plugin is registered, it has a full record in D5.7 and a reference entry in D5.2 for skill-specific metadata (invocation pattern, agent binding, activation command). D5.7 is authoritative for approval status and compliance fields. D5.2 is authoritative for operational skill invocation metadata. Do not treat them as duplicates — they serve different purposes.

**Write authority:** D5.7 is an operational Mission Control registry, not a D-series policy document. Bridge (Conductor) may update the following fields within its approved operational scope: `status` (transitions to `active` or `deprecated`), `last_reviewed`, `review_due`, `notes`, and usage data for cost tracking. Bridge may not change `approved_by`, `approval_date`, `risk_class`, `authorized_agents`, or `authorized_task_classes` without a new human approval decision. Bridge's operational register access does not affect human-only authority over capability approval and scope.

Required fields for every registered capability:

| Field | Description |
|---|---|
| `id` | CAP-### sequential |
| `name` | Short human-readable name |
| `category` | From §4 |
| `status` | `candidate` / `approved` / `active` / `deprecated` / `retired` / `declined` |
| `risk_class` | 1–4 |
| `version` | Integer, starting at 1 |
| `approved_by` | `human` |
| `approval_date` | YYYY-MM-DD |
| `authorized_agents` | Which agents may use it |
| `authorized_task_classes` | Which task classes it may be used in |
| `cost_ceiling` | Approved monthly cost limit (null if free) |
| `experiment_ref` | D5.4 experiment ID, or null |
| `source_ref` | D1.10 research packet ID, or null |
| `last_reviewed` | YYYY-MM-DD |
| `review_due` | YYYY-MM-DD (based on §8.3) |
| `notes` | Any conditions, restrictions, or known issues |

### 8.2 Versioning

When a registered capability changes in a way that affects its scope, risk class, cost, or authorized use:
- The version number increments
- The prior version is retained in the record under `change_notes`
- If the change elevates the risk class or expands scope, re-approval is required before the new version is used in production

Routine updates (e.g. SDK patch version, API endpoint change with no behavioral difference) do not require re-approval — they are recorded as a version note.

### 8.3 Review Cadence

Every active capability is reviewed on a schedule based on its risk class:

| Risk Class | Review Cadence |
|---|---|
| 1 | Annual |
| 2 | 6-month |
| 3 | Quarterly |
| 4 | Monthly |

Reviews confirm the capability is still needed, still compliant, still within cost bounds, and that no better alternative has emerged. Review is triggered by the capability register's `review_due` field surfaced in the weekly digest.

## 9. Production Adoption Policy

### 9.1 Activation Gate

An approved capability is not automatically active. Before first use in production:
1. Credentials and tokens registered in D1.14 (if applicable)
2. Scope confirmed with the authorized agents
3. A test task executed and verified
4. The capability record updated to `status: active` with activation date

An agent that attempts to use an approved-but-not-yet-activated capability must halt and surface the gap to bridge.

### 9.2 Scope Enforcement

Agents use capabilities only within their declared scope. A capability authorized for `task_class: research` is not used for `task_class: content-publishing` without a scope update. Scope creep discovered in a task record must be flagged and the capability record corrected before the out-of-scope use continues.

### 9.3 Cost Monitoring

For capabilities with usage-based or subscription costs, bridge monitors actual vs. ceiling monthly. If usage approaches 80% of the declared cost ceiling, a notice goes into the daily digest. If it hits 100%, automation halts and an escalation fires. (Source: D1.8 §16 financial track)

### 9.4 Dependency Cascade Awareness

When a capability is retired, any workflows or task classes that depend on it are identified before retirement is complete. Dependent workflows must be updated or blocked before the capability is deactivated. Bridge maintains the dependency map in the capability register.

## 10. Workflow and Method Acquisition Policy

### 10.1 Reusable Workflow Methods (CAP-WFM)

A workflow method is a reusable approach to a class of work: a prompt template, an evaluation rubric, a data transformation pattern, a multi-step decision framework. These are not the same as workflow definitions in D5.6 (which define execution sequences for recurring tasks) — they are the tools and techniques used within those sequences.

CAP-WFM capabilities are lower risk by default (Risk Class 1) because they do not introduce external dependencies. They are still subject to registration if they become consistently used across multiple task classes.

### 10.2 Prefer Registered Methods

When a registered CAP-WFM exists that covers a task requirement, it is used. Creating a new ad-hoc method when a registered one applies is flagged as a deviation in the task record.

### 10.3 Method Candidate Surface

When the same prompt template, evaluation rubric, or transformation pattern is used 3+ times by any agent, it becomes a CAP-WFM candidate. Conductor surfaces it the same way it surfaces workflow candidates (D5.6 §5.3).

## 11. Legacy Capability Migration Policy

### 11.1 Definition

A legacy capability is any capability in use that predates this policy or that was adopted without going through the acquisition gates defined here. This includes:
- Tools or APIs integrated before the organization formalized
- Skills or plugins that were added ad-hoc and never registered
- Third-party services that agents were given access to without formal review

### 11.2 Discovery and Flagging

Agents encountering an unregistered capability in active use must:
1. Record it in the task record as `unregistered_capability_detected: [name]`
2. Include it in the next daily digest
3. Not cease using it immediately if ceasing would break an active workflow — continue use pending review, but flag it

### 11.3 Review Outcome

After discovery, the human schedules a retroactive review using the evaluation stages in §6 (stages 2–5 are sufficient; stage 1 is already complete by discovery). The outcome is one of:
- **Register and continue** — passes retroactive review; added to capability register with current version
- **Register with restrictions** — passes with conditions; scope reduced or usage rules added
- **Retire** — does not pass; retirement timeline and workflow impact assessed per §13

Legacy capabilities that are not reviewed within 90 days of discovery are treated as non-compliant and bridge escalates.

## 12. Risk Classification for Capabilities

### 12.1 Risk Class Assignment

Risk class is assigned at candidate identification (§6.1) based on category defaults (§4) and the factors below. It may only be elevated during evaluation — not lowered without justification in the record.

**Factors that elevate risk class:**

| Factor | Effect |
|---|---|
| External write access (write to APIs, post to platforms) | +1 minimum |
| Financial data access or financial transaction capability | Automatically Risk Class 4 |
| Credential or secret management scope | +1 minimum |
| PII or user data exposure | +1 minimum |
| Single point of failure in a critical workflow | +1 |
| Usage-based cost with no ceiling | +1 |
| Beta / unstable API from an unestablished provider | +1 |

### 12.2 Risk Class Definitions

| Class | Name | Gate | Example |
|---|---|---|---|
| 1 | Low | Human acknowledgement | Read-only utility, prompt template, internal tool with no external dependency |
| 2 | Medium | Human explicit approval | Language model API, internal automation tool, read-only external API |
| 3 | High | Explicit approval + 48h review + experiment | External write API, platform integration, service with significant cost potential |
| 4 | Critical | Explicit approval + documented risk acceptance + written MC record | Financial transaction capability, credential management, irreversible external action scope |

### 12.3 Risk Class Is Not Automation Class

Risk class (§12.2) describes the acquisition gate required. Automation class (D1.8 §4) describes how the capability is executed once approved. They are independent. A Risk Class 2 capability may be executed as a Class 1 (cron-scheduled) automation or a Class 4 (delegated multi-step) automation, depending on how it is used.

## 13. Retirement and Deactivation Policy

### 13.1 Retirement Triggers

A capability is retired when:
- It is no longer used in any active workflow or task class
- It has been superseded by a better-approved alternative
- Its provider has deprecated it or the integration is no longer maintainable
- The human explicitly retires it
- It fails retroactive review (§11.3) and the decision is "retire"

### 13.2 Retirement Process

Retirement follows this sequence:
1. Identify all workflows and task classes that depend on the capability
2. Update or block dependent workflows before deactivation
3. Update the capability register: `status: retired`, add `retirement_date` and `retirement_reason`
4. Remove credentials from D1.14 per the secrets deactivation process
5. Confirm no agent is calling the capability post-retirement (one-cycle verification)

A retired capability is not deleted from the register. It is marked retired and retained for audit.

### 13.3 Deprecation vs. Retirement

A capability is `deprecated` when it is still functional but should no longer be used for new work (a better alternative now exists). Existing uses may continue during a transition period. It is `retired` when it is fully deactivated and no agent may use it. The transition from deprecated to retired requires a workflow migration plan.

## 14. Human and Agent Roles in Capability Acquisition

| Role | Responsibility |
|---|---|
| Human | Approves all production capability adoptions; sets risk tolerance; reviews high-risk capabilities; retires capabilities |
| Bridge (Conductor) | Surfaces capability candidates from internal patterns; manages the capability register; monitors cost ceilings; escalates unregistered capabilities; tracks review due dates |
| Brain (Archivist) | Archives experiment results and capability records to canonical memory; provides historical capability context during evaluation |
| Forge | Executes capability experiments (D5.4); reports findings; flags unregistered capabilities discovered in task execution |
| Any agent | May identify and report a capability candidate; may not approve, adopt, or register a capability without human approval |

**The boundary:** Agents are the discovery and execution layer. The human is the acquisition authority. An agent that adopts an unapproved capability in production — regardless of how confident it is in the value — has violated this policy.

## 16. Immediate Priorities

1. **Create D5.7-CAPABILITY-REGISTER.md during Phase 1 setup** — The register is the backbone of this policy. Without a defined home in Mission Control, capabilities cannot be formally tracked. Create D5.7-CAPABILITY-REGISTER.md and register it as a writable surface for bridge per the write authority rules in §8.1. D5.7 must exist before any capability is moved from `candidate` to `approved`. (Source: §8.1)

2. **Create an initial legacy capability inventory at Phase 1 start** — Before operations begin, list every tool, API, model, and service the swarm will use at Day 1. Retroactively register all of them as founding-state capabilities. This prevents the entire starting configuration from being treated as "pending review." (Source: §11)

3. **Set cost ceilings for any usage-based services before activation** — Any capability with variable cost must have a human-approved cost ceiling before it goes active. This is a prerequisite step, not a post-activation concern. (Source: §9.3, D1.8 §16)

4. **Link D1.10 research pipeline to capability candidate surface** — When a D1.10 research finding is tagged `signal_type: capability` or `architecture`, the output stage should automatically create a capability candidate stub in the register. This connection should be confirmed when both D1.10 Mode 2 and the capability register are operational. (Source: §5.1)

5. **Confirm bridge has write access to the capability register** — Bridge (Conductor) manages the register, updates status fields, monitors review due dates, and escalates unregistered capabilities. This access path must be confirmed as part of the Mission Control access policy resolution (D1.3 MC-ACC-03). (Source: §14)
